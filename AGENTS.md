# 저장소 가이드

## 문서 역할

- README: 프로젝트 소개와 로컬 사용 방법
- AGENTS: 변경 작업의 구조·스타일·검증·제출 규칙
- 명령어와 기본 구조는 두 문서에 함께 둔다.

## 구조

- DevNote는 Vite 기반 React 19·TypeScript 정적 앱이다.
- 주요 구조는 다음과 같다.

```text
.
├── src/
│   ├── main.tsx            # 해시 라우팅 초기화
│   ├── App.tsx             # 라우트 정의
│   ├── components/         # 공용 UI
│   ├── hooks/              # 재사용 상태 로직
│   ├── content.ts          # Fumadocs 콘텐츠 로더
│   ├── index.css           # 전역 스타일
│   └── test/               # 테스트 설정
├── content/
│   ├── meta.json           # 섹션 순서
│   └── <section>/
│       ├── meta.json       # 섹션 표시 이름
│       └── <note>.mdx      # 학습 노트
├── source.config.ts        # Fumadocs MDX·스키마 설정
├── vite.config.ts          # 빌드 설정
└── public/                 # 필요 시 정적 자산 추가
```

- 테스트는 컴포넌트 옆의 `ComponentName.test.tsx` 또는 `src/__tests__/`에 둔다.
- 섹션은 기술별 폴더, 노트는 MDX 파일, 토픽은 노트의 `##` 제목이다.
- `content/meta.json`의 `pages`는 Fumadocs 스키마 필드다. 화면·라우트 외에는 “page”라는 표현을 쓰지 않는다.
- `.source/`는 생성물이다. 커밋하지 않는다.

## 명령어

- `npm install`: 잠긴 의존성 설치
- `npm run dev`: 포트 3000에서 개발 서버 실행
- `npm run lint`: TypeScript 타입 검사
- `npm test`: jsdom에서 Vitest 1회 실행
- `npm run build`: `dist/`에 프로덕션 빌드
- `npm run preview`: 빌드 결과 로컬 확인
- `npm run clean`: 생성물 삭제
- 제출 전 `npm run lint && npm test && npm run build`를 실행한다.

## 코드 스타일

- TypeScript, ES 모듈, React 함수 컴포넌트, 2칸 들여쓰기를 사용한다.
- 기존 세미콜론·작은따옴표 스타일을 따른다.
- 컴포넌트 파일은 PascalCase: `SearchModal.tsx`
- 훅은 `use` 접두사의 camelCase: `useDarkMode.ts`
- 변수·함수는 camelCase를 사용한다.
- 재사용 가능한 상태 로직은 훅으로 분리한다.
- 기존 Tailwind·다크 모드 패턴을 유지한다.
- 자동 포매터와 ESLint 설정은 없다. 변경 범위를 좁히고 `npm run lint`를 실행한다.

## 학습 노트

- 콘텐츠 추가·수정은 AI 에이전트가 수행한다.
- TypeScript에 내용을 넣지 말고 `content/<section>/<note>.mdx`를 추가·수정한다.
- frontmatter에는 `title` 등 노트 전용 메타데이터만 둔다. 섹션·순서는 중복하지 않는다.
- 파일명 숫자 순서가 노트 순서다. 탐색·검색 데이터는 자동 생성되므로 별도 인덱스를 만들지 않는다.
- 검색 토픽에는 `##` 제목을 사용하고, 코드 블록에는 언어 식별자를 지정한다.
- `pages`는 Fumadocs 필드명이며, 문서에서는 섹션 순서로 부른다.

### 새 학습 섹션 작성 지침

- 새 학습 섹션의 주제는 사용자가 지정한 개발 관련 기술로 한정한다. 주제를 임의로 바꾸거나 인접 기술을 별도 섹션으로 추가하지 않는다.
- 새 학습 섹션 작업의 기본 변경 범위는 `content/`뿐이다. `content/` 밖의 파일 변경이 필요하거나 유용하다고 판단되면, 변경하기 전에 이유와 대상 파일을 설명하고 사용자에게 질문한다.
- 기존 섹션의 파일명, 제목, 설명 밀도, 코드 블록 형식을 먼저 참고해 새 콘텐츠에 적용한다.

#### 구성 및 순서

1. `content/<section>/` 폴더를 만들고 표시 이름만 담은 `meta.json`을 추가한다.
2. 학습 순서에 맞는 숫자 파일명으로 MDX 노트를 최소 4개 만든다. 파일명 숫자가 노트 순서이므로 중간 번호를 건너뛰지 않고, 각 노트 frontmatter에 `Part N. ...` 형식의 제목을 둔다.
3. 각 노트는 학습 흐름을 나타내는 `##` 토픽을 최소 4개 포함한다. 제목은 검색 결과만으로도 학습 내용을 파악할 수 있게 구체적으로 쓴다.
4. 노트와 토픽은 앞 내용을 전제로 자연스럽게 이어지게 배치한다. 전체 섹션은 최소한 다음 학습 단계를 모두 포함해야 한다.
   - 초급: 기술의 목적, 핵심 개념, 설치·실행 또는 가장 작은 사용 예시
   - 중급: 주요 API·문법, 실용 패턴, 데이터·상태·구조 등 핵심 활용법
   - 고급: 내부 동작, 성능, 보안, 오류 처리, 확장성 또는 기술별 심화 주제
   - 실전: 현실적인 기능·프로젝트 적용, 테스트·운영 고려 사항, 문제 해결과 종합 예제
5. `content/meta.json`의 `pages`에 새 섹션 식별자를 추가해 섹션 순서를 갱신한다. 사용자가 위치를 지정했으면 따른다. 지정하지 않았으면 기존 기술과의 선후 관계를 판단해 배치하고, 적절한 위치가 명확하지 않으면 마지막에 둔다.

#### 토픽의 설명과 예시

- 각 토픽은 개념의 이유와 사용 시점을 충분히 설명하고, 독자가 바로 실행하거나 변형할 수 있는 예시 코드를 제공한다. 코드만 나열하지 않는다.
- 섹션 첫 노트의 기술 소개 토픽은 기술의 목적과 학습 범위를 요약하는 데 집중할 수 있으며, 이 토픽에만 예시 코드가 없어도 된다.
- 예시 코드는 해당 토픽에서 설명한 내용과 정확히 맞아야 하며, 필요한 전제조건·입력·결과·주의점은 본문 또는 코드 주석으로 밝힌다.
- 코드 블록에는 적절한 언어 식별자를 지정하고, 여러 토픽에서 같은 예제를 확장할 때는 이전 단계와 달라진 점을 설명한다.
- 버전, 보안 권장 사항, 명령어처럼 시점에 따라 달라질 수 있는 정보는 콘텐츠 작성 시 신뢰할 수 있는 최신 공식 자료로 확인한다.

#### 완료 전 확인

- 새 섹션 폴더, 섹션 `meta.json`, 최소 4개의 MDX 노트, 그리고 `content/meta.json`의 섹션 순서가 모두 존재하는지 확인한다.
- 노트마다 `##` 토픽이 4개 이상인지, 전체 섹션이 초급·중급·고급·실전을 모두 다루는지, 각 토픽에 충분한 설명과 필요한 코드 예시가 있는지 확인한다.
- 콘텐츠 변경 후 `npm run lint && npm test && npm run build`를 실행한다.

````mdx
---
title: 'Part 1. 노트 제목'
---

설명 본문입니다.

```ts
const example = true;
```
````

## 테스트

- Vitest, Testing Library, jsdom을 사용한다.
- UI 변경 시 데스크톱·모바일, 검색, 탐색, 코드 복사, 라이트·다크 모드를 수동 확인한다.
- 커버리지 기준은 없다.

## 커밋·PR

- 짧은 명령형 커밋 메시지를 쓴다.
  - 예: `fix: preserve search navigation`
  - 예: `docs: update contributor guide`
- PR에는 변경 내용, 검증 명령어, 관련 이슈를 적는다.
- UI 변경 PR에는 전·후 스크린샷을 넣는다.
- `dist/`, 환경 파일, 자격 증명, 생성 로그는 커밋하지 않는다.
