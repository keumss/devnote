# DevNote 개발 가이드

## 문서 역할

- README는 프로젝트 소개, 기술 스택, 콘텐츠 관리 개요를 다룬다.
- AGENTS는 프로젝트 구조와 변경 작업의 스타일·검증·제출 규칙을 정의한다.

## 프로젝트 구조와 용어

DevNote는 Vite 기반 React 19·TypeScript 정적 앱이다.

```text
.
├── src/
│   ├── main.tsx            # 해시 라우팅 초기화
│   ├── App.tsx             # 라우트 정의
│   ├── components/         # 공용 UI
│   ├── hooks/              # 재사용 상태 로직
│   ├── content.ts          # Fumadocs 콘텐츠 로더
│   ├── index.css           # 전역 스타일
│   └── test/               # 테스트 설정
├── content/
│   ├── meta.json           # 섹션 순서
│   └── <section>/
│       ├── meta.json       # 섹션 표시 이름
│       └── <note>.mdx      # 학습 노트
├── source.config.ts        # Fumadocs MDX·스키마 설정
├── vite.config.ts          # 빌드 설정
└── public/                 # 정적 자산
```

- 섹션은 기술별 폴더, 노트는 MDX 파일, 토픽은 노트의 `##` 제목이다.
- `content/meta.json`의 `pages`는 Fumadocs 스키마 필드다. 문서에서는 이를 섹션 순서로 부르고, 화면·라우트 외에는 “page”라는 표현을 쓰지 않는다.
- 테스트는 구현 파일 옆이나 `src/` 아래에 `*.test.ts`·`*.test.tsx` 형식으로 둔다.
- `.source/`와 `dist/`는 생성물이므로 커밋하지 않는다.

## 작업 원칙

- 요청과 직접 관련된 범위만 변경하고 기존 구조와 패턴을 따른다.
- 자동 포매터와 ESLint 설정은 없다. 변경 범위를 좁게 유지하고 타입 검사를 실행한다.
- 환경 파일, 자격 증명, 생성 로그를 커밋하지 않는다.

## 학습 콘텐츠

### 작성 범위와 구조

- 콘텐츠는 TypeScript가 아닌 `content/<section>/<note>.mdx`에 작성한다. 기본 변경 범위는 `content/`이며, 다른 파일이 필요하면 먼저 이유와 대상을 사용자에게 알린다.
- 사용자가 지정한 기술만 다루고 인접 기술을 별도 섹션으로 확장하지 않는다. Python 섹션은 Python 3.12를 기준으로 한다.
- frontmatter에는 `title` 등 노트 전용 메타데이터만 둔다. 섹션 표시 이름은 `content/<section>/meta.json`, 섹션 순서는 `content/meta.json`에서 관리한다.
- 노트는 파일명의 자연 정렬 순서로 표시된다. 탐색·검색 데이터는 자동 생성하므로 별도 인덱스를 만들지 않는다.
- 검색 가능한 토픽에는 내용을 구체적으로 나타내는 `##` 제목을 쓰고, 코드 블록에는 적절한 언어 식별자를 붙인다.

### 학습 흐름과 설명 밀도

- SQL 섹션의 초급·중급 노트 수준을 설명 밀도의 기준으로 삼는다. 정확성과 실제 사용 빈도가 높은 핵심 개념·API·패턴을 먼저 빠르게 익히게 하고, 드문 기능과 인접 패턴은 핵심 흐름의 전제이거나 사용자가 요구한 경우에만 다룬다.
- 기초·중급 노트의 목표는 기술 전체를 빠짐없이 설명하는 것이 아니라, 학습자가 기술의 역할과 자주 쓰는 문법·API·기본 작업 흐름을 빠르게 파악하고 짧은 예제를 변형해 보는 것이다.
  - 기초: 기술의 목적을 짧게 소개한 뒤, 가장 자주 쓰는 구성 요소와 최소 실행 예시를 쉬운 순서로 배치한다.
  - 중급: 기초 예시를 바탕으로 자주 쓰는 API·문법, 데이터·실행 경계, 실용 패턴을 짧게 연결한다. 세부 원리, 모든 선택지, 예외 상황을 한 토픽에 담지 않는다.
- 기초·중급 노트는 보통 6~8개 토픽으로 구성한다. 각 토픽은 하나의 학습 목표, 1~2개의 짧은 설명 문단, 관련 코드만 담은 작은 예제를 기본 단위로 한다. 토픽 수를 줄이는 대신 한 토픽의 설명과 예제를 과도하게 늘리지 않으며, 같은 결론·준비 코드·주의사항을 반복하지 않는다.
- 고급은 희귀한 API를 나열하지 않고 앞서 배운 기능의 내부 동작, 성능, 보안, 오류 처리, 확장성 중 중요한 2~3개를 심화한다. 실전은 하나의 현실적인 적용 흐름과 직접 관련된 테스트·운영·문제 해결 방법만 다룬다. 두 단계 모두 학습 목표에 필요하지 않은 배경 설명과 변형 예제는 줄인다.
- 공통 모델, 연결 설정, 초기 데이터는 한 번 정의해 재사용한다. 모든 코드 블록이 독립 실행 파일일 필요는 없지만 노트 단위로 조합하면 핵심 예제가 실행 가능해야 한다.
- 기존 섹션은 정확하고 간결한 설명과 예제를 우선 보존한다. 전체 흐름을 먼저 읽고 국소 수정하되, 핵심 학습 순서가 불명확하거나 설명이 과밀할 때만 토픽을 이동·통합·축약·교체한다.

### 새 섹션 형식

1. 기존 섹션의 파일명, 제목, 설명 밀도와 코드 형식을 먼저 확인한다.
2. `content/<section>/meta.json`에는 표시 이름만 기록한다.
3. `<section>-part1.mdx` 형식으로 번호가 연속된 노트를 최소 4개 만들고, frontmatter 제목은 `Part N. ...` 형식으로 작성한다.
4. 각 노트에 검색만으로 내용을 파악할 수 있는 `##` 토픽을 둔다. 기초·중급 노트는 핵심 흐름을 빠르게 훑을 수 있도록 보통 6~8개로 구성하고, 고급·실전은 학습 목표에 맞는 수로 제한한다.
5. 섹션 전체가 기초→중급→고급→실전으로 이어지게 하고, `content/meta.json`에는 기술의 선후 관계에 맞춰 섹션 식별자를 추가한다. 위치를 판단하기 어려우면 마지막에 둔다.

### 예제와 완료 기준

- 첫 노트의 기술 소개를 제외한 각 토픽은 언제 쓰는지 한두 문장으로 밝히고 실행하거나 변형할 수 있는 예시를 제공한다. 한 토픽에서 원리·대안·세부 예외까지 모두 설명하지 않는다.
- 예시는 토픽과 직접 관련된 코드만 보여 준다. 필요한 전제조건·입력·결과·주의점은 본문, 주석, 출력 중 가장 적합한 한곳에 짧게 둔다.
- 결과 확인이 학습에 중요한 쿼리, 변환, 오류 처리, CLI 예제에만 바로 뒤에 핵심 예상 결과를 `text` 코드 블록으로 제시한다. 자명한 예시마다 출력 블록을 반복하지 않으며, 환경에 따라 달라지는 부분은 원인을 짧게 설명하고 고정되는 결과만 보여 준다.
- 버전, 보안 권장 사항, 명령어처럼 변할 수 있는 정보는 최신 공식 자료로 확인한다.
- 완료 전에 섹션 구조와 순서, 노트·토픽 수, 예제 실행 결과를 확인하고, 학습자가 빠르게 읽을 때 흐름이 끊기거나 설명·예제가 과도하게 반복되지 않는지 다시 읽어 수정한다.

## 코드와 UI

### 코드 스타일

- TypeScript, ES 모듈, React 함수 컴포넌트, 2칸 들여쓰기를 사용한다.
- 기존 세미콜론·작은따옴표 스타일을 따른다.
- 컴포넌트 파일은 PascalCase(`SearchModal.tsx`), 훅은 `use` 접두사의 camelCase(`useDarkMode.ts`), 변수·함수는 camelCase를 사용한다.
- 재사용 가능한 상태 로직은 훅으로 분리한다.
- 기존 Tailwind와 `dark:` 다크 모드 패턴을 유지한다.

### UI 구조와 동작

- 모든 화면은 `Layout`을 통해 헤더, 다크 모드, 검색 모달, 모바일 내비게이션을 공유한다. 전역 UI 상태를 개별 화면에 중복하지 않는다.
- 데스크톱 노트 화면의 `SidebarNav`와 모바일의 `MobileNavDrawer`는 같은 탐색 데이터를 사용한다.
- 검색은 `content/`에서 자동 생성된 토픽 데이터를 대상으로 하며, 결과 선택 시 해당 노트의 `##` 토픽 해시로 이동해야 한다.
- 검색 모달과 모바일 내비게이션의 키보드 닫기, 포커스 복원·가두기, 배경 스크롤 잠금을 유지하고 관련 접근성 속성을 점검한다.
- `useDarkMode`는 `localStorage` 및 시스템 색상 설정과 동기화되어야 한다.
- MDX 코드 블록은 `MdxContent`와 `CodeBlock`으로 렌더링한다. 언어 표시, 복사, 가로 스크롤, 라이트·다크 테마를 유지한다.

## 명령어와 검증

- `npm install`: 잠긴 의존성 설치
- `npm run dev`: 포트 3000에서 개발 서버 실행
- `npm run lint`: TypeScript 타입 검사
- `npm test`: jsdom에서 Vitest 1회 실행
- `npm run build`: `dist/`에 프로덕션 빌드
- `npm run preview`: 빌드 결과 로컬 확인
- `npm run clean`: 생성물 삭제

모든 변경을 마친 뒤 콘텐츠와 문서를 포함해 다음 전체 검증을 실행한다.

```sh
npm run lint && npm test && npm run build
```

- 테스트는 Vitest, Testing Library, jsdom을 사용하며 커버리지 기준은 없다.
- UI 변경은 바뀐 기능과 직접 연결된 화면만 수동 확인한다. 반응형, 검색·탐색, 코드 복사, 라이트·다크 모드는 영향을 받은 경우에만 확인한다.

## 커밋과 PR

- 커밋 메시지는 `fix: preserve search navigation`, `docs: update contributor guide`처럼 짧은 명령형으로 작성한다.
- 사용자가 브랜치를 지정하지 않고 커밋 또는 푸시를 요청하면 별도 브랜치를 만들지 않고 `main`에 커밋해 `origin/main`으로 푸시한다.
- 커밋 또는 푸시만 요청하면 GitHub CLI(`gh`) 없이 Git CLI로 처리하며, `gh` 미설치를 작업 중단 사유로 삼지 않는다.
- PR에 변경 내용, 검증 명령어, 관련 이슈를 적는다. UI 변경 시 전·후 스크린샷도 포함한다.

## 배포

- GitHub Pages 배포를 위해 Vite의 `base: './'`와 `HashRouter`를 유지한다.
- 배포 구조가 바뀔 때만 관련 설정을 변경한다.
