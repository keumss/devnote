# DevNote 개발 가이드

## 문서 역할

- README는 프로젝트 소개, 기술 스택, 콘텐츠 관리 개요를 다룬다.
- AGENTS는 프로젝트 구조와 변경 작업의 스타일·검증·제출 규칙을 정의한다.

## 프로젝트 구조와 용어

DevNote는 Vite 기반 React 19·TypeScript 정적 앱이다.

```text
.
├── src/
│   ├── main.tsx            # 해시 라우팅 초기화
│   ├── App.tsx             # 라우트 정의
│   ├── components/         # 공용 UI
│   ├── hooks/              # 재사용 상태 로직
│   ├── content.ts          # Fumadocs 콘텐츠 로더
│   ├── index.css           # 전역 스타일
│   └── test/               # 테스트 설정
├── content/
│   ├── meta.json           # 섹션 순서
│   └── <section>/
│       ├── meta.json       # 섹션 표시 이름
│       └── <note>.mdx      # 학습 노트
├── source.config.ts        # Fumadocs MDX·스키마 설정
├── vite.config.ts          # 빌드 설정
└── public/                 # 정적 자산
```

- 섹션은 기술별 폴더, 노트는 MDX 파일, 토픽은 노트의 `##` 제목이다.
- `content/meta.json`의 `pages`는 Fumadocs 스키마 필드다. 문서에서는 이를 섹션 순서로 부르고, 화면·라우트 외에는 “page”라는 표현을 쓰지 않는다.
- 테스트는 구현 파일 옆이나 `src/` 아래에 `*.test.ts`·`*.test.tsx` 형식으로 둔다.
- `.source/`와 `dist/`는 생성물이므로 커밋하지 않는다.

## 작업 원칙

- 요청과 직접 관련된 범위만 변경하고 기존 구조와 패턴을 따른다.
- 자동 포매터와 ESLint 설정은 없다. 변경 범위를 좁게 유지하고 타입 검사를 실행한다.
- 환경 파일, 자격 증명, 생성 로그를 커밋하지 않는다.

## 학습 콘텐츠

### 범위와 메타데이터

- 콘텐츠는 `content/<section>/<note>.mdx`에 작성한다. 기본 변경 범위는 `content/`이며, 다른 파일이 필요하면 이유와 대상을 먼저 알린다.
- 사용자가 지정한 기술만 다룬다. Python은 3.12 기준이다.
- frontmatter에는 노트 메타데이터만 둔다. 섹션 이름은 `content/<section>/meta.json`, 섹션 순서는 `content/meta.json`에서 관리한다.
- 노트는 파일명 자연 순서로 표시되며 탐색·검색 데이터는 자동 생성된다. 별도 인덱스를 만들지 않는다.
- 검색 가능한 `##` 제목과 언어가 지정된 코드 블록을 사용한다.

### 작성 원칙

#### 노트 공통 지침

- 목표는 기능 나열이 아니라, 학습자가 문제를 보고 도구를 고르고 예제를 바꿔 보는 것이다. 자주 쓰는 개념과 패턴부터 다루며 드문 기능은 필요한 경우만 포함한다.
- 기초·중급 노트는 보통 6~8개 토픽으로 구성한다.
- 한 토픽에는 하나의 학습 목표만 둔다. 토픽은 **언제 쓰는가 → 최소 예제 → 필요한 결과 → 핵심 규칙** 순서로 쓰며, 추상 정의·내부 구현·경고 목록으로 시작하지 않는다.
- 예제는 입력과 결과를 한눈에 연결할 수 있게 작성한다. 한 블록에는 하나의 핵심 API나 결정을 담고, 보일러플레이트와 추상적인 이름을 줄인다.
- 순서·변환·오류·종료 조건처럼 결과가 동작을 구분할 때는 고정된 예상 출력을 `text` 블록으로 제공한다. 자명한 출력과 환경 의존 출력은 반복하지 않는다.
- 같은 노트의 공통 모델·초기 데이터·설정은 재사용한다. 앞 블록에 의존하면 전제를 찾을 수 있게 하고, 노트를 순서대로 읽으면 핵심 예제가 실행 가능해야 한다.
- 기존 콘텐츠는 전체 흐름을 읽은 뒤, 설명이 코드보다 앞서거나 학습 순서가 흐릴 때만 수정한다.

#### 단계별 노트 지침

- 기초: 역할과 기본 사용법을 빠르게 익히게 한다. 가장 자주 쓰는 구성 요소와 최소 예제를 쉬운 순서로 소개한다.
- 중급: 기초 예시를 확장해 데이터 경계와 실용 패턴을 다룬다. 자주 쓰는 API·문법과 기본 흐름을 연결하되, 모든 선택지와 예외를 나열하지 않는다.
- 고급: 실제 개발에서 활용도가 높은 고급 기능과 패턴 2~3개에 집중한다. 필요한 상황과 선택 기준을 반례나 비교 코드로 설명해, 앞서 배운 기본 기능을 확장해 활용하게 한다.
- 실전: 해당 섹션의 핵심 기능으로 해결하는 하나의 구체적인 업무 시나리오와 완료 결과를 먼저 정한다. 보통 4~6개 토픽으로 입력 데이터·순서·제약의 계약을 밝힌 뒤, 핵심 처리 구현을 단계적으로 완성하고, 관련된 외부 경계·실패 처리와 결과 확인 또는 집중된 테스트까지 연결한다. 각 코드 블록은 앞선 코드를 재사용해 하나의 작동하는 결과물로 이어지게 하며, 목표에 필요하지 않은 파일·설정·변형 예제는 추가하지 않는다.

### 새 섹션 형식

1. 기존 섹션의 파일명·제목·코드 형식을 확인하고, 핵심 작업을 기초→중급→고급→실전으로 나눈다.
2. `content/<section>/meta.json`에는 표시 이름만 기록한다.
3. `<section>-part1.mdx` 형식의 연속된 노트를 최소 4개 만들고, 제목은 `Part N. ...`으로 쓴다.
4. 노트마다 검색만으로 내용을 파악할 수 있는 구체적인 `##` 토픽을 두고, 토픽 수와 구성은 단계별 노트 지침을 따른다.
5. `content/meta.json`에는 기술의 선후 관계에 맞춰 섹션을 추가한다. 판단하기 어려우면 마지막에 둔다.

### 완료 점검

- 버전, 보안 권장 사항, 명령어처럼 변할 수 있는 정보는 최신 공식 자료로 확인한다.
- 완료 전에는 섹션 순서, 노트·토픽 수, 예제 실행 결과를 확인한다. 각 토픽의 첫 문장으로 용도를 알 수 있고, 코드·출력으로 동작을 확인할 수 있으며, 설명·예제가 중복되지 않는지 다시 읽는다.

## 코드와 UI

### 코드 스타일

- TypeScript, ES 모듈, React 함수 컴포넌트, 2칸 들여쓰기를 사용한다.
- 기존 세미콜론·작은따옴표 스타일을 따른다.
- 컴포넌트 파일은 PascalCase(`SearchModal.tsx`), 훅은 `use` 접두사의 camelCase(`useDarkMode.ts`), 변수·함수는 camelCase를 사용한다.
- 재사용 가능한 상태 로직은 훅으로 분리한다.
- 기존 Tailwind와 `dark:` 다크 모드 패턴을 유지한다.

### UI 구조와 동작

- 모든 화면은 `Layout`을 통해 헤더, 다크 모드, 검색 모달, 모바일 내비게이션을 공유한다. 전역 UI 상태를 개별 화면에 중복하지 않는다.
- 데스크톱 노트 화면의 `SidebarNav`와 모바일의 `MobileNavDrawer`는 같은 탐색 데이터를 사용한다.
- 검색은 `content/`에서 자동 생성된 토픽 데이터를 대상으로 하며, 결과 선택 시 해당 노트의 `##` 토픽 해시로 이동해야 한다.
- 검색 모달과 모바일 내비게이션의 키보드 닫기, 포커스 복원·가두기, 배경 스크롤 잠금을 유지하고 관련 접근성 속성을 점검한다.
- `useDarkMode`는 `localStorage` 및 시스템 색상 설정과 동기화되어야 한다.
- MDX 코드 블록은 `MdxContent`와 `CodeBlock`으로 렌더링한다. 언어 표시, 복사, 가로 스크롤, 라이트·다크 테마를 유지한다.

## 명령어와 검증

- `npm install`: 잠긴 의존성 설치
- `npm run dev`: 포트 3000에서 개발 서버 실행
- `npm run lint`: TypeScript 타입 검사
- `npm test`: jsdom에서 Vitest 1회 실행
- `npm run build`: `dist/`에 프로덕션 빌드
- `npm run preview`: 빌드 결과 로컬 확인
- `npm run clean`: 생성물 삭제

모든 변경을 마친 뒤 콘텐츠와 문서를 포함해 다음 전체 검증을 실행한다.

```sh
npm run lint && npm test && npm run build
```

- 테스트는 Vitest, Testing Library, jsdom을 사용하며 커버리지 기준은 없다.
- UI 변경은 바뀐 기능과 직접 연결된 화면만 수동 확인한다. 반응형, 검색·탐색, 코드 복사, 라이트·다크 모드는 영향을 받은 경우에만 확인한다.

## 커밋과 PR

- 커밋 메시지는 `fix: preserve search navigation`, `docs: update contributor guide`처럼 짧은 명령형으로 작성한다.
- 사용자가 브랜치를 지정하지 않고 커밋 또는 푸시를 요청하면 별도 브랜치를 만들지 않고 `main`에 커밋해 `origin/main`으로 푸시한다.
- 커밋 또는 푸시만 요청하면 GitHub CLI(`gh`) 없이 Git CLI로 처리하며, `gh` 미설치를 작업 중단 사유로 삼지 않는다.
- PR에 변경 내용, 검증 명령어, 관련 이슈를 적는다. UI 변경 시 전·후 스크린샷도 포함한다.

## 배포

- GitHub Pages 배포를 위해 Vite의 `base: './'`와 `HashRouter`를 유지한다.
- 배포 구조가 바뀔 때만 관련 설정을 변경한다.
