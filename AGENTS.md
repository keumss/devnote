# DevNote 개발 가이드

## 문서 역할

- README는 프로젝트 소개, 기술 스택, 콘텐츠 관리 개요를 다룬다.
- AGENTS는 프로젝트 구조와 변경 작업의 스타일·검증·제출 규칙을 정의한다.

## 프로젝트 구조와 용어

DevNote는 Vite 기반 React 19·TypeScript 정적 앱이다.

```text
.
├── src/
│   ├── main.tsx            # 해시 라우팅 초기화
│   ├── App.tsx             # 라우트 정의
│   ├── components/         # 공용 UI
│   ├── hooks/              # 재사용 상태 로직
│   ├── content.ts          # Fumadocs 콘텐츠 로더
│   ├── index.css           # 전역 스타일
│   └── test/               # 테스트 설정
├── content/
│   ├── meta.json           # 섹션 순서
│   └── <section>/
│       ├── meta.json       # 섹션 표시 이름
│       └── <note>.mdx      # 학습 노트
├── source.config.ts        # Fumadocs MDX·스키마 설정
├── vite.config.ts          # 빌드 설정
└── public/                 # 정적 자산
```

- 섹션은 기술별 폴더, 노트는 MDX 파일, 토픽은 노트의 `##` 제목이다.
- `content/meta.json`의 `pages`는 Fumadocs 스키마 필드다. 문서에서는 이를 섹션 순서로 부르고, 화면·라우트 외에는 “page”라는 표현을 쓰지 않는다.
- 테스트는 구현 파일 옆이나 `src/` 아래에 `*.test.ts`·`*.test.tsx` 형식으로 둔다.
- `.source/`와 `dist/`는 생성물이므로 커밋하지 않는다.

## 작업 원칙

- 요청과 직접 관련된 범위만 변경하고 기존 구조와 패턴을 따른다.
- 자동 포매터와 ESLint 설정은 없다. 변경 범위를 좁게 유지하고 타입 검사를 실행한다.
- 환경 파일, 자격 증명, 생성 로그를 커밋하지 않는다.

## 학습 콘텐츠

### 공통 규칙

- 콘텐츠 추가·수정은 AI 에이전트가 수행하며, TypeScript가 아닌 `content/<section>/<note>.mdx`에 작성한다.
- frontmatter에는 `title` 등 노트 전용 메타데이터만 둔다. 섹션과 순서는 중복해서 기록하지 않는다.
- 노트는 파일명의 자연 정렬 순서로 표시된다. 탐색·검색 데이터는 자동 생성되므로 별도 인덱스를 만들지 않는다.
- 검색 가능한 토픽에는 `##` 제목을, 코드 블록에는 적절한 언어 식별자를 사용한다.

### 내용 선정과 학습 흐름

- 기술 이해와 실제 개발에 자주 쓰이는 핵심 개념·API·패턴을 우선한다. 드문 기능은 핵심 흐름의 전제이거나 사용자가 요구한 경우에만 별도 토픽으로 다룬다.
- 고급 단계는 희귀한 API 나열보다 앞서 배운 기능의 내부 동작과 조합을 심화한다. 성능·보안·오류 처리·운영·문제 해결 중 해당 기술에서 중요한 주제만 선택한다.
- 토픽은 목적과 최소 사용법에서 시작해 핵심 동작과 관계, 실용적 조합, 운영 설정·테스트·문제 해결 순으로 배치한다. 토픽마다 하나의 학습 목표에 집중하고 앞선 설명을 반복하지 않는다.
- 기존 섹션을 보강할 때는 정확하고 간결한 기존 설명과 예제를 우선 보존한다. 섹션 전체를 읽고 오류와 흐름의 단절을 먼저 국소적으로 수정하며, 기존 구조로 핵심 개념을 설명하기 어려울 때만 토픽을 이동·통합·축약·교체한다.
- 설명과 예제는 독자가 동작과 결과를 예측할 수 있을 만큼 제공한다. 설정이나 주변 맥락이 학습 목표보다 길어지면 예제를 줄이거나 토픽을 나누고, 같은 결론을 본문·주석·출력에서 반복하지 않는다.

### 빠른 핵심 학습과 설명 밀도

- 기본 목표는 독자가 핵심 개념과 자주 쓰는 API를 빠르게 이해하고 직접 변형할 수 있게 하는 것이다. 완전한 미니 프로젝트보다 핵심 동작이 드러나는 최소 예제를 우선한다.
- 내용의 우선순위는 정확성, 핵심 개념, 자주 쓰는 실용 패턴, 심화·운영 주제 순이다. 분량을 늘려야 한다면 후순위 내용을 모두 추가하지 말고 핵심 흐름에 필요한 것만 선택한다.
- 노트당 토픽은 기본적으로 4~6개로 구성한다. 각 토픽은 짧은 설명과 하나의 집중된 예제를 기본 단위로 하며, 전제조건·결과·주의점이 실제 이해에 필요한 경우에만 확장한다.
- 공통 모델, 연결 설정, 초기 데이터는 섹션에서 한 번 정의하고 이후 예제에서 재사용한다. 이후 토픽은 달라진 코드와 동작을 중심으로 보여 주고 같은 준비 코드를 반복하지 않는다.
- 모든 코드 블록이 독립 실행 파일일 필요는 없다. 공유하는 전제조건을 명시하고, 노트 단위로 조합했을 때 핵심 예제가 실행 가능해야 한다.
- 여러 파일로 구성된 애플리케이션, 테스트 픽스처, 운영 설정은 해당 구성이 학습 목표인 실전 노트에서만 사용한다.
- 고급·실전 단계는 가능한 주제를 모두 나열하지 않는다. 해당 기술에서 중요도가 높은 내부 동작, 성능, 오류 처리, 테스트, 운영 주제 중 2~3개를 선택해 앞선 내용을 심화한다.
- 독자가 핵심 동작을 설명하고 예제를 변형하며 결과를 확인할 수 있으면 충분하다. 완결성을 높이기 위한 인접 패턴이나 예외적인 API는 추가하지 않는다.

### 새 학습 섹션

#### 범위와 형식

- 사용자가 지정한 개발 기술만 다룬다. 주제를 바꾸거나 인접 기술을 별도 섹션으로 추가하지 않는다.
- Python 섹션은 Python 3.12를 기준으로 작성한다.
- 기본 변경 범위는 `content/`뿐이다. 그 밖의 파일 변경이 필요하거나 유용하면 먼저 이유와 대상 파일을 설명하고 사용자에게 묻는다.
- 기존 섹션의 파일명, 제목, 설명 밀도, 코드 블록 형식을 먼저 확인해 따른다.

다음 요건을 모두 충족한다.

1. `content/<section>/meta.json`에 표시 이름만 기록한다.
2. `<section>-part1.mdx` 형식으로 번호가 연속된 노트를 최소 4개 만들고, 각 frontmatter 제목을 `Part N. ...` 형식으로 작성한다.
3. 각 노트에 구체적인 `##` 토픽을 최소 4개 둔다. 제목만으로도 검색 결과에서 학습 내용을 파악할 수 있어야 한다.
4. 노트와 토픽을 선행 내용을 바탕으로 이어지게 구성하고, 섹션 전체에서 다음 단계를 다룬다.
   - 초급: 목적, 핵심 개념, 설치·실행 또는 최소 예시
   - 중급: 주요 API·문법, 실용 패턴, 핵심 활용법
   - 고급: 내부 동작, 성능, 보안, 오류 처리, 확장성 중 해당 기술에서 중요한 주제 2~3개
   - 실전: 하나의 현실적인 적용 흐름과 테스트·운영·문제 해결 중 직접 관련된 검증 방법
5. `content/meta.json`의 `pages`에 섹션 식별자를 추가한다. 사용자가 위치를 정하지 않았다면 기술의 선후 관계에 따라 배치하고, 판단하기 어려우면 마지막에 둔다.

#### 설명과 예제

- 최소 노트·토픽 수는 형식 요건일 뿐이다. 독자가 개념의 목적과 동작을 이해하고 예제를 실행·변형하며 결과를 판단할 수 있을 만큼 설명한다.
- 각 토픽은 개념의 이유와 사용 시점을 설명하고 실행하거나 변형할 수 있는 예시 코드를 제공한다. 단, 첫 노트의 기술 소개 토픽은 목적과 학습 범위에 집중하며 코드를 생략할 수 있다.
- 예시는 해당 토픽과 정확히 일치해야 한다. 필요한 전제조건·입력·결과·주의점은 본문, 코드 주석, 실행 결과 중 가장 적합한 수단으로 보강한다.
- 같은 예제를 여러 토픽에서 확장하면 이전 단계와 달라진 점을 설명한다.
- 출력이 개념 이해나 동작 검증에 필요한 쿼리, 변환, 오류 처리, CLI 예제에는 실행 코드 바로 뒤에 예상 결과를 `text` 코드 블록으로 제시한다. 설정 생성이나 결과가 자명한 기본 CRUD는 짧은 문장으로 결과를 설명하고 출력 블록을 생략할 수 있다.
- 출력이 환경에 따라 달라지면 고정되는 핵심 부분만 보여 주고 차이의 원인을 설명한다.
- 버전, 보안 권장 사항, 명령어처럼 변할 수 있는 정보는 신뢰할 수 있는 최신 공식 자료로 확인한다.

#### 완료 기준

- 섹션 폴더와 `meta.json`, 최소 4개의 MDX 노트, `content/meta.json`의 섹션 순서가 모두 존재한다.
- 모든 노트에 `##` 토픽이 4개 이상 있고, 섹션 전체가 초급·중급·고급·실전 단계를 다룬다.
- 학습자 관점에서 전체를 다시 읽고, 흐름이 끊기거나 동작·결과를 예측하고 검증하는 데 필요한 설명이 빠진 곳을 보완한다.

````mdx
---
title: 'Part 1. 노트 제목'
---

## 첫 번째 주제

설명 본문입니다.

```ts
const example = true;
```
````

## 코드와 UI

### 코드 스타일

- TypeScript, ES 모듈, React 함수 컴포넌트, 2칸 들여쓰기를 사용한다.
- 기존 세미콜론·작은따옴표 스타일을 따른다.
- 컴포넌트 파일은 PascalCase(`SearchModal.tsx`), 훅은 `use` 접두사의 camelCase(`useDarkMode.ts`), 변수·함수는 camelCase를 사용한다.
- 재사용 가능한 상태 로직은 훅으로 분리한다.
- 기존 Tailwind와 `dark:` 다크 모드 패턴을 유지한다.

### UI 구조와 동작

- 모든 화면은 `Layout`을 통해 헤더, 다크 모드, 검색 모달, 모바일 내비게이션을 공유한다. 전역 UI 상태를 개별 화면에 중복하지 않는다.
- 데스크톱 노트 화면의 `SidebarNav`와 모바일의 `MobileNavDrawer`는 같은 탐색 데이터를 사용한다.
- 검색은 `content/`에서 자동 생성된 토픽 데이터를 대상으로 하며, 결과 선택 시 해당 노트의 `##` 토픽 해시로 이동해야 한다.
- 검색 모달과 모바일 내비게이션의 키보드 닫기, 포커스 복원·가두기, 배경 스크롤 잠금을 유지하고 관련 접근성 속성을 점검한다.
- `useDarkMode`는 `localStorage` 및 시스템 색상 설정과 동기화되어야 한다.
- MDX 코드 블록은 `MdxContent`와 `CodeBlock`으로 렌더링한다. 언어 표시, 복사, 가로 스크롤, 라이트·다크 테마를 유지한다.

## 명령어와 검증

- `npm install`: 잠긴 의존성 설치
- `npm run dev`: 포트 3000에서 개발 서버 실행
- `npm run lint`: TypeScript 타입 검사
- `npm test`: jsdom에서 Vitest 1회 실행
- `npm run build`: `dist/`에 프로덕션 빌드
- `npm run preview`: 빌드 결과 로컬 확인
- `npm run clean`: 생성물 삭제

모든 변경을 마친 뒤 콘텐츠와 문서를 포함해 다음 전체 검증을 실행한다.

```sh
npm run lint && npm test && npm run build
```

- 테스트는 Vitest, Testing Library, jsdom을 사용하며 커버리지 기준은 없다.
- UI 변경은 바뀐 기능과 직접 연결된 화면만 수동 확인한다. 반응형, 검색·탐색, 코드 복사, 라이트·다크 모드는 영향을 받은 경우에만 확인한다.

## 커밋과 PR

- 커밋 메시지는 `fix: preserve search navigation`, `docs: update contributor guide`처럼 짧은 명령형으로 작성한다.
- 사용자가 브랜치를 지정하지 않고 커밋 또는 푸시를 요청하면 별도 브랜치를 만들지 않고 `main`에 커밋해 `origin/main`으로 푸시한다.
- 커밋 또는 푸시만 요청하면 GitHub CLI(`gh`) 없이 Git CLI로 처리하며, `gh` 미설치를 작업 중단 사유로 삼지 않는다.
- PR에 변경 내용, 검증 명령어, 관련 이슈를 적는다. UI 변경 시 전·후 스크린샷도 포함한다.

## 배포

- GitHub Pages 배포를 위해 Vite의 `base: './'`와 `HashRouter`를 유지한다.
- 배포 구조가 바뀔 때만 관련 설정을 변경한다.
