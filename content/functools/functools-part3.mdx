## 캐시는 무엇을 저장하고 언제 피해야 할까요?

`cache()`와 `lru_cache()`는 인자와 반환값을 캐시 항목이 남아 있는 동안 참조합니다. 큰 객체를 반환하거나 사용자 입력처럼 키가 끝없이 늘면 메모리 사용량이 예상보다 커질 수 있습니다.

```python
from functools import lru_cache

@lru_cache(maxsize=2)
def build_report(report_id: str) -> str:
    print(f'build: {report_id}')
    return f'report for {report_id}'

build_report('daily')
build_report('weekly')
build_report('monthly')
print(build_report.cache_info())
```

```text
build: daily
build: weekly
build: monthly
CacheInfo(hits=0, misses=3, maxsize=2, currsize=2)
```

`maxsize`는 항목 수만 제한할 뿐, 항목 하나의 크기는 제한하지 않습니다. 현재 시각, 난수, 쓰기 작업, 제너레이터·코루틴처럼 호출마다 새 결과나 부수 효과가 필요한 함수는 캐시하지 않습니다. 캐시는 순수한 읽기·계산을 재사용하는 경계에 둡니다.

## 동시 호출과 cached_property를 어떻게 설계할까요?

캐시 데코레이터는 내부 자료구조를 안전하게 유지하지만, 첫 결과가 저장되기 전에 같은 키를 다른 스레드가 호출하면 실제 함수가 두 번 이상 실행될 수 있습니다. Python 3.12의 `cached_property`도 인스턴스별 한 번 실행을 보장하던 내부 잠금이 제거되었습니다.

```python
from functools import cached_property
from threading import Lock

class Report:
    def __init__(self, rows: list[int]) -> None:
        self.rows = rows
        self._total_lock = Lock()

    @cached_property
    def total(self) -> int:
        with self._total_lock:
            return sum(self.rows)
```

이 계산처럼 여러 번 실행돼도 결과와 외부 상태가 안전하다면 잠금이 필요 없습니다. 파일 생성·결제처럼 중복 실행이 위험하다면 캐시가 아니라 작업 자체의 멱등성 키와 별도 동기화로 중복을 막습니다. 잠금은 오래 걸리는 I/O를 잡고 있지 않도록 범위를 작게 설계합니다.

## singledispatch를 플러그인 경계로 쓰려면 무엇을 확인할까요?

여러 모듈이 새 입력 타입을 추가해야 할 때 `singledispatch`의 `register()`를 확장 지점으로 사용할 수 있습니다. 기본 구현은 알 수 없는 입력을 조용히 문자열로 바꾸지 않고 오류로 만들어, 계약 밖 데이터가 지나가지 않게 합니다.

```python
from dataclasses import dataclass
from functools import singledispatch

@dataclass(frozen=True)
class PaidOrder:
    order_id: str
    amount: int

@singledispatch
def serialize(value: object) -> dict[str, object]:
    raise TypeError(f'unsupported value: {type(value).__name__}')

@serialize.register
def _(value: PaidOrder) -> dict[str, object]:
    return {'id': value.order_id, 'amount': value.amount}

print(serialize(PaidOrder('A-1', 12000)))
```

```text
{'id': 'A-1', 'amount': 12000}
```

디스패치는 첫 번째 인자의 타입과 상속 관계를 따라 가장 알맞은 구현을 고릅니다. 새 등록 함수는 별도 단위 테스트로 직접 검증하고, `serialize.dispatch(PaidOrder)`로 실제 선택될 구현도 확인할 수 있습니다. 등록 순서가 아니라 타입 관계가 선택을 결정한다는 점을 기억합니다.
