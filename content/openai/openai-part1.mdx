## OpenAI Python SDK는 언제 쓰는가?

Python 서버나 명령줄 프로그램에서 모델에게 텍스트를 요청하고, 그 결과를 다음 처리에 연결할 때 공식 `openai` 라이브러리를 사용합니다. HTTP 요청과 인증 헤더를 직접 만들지 않고 `OpenAI` 클라이언트와 타입이 있는 응답 객체를 사용하므로, 텍스트 생성·구조화된 결과·스트리밍을 같은 방식으로 다룰 수 있습니다.

API 키는 브라우저나 모바일 앱에 넣지 않습니다. 키를 가진 Python 서버가 요청을 보내고, 클라이언트 화면에는 서버가 필요한 결과만 돌려주는 구조로 만듭니다.

## pip install openai와 OPENAI_API_KEY로 실행 환경 준비하기

처음 API 요청을 실행할 때는 가상 환경에 라이브러리를 설치하고, API 키를 환경 변수에 둡니다. `OpenAI()`는 기본적으로 `OPENAI_API_KEY`를 읽으므로 키를 Python 코드에 적을 필요가 없습니다.

```sh
python -m venv .venv
source .venv/bin/activate
python -m pip install openai
export OPENAI_API_KEY="your_api_key_here"
```

키 값은 `.env` 같은 로컬 환경 파일이나 배포 플랫폼의 비밀 설정에만 두고, 저장소에는 커밋하지 않습니다. 새 터미널을 열었다면 환경 변수가 다시 설정되어 있는지도 확인합니다.

## OpenAI()로 한 번만 만든 클라이언트 재사용하기

여러 요청을 보내는 프로그램에서는 `OpenAI()`를 작업마다 새로 만들기보다 애플리케이션 시작 시 한 번 만들고 전달합니다. 기본 동기 클라이언트는 일반 Python 함수와 함께 쓰기 좋습니다.

```python
from openai import OpenAI

client = OpenAI()
```

명시적으로 `api_key=`를 줄 수도 있지만, 코드와 키를 분리할 수 있는 환경 변수 방식이 기본 선택입니다. 웹 요청마다 클라이언트를 만드는 대신 모듈이나 애플리케이션 수명 범위에서 재사용합니다.

## responses.create로 첫 텍스트 응답 요청하기

질문 하나에 대한 텍스트 응답이 필요할 때 `client.responses.create()`에 모델과 `input`을 전달합니다. 현재 예제의 `gpt-5.6`은 공식 문서의 새 프로젝트 예시 모델이며, 계정에서 사용할 수 있는 모델은 모델 목록에서 확인합니다.

```python
from openai import OpenAI

MODEL = "gpt-5.6"
client = OpenAI()

response = client.responses.create(
    model=MODEL,
    input="Python의 list와 tuple 차이를 두 문장으로 설명해 줘.",
)

print(response.output_text)
```

응답 문장은 모델과 입력에 따라 달라지므로 고정된 출력과 비교하지 않습니다. 실행하려면 유효한 API 키와 모델 사용 권한이 필요하며, 요청 자체에는 비용이 발생할 수 있습니다.

## output_text로 텍스트 결과만 읽기

텍스트를 화면에 표시하거나 다음 함수에 넘길 때는 `response.output_text`를 읽습니다. `response.output`에는 텍스트 외에도 도구 호출이나 추론 관련 항목이 섞일 수 있으므로, `output[0]`처럼 내부 위치를 가정하지 않습니다.

```python
from openai import OpenAI

client = OpenAI()
response = client.responses.create(
    model="gpt-5.6",
    input="함수 이름을 짓는 짧은 원칙 세 가지를 알려 줘.",
)

answer = response.output_text
print(answer)
```

`output_text`는 SDK가 모든 텍스트 출력을 합쳐 제공하는 편의 속성입니다. 다음 Part에서 도구 호출을 처리할 때만 `output`을 순회해 항목의 `type`을 확인합니다.

## instructions로 응답의 역할과 기준 정하기

번역 규칙이나 답변 형식을 여러 요청에 일관되게 적용할 때 `instructions`에 높은 수준의 역할과 제약을 둡니다. 사용자의 구체적인 질문은 `input`에 두어 역할과 요청을 분리합니다.

```python
from openai import OpenAI

client = OpenAI()
response = client.responses.create(
    model="gpt-5.6",
    instructions=(
        "당신은 Python 학습 도우미다. 한국어로 답하고, "
        "예시는 최대 한 개만 제시한다."
    ),
    input="dict.get()은 언제 쓰나요?",
)

print(response.output_text)
```

`instructions`는 말투, 목표, 올바른 답의 예처럼 응답 전체에 적용할 기준에 사용합니다. 요청마다 바뀌는 데이터나 질문까지 긴 지시문에 섞으면 수정과 테스트가 어려워집니다.

## input의 developer와 user 역할로 요청 나누기

대화 이력이나 역할별 메시지를 함께 보낼 때는 문자열 대신 `input` 배열을 사용합니다. 애플리케이션이 정한 동작은 `developer`, 사용자가 입력한 내용은 `user` 역할로 분리하면 어떤 내용이 어디서 왔는지 코드에서 분명해집니다.

```python
from openai import OpenAI

client = OpenAI()
response = client.responses.create(
    model="gpt-5.6",
    input=[
        {
            "role": "developer",
            "content": "답변은 목록 세 개와 한 문장 결론으로 구성한다.",
        },
        {
            "role": "user",
            "content": "Python에서 파일을 읽을 때 주의할 점을 알려 줘.",
        },
    ],
)

print(response.output_text)
```

신뢰하지 않는 사용자 입력을 `developer` 역할에 넣지 않습니다. 역할은 출처와 책임을 표현하는 도구이며, 입력 검증과 권한 검사를 대신하지는 않습니다.

## 모델 이름을 상수로 두고 교체하기

프로토타입을 운영 모델로 바꾸거나 모델 스냅샷을 고정할 때는 모델 이름을 한곳에 둡니다. 호출마다 문자열을 반복하면 모델 교체와 결과 비교가 어려워집니다.

```python
from openai import OpenAI

MODEL = "gpt-5.6"

def ask_python_question(client: OpenAI, question: str) -> str:
    response = client.responses.create(
        model=MODEL,
        input=question,
    )
    return response.output_text


client = OpenAI()
print(ask_python_question(client, "set은 어떤 중복을 제거하나요?"))
```

개발 중에는 최신 별칭으로 실험할 수 있지만, 일관된 동작을 측정해야 하는 운영 환경에서는 특정 스냅샷과 평가 결과를 함께 관리합니다. 모델의 지원 기능과 이름은 바뀔 수 있으므로 배포 전에 공식 모델 목록을 다시 확인합니다.

Python SDK 설치·환경 변수·첫 Responses API 호출은 [OpenAI 개발자 빠른 시작](https://developers.openai.com/api/docs/quickstart), `output_text`와 역할별 입력 규칙은 [텍스트 생성 가이드](https://developers.openai.com/api/docs/guides/text), 모델 선택은 [모델 목록](https://developers.openai.com/api/docs/models)을 기준으로 확인합니다.
