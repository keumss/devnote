title: "Part 2. 중급: 데이터 모양과 값 범위 정확히 표현하기"
---

기본 타입만으로는 딕셔너리의 필수 키, 상태 값의 범위, 같은 문자열로 표현된 서로 다른 식별자를 구분하기 어렵습니다. 이 노트에서는 데이터가 가질 수 있는 모양과 값을 더 좁게 표현하고, 조건문과 재사용 가능한 검사 함수가 정적 타입 정보로 이어지게 만듭니다.

## TypedDict로 딕셔너리의 키와 값 타입 선언하기

JSON과 닮은 딕셔너리 구조를 유지하면서 키별 타입을 알리고 싶을 때 `TypedDict`를 사용합니다. 런타임 값은 일반 `dict`이지만 정적 타입 검사기는 누락된 키와 잘못된 값 타입을 확인할 수 있습니다.

```python
from typing import TypedDict


class ProductRow(TypedDict):
    sku: str
    price: int
    stock: int


def inventory_value(product: ProductRow) -> int:
    return product['price'] * product['stock']


keyboard: ProductRow = {'sku': 'KB-01', 'price': 59_000, 'stock': 3}
print(inventory_value(keyboard))

# broken: ProductRow = {'sku': 'KB-02', 'price': '59000', 'stock': 3}
# 정적 타입 검사: price는 int여야 하므로 오류
```

```text
177000
```

`TypedDict`는 값을 검증하거나 변환하지 않습니다. 외부에서 받은 딕셔너리는 키와 값의 실제 타입을 확인한 뒤 `ProductRow`로 만들어야 합니다.

## Required·NotRequired로 필수 키와 선택 키 구분하기

대부분의 키가 선택 사항이거나 일부 키만 생략 가능할 때 `Required`와 `NotRequired`를 사용합니다. 키가 없는 경우와 값이 `None`인 경우를 구분할 수 있습니다.

```python
from typing import NotRequired, Required, TypedDict


class ProductPatch(TypedDict, total=False):
    sku: Required[str]
    name: NotRequired[str]
    stock: NotRequired[int]


def changed_fields(patch: ProductPatch) -> list[str]:
    return [key for key in ('name', 'stock') if key in patch]


patch: ProductPatch = {'sku': 'KB-01', 'stock': 7}
print(changed_fields(patch))
```

```text
['stock']
```

`total=False`는 기본적으로 모든 키를 선택 사항으로 만들고, `Required[str]`인 `sku`만 반드시 요구합니다. 값 자체로 `None`도 허용하려면 `str | None`처럼 별도로 선언합니다.

## Literal로 허용하는 값의 범위 제한하기

주문 상태처럼 정해진 문자열만 받아야 할 때 `Literal`로 가능한 값을 나열합니다. 단순한 `str`보다 분기와 잘못된 호출을 더 정확하게 검사할 수 있습니다.

```python
from typing import Literal


type OrderStatus = Literal['pending', 'paid', 'cancelled']


def can_cancel(status: OrderStatus) -> bool:
    return status == 'pending'


print(can_cancel('pending'))
print(can_cancel('paid'))

# can_cancel('shipped')
# 정적 타입 검사: 허용된 리터럴이 아니므로 오류
```

```text
True
False
```

`Literal`도 실행 중에 값을 막지는 않습니다. 외부 문자열은 허용 목록과 비교해 검증한 뒤 `OrderStatus`로 전달합니다.

## NewType으로 같은 기반 타입의 식별자 구분하기

고객 ID와 주문 ID가 모두 문자열이어도 서로 바뀌면 안 된다면 `NewType`으로 별개의 정적 타입을 만듭니다. 런타임에는 원래 문자열을 그대로 사용하므로 작은 비용으로 논리적 혼동을 줄일 수 있습니다.

```python
from typing import NewType


CustomerId = NewType('CustomerId', str)
OrderId = NewType('OrderId', str)


def order_key(customer_id: CustomerId, order_id: OrderId) -> str:
    return f'{customer_id}:{order_id}'


print(order_key(CustomerId('C-10'), OrderId('O-21')))

# order_key(OrderId('O-21'), CustomerId('C-10'))
# 정적 타입 검사: 두 식별자의 위치가 바뀌었으므로 오류
```

```text
C-10:O-21
```

`type CustomerId = str`은 별칭이라 원래 문자열과 동일하게 취급됩니다. 서로 구분해야 하는 값에는 `NewType`, 긴 타입을 줄이는 데는 타입 별칭을 사용합니다.

## Final·ClassVar로 상수와 클래스 변수를 구분하기

재할당하지 않을 값은 `Final`, 모든 인스턴스가 공유하고 인스턴스 필드가 아닌 값은 `ClassVar`로 표시합니다. 두 표시는 값의 역할을 명확히 하며 정적 검사기가 잘못된 대입을 찾게 합니다.

```python
from typing import ClassVar, Final


TAX_RATE: Final[float] = 0.1


class Product:
    category: ClassVar[str] = 'goods'

    def __init__(self, price: int) -> None:
        self.price = price

    def price_with_tax(self) -> int:
        return int(self.price * (1 + TAX_RATE))


keyboard = Product(59_000)
print(Product.category)
print(keyboard.price_with_tax())

# TAX_RATE = 0.2
# keyboard.category = 'device'
# 정적 타입 검사: Final 재할당과 인스턴스를 통한 ClassVar 변경은 오류
```

```text
goods
64900
```

`Final`과 `ClassVar`는 런타임 불변성을 강제하지 않습니다. 실제 변경을 막아야 한다면 읽기 전용 프로퍼티나 불변 객체 같은 실행 시점 설계를 함께 사용합니다.

## @override로 부모 메서드 재정의 의도 확인하기

부모 클래스의 메서드를 재정의할 때 Python 3.12의 `@override`를 붙이면 이름을 잘못 쓰거나 부모 계약이 바뀐 경우를 정적 검사에서 찾을 수 있습니다. 데코레이터는 재정의 의도를 코드에 남깁니다.

```python
from typing import override


class Formatter:
    def format(self, value: int) -> str:
        return str(value)


class WonFormatter(Formatter):
    @override
    def format(self, value: int) -> str:
        return f'{value}원'


formatter: Formatter = WonFormatter()
print(formatter.format(59_000))
```

```text
59000원
```

`@override`는 호환되지 않는 매개변수나 반환 타입도 타입 검사기가 확인하도록 돕습니다. 실행 중에는 부모 메서드의 존재나 시그니처를 검증하지 않습니다.

## isinstance로 유니온 타입 좁히기

서로 다른 타입을 받을 수 있는 함수에서 각 타입의 연산을 안전하게 사용하려면 `isinstance()`로 분기합니다. 정적 타입 검사기는 분기 안에서 유니온을 해당 타입으로 좁힙니다.

```python
def normalize_quantity(value: int | str) -> int:
    if isinstance(value, int):
        return value

    stripped = value.strip()
    if not stripped.isdigit():
        raise ValueError('수량은 정수여야 합니다.')
    return int(stripped)


print(normalize_quantity(3))
print(normalize_quantity(' 12 '))
```

```text
3
12
```

분기 전에는 `int`와 `str`에 공통으로 허용되는 동작만 사용할 수 있습니다. 확인과 변환을 경계에 모으면 이후 코드는 `int` 하나만 다루게 됩니다.

## TypeGuard로 반복 검증을 타입 좁히기 함수로 분리하기

여러 곳에서 같은 복합 조건을 확인한다면 반환 타입을 `TypeGuard[T]`로 선언해 재사용합니다. 조건이 참인 분기에서 검사 대상이 `T`라는 정보를 정적 타입 검사기에 전달합니다.

```python
from typing import TypeGuard


def is_text_tuple(values: tuple[object, ...]) -> TypeGuard[tuple[str, ...]]:
    return all(isinstance(value, str) for value in values)


def join_tags(values: tuple[object, ...]) -> str:
    if not is_text_tuple(values):
        raise TypeError('모든 태그는 문자열이어야 합니다.')
    return ','.join(value.upper() for value in values)


print(join_tags(('new', 'sale')))
```

```text
NEW,SALE
```

`TypeGuard` 함수의 구현이 실제 조건과 다르면 잘못된 타입 정보를 만들 수 있습니다. 반환 타입만 믿지 말고 함수 본문에서 약속한 조건을 빠짐없이 검사합니다.

공식 Python 3.12 [`typing` 문서](https://docs.python.org/3.12/library/typing.html)의 `TypedDict`, `Literal`, `NewType`, `Final`, `ClassVar`, `override`, `TypeGuard` 동작을 기준으로 구성했습니다.
