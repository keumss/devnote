## SQLModel이 줄여 주는 중복과 남겨야 하는 경계 이해하기

SQLModel은 SQLAlchemy의 테이블 매핑과 Pydantic의 데이터 검증을 한 모델 계층에서 사용할 수 있게 합니다. `table=True`인 클래스는 DB 테이블에 대응하고, `table=True`가 없는 클래스는 요청·응답처럼 DB에 저장하지 않는 데이터 형식을 표현합니다.

그렇다고 테이블 모델 하나를 입력과 응답에 그대로 쓰면 안 됩니다. 생성 입력에 DB 기본 키가 노출되거나, 테이블에만 필요한 비밀 필드가 응답에 포함될 수 있기 때문입니다. 공통 필드는 상속하되 생성 입력, 테이블, 공개 응답의 책임은 분리하는 것이 이 섹션의 기본 원칙입니다.

이 노트는 Python 3.12와 SQLModel 0.0.38을 기준으로 합니다. 가상 환경에서 다음처럼 설치하고 버전을 확인할 수 있습니다.

```sh
python -m pip install 'sqlmodel==0.0.38'
python -c 'import sqlmodel; print(sqlmodel.__version__)'
```

```text
0.0.38
```

## 생성 입력, 테이블, 공개 응답 모델 분리하기

공통 필드는 `HeroBase`에 두고, 각 경계에 필요한 필드만 하위 모델에 추가합니다. `HeroCreate`는 클라이언트가 보낼 필드만 받고, `HeroPublic`은 DB에서 생성된 `id`를 포함하되 `secret_name`은 제외합니다.

```python
from pydantic import ConfigDict
from sqlmodel import Field, SQLModel


class HeroBase(SQLModel):
    name: str = Field(index=True, min_length=1, max_length=50)
    age: int | None = Field(default=None, ge=0)


class Hero(HeroBase, table=True):
    id: int | None = Field(default=None, primary_key=True)
    secret_name: str = Field(min_length=1, max_length=50)


class HeroCreate(HeroBase):
    model_config = ConfigDict(extra='forbid')

    secret_name: str = Field(min_length=1, max_length=50)


class HeroPublic(HeroBase):
    id: int


payload = HeroCreate(name='Deadpond', secret_name='Dive Wilson')
print(payload.model_dump())
```

```text
{'name': 'Deadpond', 'age': None, 'secret_name': 'Dive Wilson'}
```

`extra='forbid'`는 `id`처럼 생성 스키마에 없는 필드를 조용히 무시하지 않고 검증 오류로 처리합니다. `min_length`, `max_length`, `ge`는 애플리케이션 입력을 검증하지만 DB 제약을 대신하지 않으므로 운영 스키마의 `NOT NULL`, `UNIQUE`, `CHECK`도 별도로 설계해야 합니다.

## 인메모리 SQLite 엔진과 테이블 생성 확인하기

학습 예제는 실행할 때마다 초기화되는 인메모리 SQLite를 사용합니다. 모든 테이블 모델을 정의한 뒤 `SQLModel.metadata.create_all()`을 호출해야 메타데이터에 등록된 테이블이 생성됩니다.

```python
from sqlalchemy import inspect
from sqlmodel import create_engine
from sqlmodel.pool import StaticPool


engine = create_engine('sqlite://', poolclass=StaticPool, echo=False)
SQLModel.metadata.create_all(engine)

print(inspect(engine).get_table_names())
```

```text
['hero']
```

`StaticPool`은 예제 전체가 같은 인메모리 DB 연결을 재사용하게 합니다. 파일 DB를 사용할 때는 `sqlite:///database.db`처럼 경로를 지정합니다. `create_all()`은 없는 테이블을 만드는 학습·테스트 용도에는 편하지만 기존 운영 컬럼을 안전하게 변경하지 않으므로, 운영 스키마 변경에는 Alembic 리비전을 사용합니다.

## Session.add와 refresh로 공개 가능한 생성 결과 확인하기

입력 모델을 테이블 객체로 검증·변환한 뒤 세션에 추가합니다. `commit()`은 트랜잭션을 확정하고, `refresh()`는 DB가 만든 기본 키 같은 값을 객체에 다시 읽어 옵니다. 마지막에는 공개 응답 모델로 변환해 비밀 필드가 제외되는지 확인합니다.

```python
from sqlmodel import Session


hero_data = HeroCreate(name='Deadpond', secret_name='Dive Wilson')
hero = Hero.model_validate(hero_data)

with Session(engine) as session:
    session.add(hero)
    session.commit()
    session.refresh(hero)

    public_hero = HeroPublic.model_validate(hero)
    print(public_hero.model_dump())
```

```text
{'name': 'Deadpond', 'age': None, 'id': 1}
```

세션 컨텍스트는 사용이 끝난 연결을 반환하지만, 실패한 트랜잭션을 계속 사용할 계획이라면 예외를 처리하면서 `session.rollback()`을 호출해야 합니다. `secret_name`은 테이블에는 남아 있지만 `HeroPublic`에 선언하지 않았으므로 공개 결과에는 포함되지 않습니다.
