## SQLModel의 역할과 학습 범위 이해하기

SQLModel은 SQLAlchemy의 테이블 매핑과 Pydantic의 타입 기반 데이터 모델을 같은 클래스 문법으로 사용할 수 있게 합니다. 앞선 SQLAlchemy 섹션에서 배운 엔진, 세션, 트랜잭션은 그대로 동작하며, SQLModel은 모델 선언과 타입이 명확한 조회 API를 더합니다.

이 섹션은 Python 3.12와 SQLModel 0.0.38을 기준으로 합니다. 가상 환경에서 설치하고 버전을 확인합니다.

```sh
python -m pip install 'sqlmodel==0.0.38'
python -c 'import sqlmodel; print(sqlmodel.__version__)'
```

```text
0.0.38
```

Part 1에서는 하나의 파일을 위에서 아래로 실행하며 테이블 생성부터 CRUD까지 빠르게 완성합니다. 입력·응답 모델 분리와 오류 처리는 Part 2에서 같은 흐름을 안전하게 확장합니다.

## table과 Field로 최소 테이블 모델 선언하기

`SQLModel`을 상속하고 `table=True`를 지정하면 클래스가 DB 테이블에 매핑됩니다. `Field()`는 기본 키, 인덱스처럼 DB 컬럼에 필요한 정보를 타입 힌트에 덧붙입니다.

```python
from sqlmodel import Field, SQLModel


class Hero(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(index=True, max_length=50)
    secret_name: str = Field(max_length=50)
    age: int | None = Field(default=None, ge=0)


print(Hero.__tablename__)
```

```text
hero
```

`id`는 INSERT 전에는 없고 DB가 생성한 뒤 채우므로 `int | None`입니다. `age`는 값이 없어도 되는 선택 필드입니다. `max_length`와 `ge` 같은 애플리케이션 검증이 DB의 `UNIQUE`, `CHECK` 제약을 자동으로 대신하지는 않습니다.

## Engine과 metadata로 SQLite 테이블 만들기

Engine은 DB 연결 설정을 관리하고, `SQLModel.metadata`는 지금까지 선언한 테이블 정보를 모읍니다. 모든 테이블 모델을 선언한 뒤 `create_all()`을 호출해야 등록된 테이블이 생성됩니다.

```python
from sqlalchemy import inspect
from sqlmodel import create_engine
from sqlmodel.pool import StaticPool


engine = create_engine('sqlite://', poolclass=StaticPool, echo=False)
SQLModel.metadata.create_all(engine)

print(inspect(engine).get_table_names())
```

```text
['hero']
```

인메모리 SQLite와 `StaticPool`을 사용해 예제 전체가 같은 연결을 재사용하게 했습니다. `create_all()`은 없는 테이블을 만드는 학습·테스트 도구이며, 이미 운영 중인 테이블의 컬럼을 변경할 때는 Alembic 리비전을 사용합니다.

## add, commit, refresh로 행 생성하기

`Session`은 조회와 쓰기를 하나의 트랜잭션 경계에서 관리합니다. 객체를 `add()`하고 `commit()`하면 INSERT가 확정되며, `refresh()`는 DB가 만든 기본 키처럼 새로 생긴 값을 다시 읽습니다.

```python
from sqlmodel import Session


with Session(engine) as session:
    heroes = [
        Hero(name='Deadpond', secret_name='Dive Wilson'),
        Hero(name='Spider-Boy', secret_name='Pedro Parqueador', age=15),
        Hero(name='Rusty-Man', secret_name='Tommy Sharp', age=48),
    ]
    session.add_all(heroes)
    session.commit()
    session.refresh(heroes[0])

    print(heroes[0].id, heroes[0].name)
```

```text
1 Deadpond
```

세션 컨텍스트가 연결을 반환하는 것과 트랜잭션을 성공으로 확정하는 것은 별개입니다. 쓰기를 DB에 남기려면 `commit()`이 필요합니다.

## select와 Session.exec로 원하는 행 조회하기

SQLModel의 `select()`로 조회 문장을 만들고 `Session.exec()`로 실행합니다. 모델 하나를 선택하면 `.all()`은 `Hero` 객체 목록을 반환하므로 별도의 `scalars()` 호출 없이 속성에 바로 접근할 수 있습니다.

```python
from sqlmodel import select


with Session(engine) as session:
    statement = (
        select(Hero)
        .where(Hero.age >= 18)
        .order_by(Hero.id)
    )
    adult_heroes = session.exec(statement).all()

    print([(hero.id, hero.name) for hero in adult_heroes])
```

```text
[(3, 'Rusty-Man')]
```

조건이 없는 `select(Hero)`는 모든 행을 읽습니다. 결과가 많아질 수 있는 실제 API에서는 정렬과 `limit()`을 함께 적용합니다.

## 변경 감지와 delete로 수정·삭제 완료하기

세션에서 조회한 객체의 속성을 바꾸면 SQLAlchemy의 변경 감지가 UPDATE를 준비합니다. 다시 `add()`하지 않아도 `commit()`할 때 변경이 반영됩니다. 삭제는 `delete()`한 뒤 같은 방식으로 커밋합니다.

```python
with Session(engine) as session:
    spider_boy = session.exec(
        select(Hero).where(Hero.name == 'Spider-Boy')
    ).one()
    spider_boy.name = 'Spider-Man'
    spider_boy.age = 16

    rusty_man = session.exec(
        select(Hero).where(Hero.name == 'Rusty-Man')
    ).one()
    session.delete(rusty_man)
    session.commit()

with Session(engine) as session:
    remaining = session.exec(select(Hero).order_by(Hero.id)).all()
    print([(hero.name, hero.age) for hero in remaining])
```

```text
[('Deadpond', None), ('Spider-Man', 16)]
```

서로 함께 성공해야 하는 수정과 삭제를 한 번의 `commit()`으로 묶었습니다. 중간 SQL이 실패하면 전체 트랜잭션을 rollback해야 하며, 그 흐름은 Part 2에서 DB 제약 오류와 함께 다룹니다.
