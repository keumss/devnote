## foreign_key와 양방향 Relationship 함께 정의하기

`foreign_key`는 DB 컬럼 사이의 참조 무결성을 정의하고, `Relationship()`은 Python 객체에서 연관 객체를 탐색하게 합니다. 둘은 역할이 다르므로 외래 키만으로 `team.heroes`가 생기거나 관계 속성만으로 DB 제약이 생기지는 않습니다. 이 노트는 아래 Team–Hero 모델을 같은 파일에서 이어서 사용합니다.

```python
from sqlalchemy import event
from sqlmodel import Field, Relationship, Session, SQLModel, create_engine, select
from sqlmodel.pool import StaticPool


class Team(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(index=True, unique=True)

    heroes: list['Hero'] = Relationship(
        back_populates='team',
        cascade_delete=True,
    )


class Hero(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(index=True, unique=True)
    secret_name: str
    team_id: int | None = Field(
        default=None,
        foreign_key='team.id',
        ondelete='CASCADE',
    )

    team: Team | None = Relationship(back_populates='heroes')


engine = create_engine('sqlite://', poolclass=StaticPool)


@event.listens_for(engine, 'connect')
def enable_sqlite_foreign_keys(dbapi_connection, connection_record):
    cursor = dbapi_connection.cursor()
    cursor.execute('PRAGMA foreign_keys=ON')
    cursor.close()


SQLModel.metadata.create_all(engine)

with Session(engine) as session:
    z_force = Team(
        name='Z-Force',
        heroes=[Hero(name='Deadpond', secret_name='Dive Wilson')],
    )
    preventers = Team(
        name='Preventers',
        heroes=[Hero(name='Rusty-Man', secret_name='Tommy Sharp')],
    )
    solo = Hero(name='Solo', secret_name='Unknown')
    session.add_all([z_force, preventers, solo])
    session.commit()

    print(z_force.heroes[0].team is z_force)
```

```text
True
```

`back_populates`는 양쪽 속성이 같은 관계임을 알려 메모리 안의 변경도 동기화합니다. SQLite는 외래 키 제약이 연결마다 기본 비활성화되어 있으므로 `connect` 이벤트에서 `PRAGMA foreign_keys=ON`을 적용했습니다.

## 지연 로딩과 세션 수명이 만드는 N+1 측정하기

기본 관계 속성은 처음부터 채워져 있지 않습니다. 로딩되지 않은 `team.heroes`에 접근하는 순간 SELECT가 실행되며, 부모 N개마다 쿼리가 하나씩 추가될 수 있습니다. 다음 예제는 실제 SELECT 횟수를 세어 팀 목록 1회와 두 팀의 관계 조회 2회를 확인합니다.

```python
select_count = 0


def count_selects(conn, cursor, statement, parameters, context, executemany):
    global select_count
    if statement.lstrip().upper().startswith('SELECT'):
        select_count += 1


event.listen(engine, 'before_cursor_execute', count_selects)

with Session(engine) as session:
    teams = session.exec(select(Team).order_by(Team.id)).all()
    summary = [(team.name, len(team.heroes)) for team in teams]

event.remove(engine, 'before_cursor_execute', count_selects)

print(summary)
print('SELECT count:', select_count)
```

```text
[('Z-Force', 1), ('Preventers', 1)]
SELECT count: 3
```

세션이 닫힌 뒤 아직 읽지 않은 관계에 접근하면 분리된 객체 오류가 발생할 수 있습니다. 응답에 필요한 관계는 세션 안에서 명시적으로 로딩하고, 직렬화를 위해 세션을 불필요하게 오래 열어 두지 않습니다.

## 관계 조인과 로더 옵션을 목적에 맞게 선택하기

`join()`은 관계를 필터링하거나 여러 모델의 값을 같은 결과 행에서 사용할 때 적합합니다. 반면 로더 옵션은 주 모델의 결과 모양을 유지하면서 관계 속성을 미리 채웁니다. 컬렉션에는 보통 `selectinload()`, 여러 행이 하나의 부모를 가리키는 N:1 관계에는 `joinedload()`를 먼저 고려합니다.

```python
from sqlalchemy.orm import joinedload, selectinload


with Session(engine) as session:
    joined_rows = session.exec(
        select(Hero, Team)
        .join(Hero.team)
        .order_by(Hero.id)
    ).all()
    print([(hero.name, team.name) for hero, team in joined_rows])

    teams = session.exec(
        select(Team)
        .options(selectinload(Team.heroes))
        .order_by(Team.id)
    ).all()
    print([(team.name, [hero.name for hero in team.heroes]) for team in teams])

    heroes = session.exec(
        select(Hero)
        .options(joinedload(Hero.team))
        .order_by(Hero.id)
    ).all()
    print([(hero.name, hero.team.name if hero.team else None) for hero in heroes])
```

```text
[('Deadpond', 'Z-Force'), ('Rusty-Man', 'Preventers')]
[('Z-Force', ['Deadpond']), ('Preventers', ['Rusty-Man'])]
[('Deadpond', 'Z-Force'), ('Rusty-Man', 'Preventers'), ('Solo', None)]
```

조인은 팀이 없는 `Solo`를 제외하지만, `joinedload(Hero.team)`은 Hero 목록을 바꾸지 않고 `team` 속성만 채웁니다. 실제 로더 선택은 쿼리 횟수뿐 아니라 관계 크기와 전송되는 중복 행도 함께 측정해 결정합니다.

## cascade_delete, ondelete, passive_deletes 조합하기

연관 삭제 정책은 Python ORM과 DB 양쪽의 동작을 함께 설계해야 합니다. 현재 모델의 `cascade_delete=True`는 SQLAlchemy가 자식 객체를 삭제하게 하고, `ondelete='CASCADE'`는 ORM을 거치지 않은 부모 삭제도 DB가 처리하게 합니다.

```python
from sqlalchemy import inspect as sqlalchemy_inspect


relationship = sqlalchemy_inspect(Team).relationships.heroes
foreign_key = next(iter(Hero.__table__.c.team_id.foreign_keys))

print('DB on delete:', foreign_key.ondelete)
print('ORM deletes:', 'delete' in relationship.cascade)
print('ORM delete-orphan:', 'delete-orphan' in relationship.cascade)
```

```text
DB on delete: CASCADE
ORM deletes: True
ORM delete-orphan: True
```

도메인 정책에 따라 관계와 외래 키를 다음처럼 함께 바꿉니다.

| 정책 | `Relationship()` | 외래 키 필드 | 결과 |
| --- | --- | --- | --- |
| 자식도 삭제 | `cascade_delete=True` | `ondelete='CASCADE'` | ORM과 DB 직접 삭제 모두 자식 삭제 |
| 자식은 보존 | `passive_deletes='all'` | nullable 필드와 `ondelete='SET NULL'` | DB가 외래 키를 `NULL`로 변경 |
| 부모 삭제 금지 | `passive_deletes='all'` | `ondelete='RESTRICT'` | 자식이 있으면 DB가 부모 삭제 거부 |

`SET NULL`이나 `RESTRICT`로 바꾸면서 `cascade_delete=True`를 남겨 두면 ORM이 자식을 먼저 삭제해 의도한 DB 정책이 실행되지 않습니다. 반대로 `passive_deletes='all'`은 SQLAlchemy가 외래 키를 먼저 `NULL`로 갱신하는 기본 동작을 막고 DB 제약에 결정을 맡깁니다.

## ORM 삭제와 DB 직접 삭제를 모두 검증하기

ORM 경로와 DB 제약 경로가 같은 결과를 만드는지 각각 확인합니다. 첫 삭제는 객체와 관계를 로딩하는 `Session.delete()`를 사용하고, 두 번째는 ORM 단위 작업을 우회하는 SQL DELETE를 실행합니다.

```python
from sqlalchemy import delete


with Session(engine) as session:
    preventers = session.exec(
        select(Team).where(Team.name == 'Preventers')
    ).one()
    session.delete(preventers)
    session.commit()

with engine.begin() as connection:
    connection.execute(delete(Team).where(Team.name == 'Z-Force'))

with Session(engine) as session:
    rusty_man = session.exec(
        select(Hero).where(Hero.name == 'Rusty-Man')
    ).first()
    deadpond = session.exec(
        select(Hero).where(Hero.name == 'Deadpond')
    ).first()
    solo = session.exec(
        select(Hero).where(Hero.name == 'Solo')
    ).one()

    print(rusty_man, deadpond)
    print(solo.name, solo.team_id)
```

```text
None None
Solo None
```

모델 코드의 `ondelete`를 바꾸는 것만으로 이미 생성된 운영 제약은 변경되지 않습니다. Alembic으로 외래 키 제약을 마이그레이션하고, SQLite를 사용한다면 외래 키 활성화 여부까지 포함해 세 정책의 성공·실패 결과를 테스트합니다.
