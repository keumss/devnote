## 개발·운영 서버와 프로세스 모델 구분하기

`fastapi dev`는 코드 변경 감지와 개발용 로그를 제공하지만 운영에서 사용하지 않습니다. 운영에서는 `fastapi run`으로 모든 인터페이스에 바인딩하고, 프록시나 플랫폼이 전달할 포트를 명시합니다.

```bash
# 개발
fastapi dev app/main.py

# 운영
fastapi run app/main.py --host 0.0.0.0 --port 8000
```

운영 서버 앞에서 TLS를 종료하고 전달 헤더를 설정하는 신뢰할 수 있는 프록시가 있을 때만 `--proxy-headers`를 추가합니다. 인터넷 클라이언트가 전달 헤더를 직접 위조할 수 없도록 신뢰 프록시 범위도 배포 환경에서 제한합니다.

단일 서버에서는 `--workers`로 여러 프로세스를 실행할 수 있지만 워커마다 메모리, 전역 변수, 연결 풀이 따로 존재합니다. 워커 수를 두 배로 늘리면 모델과 연결 풀 메모리도 대체로 함께 늘어납니다.

```bash
fastapi run app/main.py --host 0.0.0.0 --port 8000 --workers 4
```

인메모리 딕셔너리, 로그인 세션, 작업 상태를 워커 사이의 공유 저장소로 사용하면 안 됩니다. DB나 Redis 같은 외부 저장소를 사용하고 실제 부하와 메모리를 측정해 워커 수를 정합니다.

Kubernetes처럼 복제와 로드 밸런싱을 플랫폼이 담당하면 일반적으로 컨테이너마다 프로세스 하나를 실행하고 컨테이너 수를 늘립니다. 단일 서버의 Docker Compose처럼 컨테이너 복제가 어려운 환경에서는 여러 워커가 합리적일 수 있습니다.

## 비루트 사용자로 Python 3.12 이미지 실행하기

의존성 파일을 애플리케이션 코드보다 먼저 복사하면 코드만 바뀐 빌드에서 설치 레이어를 재사용할 수 있습니다. exec-form `CMD`를 사용해야 종료 신호가 서버 프로세스에 전달되어 lifespan 정리가 실행됩니다.

```dockerfile
FROM python:3.12-slim

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

RUN addgroup --system app && adduser --system --ingroup app app

WORKDIR /code

COPY requirements.txt /code/requirements.txt
RUN pip install --no-cache-dir --upgrade -r /code/requirements.txt

COPY --chown=app:app app /code/app

USER app

CMD ["fastapi", "run", "app/main.py", "--host", "0.0.0.0", "--port", "8000"]
```

```bash
docker build -t fastapi-study .
docker image inspect fastapi-study --format '{{json .Config.Cmd}}'
```

```text
["fastapi","run","app/main.py","--host","0.0.0.0","--port","8000"]
```

`.dockerignore`에는 `.git`, `.venv`, `__pycache__`, 테스트 캐시, `.env`, 로컬 업로드 파일을 넣습니다. 업로드를 컨테이너 파일 시스템에만 저장하면 새 컨테이너나 여러 복제본에서 공유되지 않으므로 운영 파일은 외부 저장소에 둡니다.

## Compose에서 DB 준비 상태와 비밀값 주입하기

Compose의 실행 순서만으로는 DB가 연결을 받을 준비가 되었는지 알 수 없습니다. DB healthcheck가 성공한 뒤 API를 시작하고, 비밀번호가 없으면 설정 단계에서 실패하도록 필수 변수 문법을 사용합니다.

```yaml
services:
  db:
    image: postgres:18-alpine
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-app}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:?POSTGRES_PASSWORD is required}
      POSTGRES_DB: ${POSTGRES_DB:-app}
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB"]
      interval: 5s
      timeout: 3s
      retries: 10

  web:
    build: .
    ports:
      - "8000:8000"
    environment:
      DATABASE_URL: postgresql+psycopg://${POSTGRES_USER:-app}:${POSTGRES_PASSWORD}@db:5432/${POSTGRES_DB:-app}
      SECRET_KEY: ${SECRET_KEY:?SECRET_KEY is required}
    depends_on:
      db:
        condition: service_healthy
```

```bash
docker compose config --quiet
docker compose up --build --detach
docker compose ps
```

`config --quiet`가 출력 없이 상태 코드 0으로 끝나면 YAML과 변수 치환이 유효합니다. 예제의 환경변수 방식은 로컬 학습용이며, 운영에서는 파일이나 프로세스 목록에 노출되지 않는 플랫폼 secret 기능을 사용합니다.

## 마이그레이션을 API 워커 시작과 분리하기

lifespan에서 `create_all()`이나 마이그레이션을 실행하면 워커와 컨테이너 수만큼 동시에 실행될 수 있습니다. 배포 파이프라인이나 일회성 작업에서 스키마를 한 번 갱신한 뒤 API 복제본을 시작합니다.

```bash
# 배포 단계의 일회성 작업
alembic upgrade head

# 성공한 뒤 API 프로세스 시작
fastapi run app/main.py --host 0.0.0.0 --port 8000
```

마이그레이션 실패 시 새 API 버전을 시작하지 않고 중단합니다. 이전 애플리케이션과 새 스키마가 잠시 함께 동작해야 하는 무중단 배포에서는 컬럼 추가, 양쪽 버전 배포, 데이터 채우기, 이전 컬럼 제거 순서의 호환 가능한 변경을 사용합니다.

## 생존 상태와 준비 상태를 별도 엔드포인트로 제공하기

생존 확인은 프로세스가 HTTP 요청을 처리하는지만 빠르게 검사합니다. 준비 상태는 새 트래픽을 받아도 되는지 판단하며 DB 같은 필수 의존성을 짧은 타임아웃으로 확인할 수 있습니다.

```python
import asyncio

from fastapi import FastAPI, HTTPException

app = FastAPI()


async def database_is_ready() -> bool:
    # 실제 구현에서는 연결 풀로 SELECT 1 같은 가벼운 쿼리를 실행합니다.
    return True


@app.get("/healthz")
def healthz() -> dict[str, str]:
    return {"status": "ok"}


@app.get("/readyz")
async def readyz() -> dict[str, str]:
    try:
        async with asyncio.timeout(1):
            ready = await database_is_ready()
    except TimeoutError as exc:
        raise HTTPException(status_code=503, detail="Database timeout") from exc

    if not ready:
        raise HTTPException(status_code=503, detail="Database unavailable")
    return {"status": "ready"}
```

```bash
curl --fail --silent http://127.0.0.1:8000/healthz
curl --fail --silent http://127.0.0.1:8000/readyz
```

```json
{"status":"ok"}
{"status":"ready"}
```

생존 확인에 느린 외부 의존성을 모두 넣으면 일시적인 DB 지연이 API 프로세스 재시작 폭주로 이어질 수 있습니다. 준비 확인 실패는 트래픽에서 제외하되 프로세스 자체를 즉시 종료하지 않는 구성이 일반적입니다.

## 컨테이너 로그로 시작·연결·요청 장애 구분하기

배포 장애는 서버 시작 전 오류, 외부 의존성 연결 오류, 요청 처리 오류로 나누어 확인합니다. 로그는 표준 출력과 표준 오류로 보내 컨테이너 플랫폼이 수집하게 합니다.

```bash
docker compose ps
docker compose logs --tail=100 web
docker compose logs --tail=100 db
```

애플리케이션이 즉시 종료되면 import 경로, 필수 환경변수, 포트와 시작 명령을 먼저 확인합니다. 실행 중 5xx가 발생하면 요청 ID로 프록시와 애플리케이션 로그를 연결하고, DB 오류라면 연결 URL·풀 고갈·마이그레이션 상태를 확인합니다.

운영 준비가 끝났다는 기준은 컨테이너가 한 번 실행되는 것이 아니라 재시작, 종료 신호, 준비 상태 전환, 로그 수집, 비밀 교체, 롤백 절차를 반복해서 검증할 수 있는 것입니다.
