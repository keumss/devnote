## 라우터와 의존성을 하나의 애플리케이션으로 조립하기

앞선 노트의 모델, 저장소, 인증, 업로드 코드를 역할별 모듈에 둡니다. 진입점은 구체적인 처리 로직을 다시 구현하지 않고 애플리케이션 수명과 라우터 조합만 담당합니다.

```text
app/
├── main.py
├── settings.py
├── security.py
├── auth.py
├── models.py
├── repository.py
└── api/
    ├── posts.py
    └── uploads.py
tests/
├── conftest.py
└── test_api.py
```

Part 6에서 작성한 `/images` 경로는 `app/api/uploads.py`의 `router`로 옮기고, Part 5의 `/token` 경로도 인증 라우터로 노출합니다. 각 모듈의 공개 의존성 함수는 테스트 override에서 같은 함수 객체를 참조할 수 있게 유지합니다.

```python
# app/main.py
from contextlib import asynccontextmanager
from collections.abc import AsyncIterator

from fastapi import FastAPI

from app.api.posts import router as posts_router
from app.api.uploads import router as uploads_router
from app.auth import router as auth_router


@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    app.state.ready = True
    try:
        yield
    finally:
        app.state.ready = False


app = FastAPI(title="FastAPI Study", lifespan=lifespan)
app.include_router(auth_router)
app.include_router(posts_router)
app.include_router(uploads_router)
```

`/openapi.json`의 `paths`에서 `/token`, `/posts`, `/posts/search`, `/posts/{post_id}`, `/images`가 모두 보이면 라우터 조합이 완료된 것입니다.

## 로그인부터 게시물 삭제까지 한 흐름으로 확인하기

개발용 사용자 `alice`로 서버의 Swagger UI나 curl에서 다음 순서대로 호출합니다. 로그인 응답의 `access_token`은 이후 요청의 `Authorization: Bearer <token>` 헤더에 사용합니다.

```text
POST   /token             username=alice&password=correct-password → 200
POST   /posts             제목과 본문 생성                         → 201
GET    /posts/search?q=로그인                                  → 200
PATCH  /posts/1           작성자가 제목 수정                      → 200
DELETE /posts/1           작성자가 삭제                           → 204
GET    /posts/1           삭제 결과 확인                          → 404
```

토큰과 생성 시각은 요청마다 달라집니다. 실패하면 `/token`의 401, 라우터 등록, 요청 본문의 422, 리소스 소유권의 403 순서로 경계를 좁힙니다.

## 저장소·사용자·업로드 경로를 테스트마다 격리하기

테스트 fixture는 운영 DB, 실제 토큰 검증, 로컬 업로드 디렉터리에 접근하지 않게 세 경계를 교체합니다. 정상 기능 테스트에서는 고정 사용자를 주입하고, 인증 자체를 검사하는 테스트에서만 override를 제거합니다.

```python
# tests/conftest.py
from collections.abc import Iterator

import pytest
from fastapi.testclient import TestClient

from app.api import uploads
from app.api.posts import get_repository
from app.auth import UserRecord, get_current_user
from app.main import app
from app.repository import PostRepository


@pytest.fixture
def client(tmp_path, monkeypatch) -> Iterator[TestClient]:
    test_repository = PostRepository()
    test_user = UserRecord(username="alice", hashed_password="not-used")

    app.dependency_overrides[get_repository] = lambda: test_repository
    app.dependency_overrides[get_current_user] = lambda: test_user
    monkeypatch.setattr(uploads, "UPLOAD_DIR", tmp_path)

    try:
        with TestClient(app) as test_client:
            yield test_client
    finally:
        app.dependency_overrides.clear()
```

fixture마다 저장소와 임시 디렉터리가 새로 만들어지므로 테스트 순서와 로컬 파일 상태가 결과에 영향을 주지 않습니다. context manager 안에서 요청해야 lifespan의 시작과 종료도 함께 검증됩니다.

## 인증·인가·검증·업로드 실패를 회귀 테스트하기

정상 CRUD 흐름은 Part 4에서 확인했습니다. 통합 테스트에서는 모듈 경계를 연결할 때 자주 깨지는 인증 헤더, 소유권, 입력 검증, 업로드 정리를 집중해서 검사합니다.

```python
# tests/test_api.py
from fastapi.testclient import TestClient

from app.auth import get_current_user
from app.main import app


def test_validation_and_upload_cleanup(
    client: TestClient,
    tmp_path,
) -> None:
    assert client.get("/posts/not-an-integer").status_code == 422
    assert client.get("/posts", params={"limit": 101}).status_code == 422
    response = client.post(
        "/images",
        files={"file": ("payload.html", b"<script></script>", "text/html")},
    )
    assert response.status_code == 415
    assert list(tmp_path.iterdir()) == []


def test_missing_token_is_401() -> None:
    try:
        app.dependency_overrides.pop(get_current_user, None)
        with TestClient(app) as anonymous_client:
            response = anonymous_client.post(
                "/posts",
                json={"title": "인증 필요", "content": "본문"},
            )
    finally:
        app.dependency_overrides.clear()

    assert response.status_code == 401
    assert response.headers["www-authenticate"] == "Bearer"
```

다른 작성자의 게시물을 PATCH·DELETE하면 403인지도 Part 5의 정책 테스트에 이어 HTTP 수준에서 확인합니다. 크기 제한에는 경계값과 한 바이트 큰 파일을 넣고, 다중 업로드의 두 번째 파일이 실패하면 첫 번째 파일까지 제거되는지 검사합니다.

## 필요한 브라우저 출처만 CORS로 허용하기

CORS는 브라우저에서 다른 origin의 프론트엔드가 API를 호출할 때 적용됩니다. 서버 간 요청이나 같은 origin 배포에는 필요하지 않으며, credential을 허용하면 운영 origin을 명시적으로 나열합니다.

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:5173",
        "https://www.example.com",
    ],
    allow_credentials=True,
    allow_methods=["GET", "POST", "PATCH", "DELETE"],
    allow_headers=["Authorization", "Content-Type"],
)
```

origin에는 스킴, 호스트, 포트가 모두 포함됩니다. `https://example.com`과 `https://www.example.com`은 다른 origin이며, 허용 목록은 환경별 설정에서 주입합니다. CORS는 인증이나 CSRF 방어를 대신하지 않습니다.

## 성공과 예외 요청을 구조화해 기록하기

운영 로그에는 비밀번호, 토큰, 요청 본문 대신 요청 ID, 메서드, 경로, 상태 코드, 처리 시간을 남깁니다. 성공과 예외를 분리해 기록하면 5xx에서도 요청 ID와 경과 시간이 사라지지 않습니다.

```python
import logging
from time import perf_counter
from uuid import uuid4

from fastapi import Request

logger = logging.getLogger("app.requests")


@app.middleware("http")
async def log_request(request: Request, call_next):
    request_id = uuid4().hex
    started = perf_counter()
    context = {
        "request_id": request_id,
        "method": request.method,
        "path": request.url.path,
    }

    try:
        response = await call_next(request)
    except Exception:
        logger.exception(
            "request_failed",
            extra={
                **context,
                "status_code": 500,
                "elapsed_ms": round((perf_counter() - started) * 1000, 2),
            },
        )
        raise

    response.headers["X-Request-ID"] = request_id
    logger.info(
        "request_complete",
        extra={
            **context,
            "status_code": response.status_code,
            "elapsed_ms": round((perf_counter() - started) * 1000, 2),
        },
    )
    return response
```

프록시도 요청 ID를 만든다면 어떤 값을 신뢰하고 전달할지 정책을 정합니다. 같은 예외를 미들웨어와 예외 처리기에서 중복 기록하지 않고, 로그 집계 시스템에서 요청 ID로 프록시·API·작업 큐의 이벤트를 연결합니다.
