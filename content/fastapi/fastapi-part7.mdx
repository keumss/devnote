## TestClient로 상태 코드와 공개 응답 계약 검사하기

`TestClient`는 실제 네트워크 포트를 열지 않고 ASGI 애플리케이션에 요청을 보냅니다. 구현 내부값보다 클라이언트가 관찰하는 상태 코드, 헤더, JSON 구조를 검증합니다. 먼저 인증이 필요 없는 목록 조회 계약으로 사용법을 익힙니다.

```python
from fastapi.testclient import TestClient

from app.main import app


def test_empty_post_list_returns_public_contract() -> None:
    with TestClient(app) as client:
        response = client.get("/posts")

    assert response.status_code == 200
    assert response.json() == []
```

context manager를 사용하면 테스트 시작 전에 lifespan의 진입 부분이, 종료할 때 정리 부분이 실행됩니다. 테스트 밖에서 전역 `client`를 만들면 lifespan과 자원 정리 시점을 놓치기 쉽습니다.

## dependency_overrides로 저장소와 현재 사용자 격리하기

운영 DB나 실제 토큰 검증을 테스트에서 호출하지 않도록 의존성을 교체합니다. 원래 의존성 함수를 키로, 테스트 함수를 값으로 등록합니다.

```python
from collections.abc import Iterator

import pytest

from app.dependencies import get_current_user, get_repository
from app.models import PostCreate, UserRecord
from app.repository import PostRepository


@pytest.fixture
def client() -> Iterator[TestClient]:
    test_repository = PostRepository()

    def override_repository() -> PostRepository:
        return test_repository

    def override_current_user() -> UserRecord:
        return UserRecord(
            username="alice",
            hashed_password="not-used-in-this-test",
        )

    app.dependency_overrides[get_repository] = override_repository
    app.dependency_overrides[get_current_user] = override_current_user

    with TestClient(app) as test_client:
        yield test_client

    app.dependency_overrides.clear()
```

테스트가 끝날 때 override를 지우지 않으면 다음 테스트가 이전 사용자나 저장소를 그대로 사용할 수 있습니다. fixture마다 새 저장소를 만들어 테스트 순서와 무관한 결과를 보장합니다.

## 정상·검증·인증·인가 실패를 상태별로 테스트하기

성공 경로 하나만 확인하면 라우팅 순서나 권한 누락을 잡기 어렵습니다. 오류가 발생하는 입력과 기대 상태를 표로 고정합니다.

```python
def test_fixed_search_route_wins(client: TestClient) -> None:
    response = client.get("/posts/search", params={"q": "fastapi"})
    assert response.status_code == 200


def test_invalid_dynamic_id_is_422(client: TestClient) -> None:
    response = client.get("/posts/not-an-integer")
    assert response.status_code == 422


def test_missing_post_is_404(client: TestClient) -> None:
    response = client.get("/posts/999")
    assert response.status_code == 404


def test_duplicate_title_is_409(client: TestClient) -> None:
    payload = {"title": "중복 제목", "content": "본문"}
    assert client.post("/posts", json=payload).status_code == 201
    assert client.post("/posts", json=payload).status_code == 409


def test_missing_bearer_token_is_401() -> None:
    app.dependency_overrides.pop(get_current_user, None)
    with TestClient(app) as anonymous_client:
        response = anonymous_client.post(
            "/posts",
            json={"title": "인증 필요", "content": "본문"},
        )
    assert response.status_code == 401
    assert response.headers["www-authenticate"] == "Bearer"


def test_other_author_cannot_delete() -> None:
    test_repository = PostRepository()
    post = test_repository.add(
        PostCreate(title="Bob의 글", content="수정할 수 없습니다."),
        author="bob",
    )
    alice = UserRecord(username="alice", hashed_password="not-used")
    app.dependency_overrides[get_repository] = lambda: test_repository
    app.dependency_overrides[get_current_user] = lambda: alice

    try:
        with TestClient(app) as test_client:
            response = test_client.delete(f"/posts/{post.id}")
    finally:
        app.dependency_overrides.clear()

    assert response.status_code == 403
```

다른 사용자가 만든 글을 수정·삭제할 때는 403, 제목이나 페이지 범위가 잘못되면 422를 추가로 검사합니다. 401과 403을 구분하면 클라이언트가 재로그인할지 권한 부족을 표시할지 판단할 수 있습니다.

## 업로드 크기와 형식 실패 후 파일 정리 확인하기

업로드 테스트는 메모리에서 파일 내용을 만들어 전송할 수 있습니다. 허용하지 않은 시그니처와 크기 초과가 올바른 상태를 반환하고 디렉터리에 부분 파일이 남지 않는지 확인합니다.

```python
def test_rejects_non_image(client: TestClient, tmp_path, monkeypatch) -> None:
    monkeypatch.setattr("app.uploads.UPLOAD_DIR", tmp_path)

    response = client.post(
        "/images",
        files={"file": ("payload.html", b"<script>alert(1)</script>", "text/html")},
    )

    assert response.status_code == 415
    assert list(tmp_path.iterdir()) == []


def test_rejects_oversized_image(client: TestClient, tmp_path, monkeypatch) -> None:
    monkeypatch.setattr("app.uploads.UPLOAD_DIR", tmp_path)
    oversized_png = b"\x89PNG\r\n\x1a\n" + b"0" * (5 * 1024 * 1024)

    response = client.post(
        "/images",
        files={"file": ("large.png", oversized_png, "image/png")},
    )

    assert response.status_code == 413
    assert list(tmp_path.iterdir()) == []
```

테스트 데이터 크기는 구현의 경계값과 정확히 맞춥니다. 운영 프록시의 본문 제한은 애플리케이션 테스트와 별도로 배포 환경에서 확인합니다.

## BackgroundTasks의 실행 시점과 한계 이해하기

`BackgroundTasks`는 응답을 만든 뒤 같은 애플리케이션 프로세스에서 작업을 실행합니다. 짧고 실패해도 복구 가능한 로그 기록이나 알림에는 편리하지만, 작업 큐처럼 영속 저장·재시도·다중 서버 분배를 보장하지 않습니다.

```python
from fastapi import BackgroundTasks, status


def write_audit_log(post_id: int) -> None:
    print(f"created post: {post_id}")


@router.post("", response_model=PostPublic, status_code=status.HTTP_201_CREATED)
def create_post(
    post: PostCreate,
    background_tasks: BackgroundTasks,
    current_user: CurrentUser,
    repository: Repo,
) -> PostRecord:
    created = repository.add(post, author=current_user.username)
    background_tasks.add_task(write_audit_log, created.id)
    return created
```

결제, 메일 발송, 이미지 변환처럼 유실되면 안 되거나 오래 걸리는 작업은 DB 트랜잭션과 연결되는 outbox 패턴 및 Celery 같은 외부 작업 큐를 사용합니다. 여러 워커에서는 어느 프로세스가 작업을 수행할지도 고정되지 않습니다.

## 필요한 출처만 허용하도록 CORS 설정하기

CORS는 브라우저에서 다른 origin의 프론트엔드가 API를 호출할 때 적용됩니다. 서버 간 요청이나 같은 origin 배포에는 필요하지 않습니다. credential을 허용한다면 운영 origin을 명시적으로 나열합니다.

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:5173",
        "https://www.example.com",
    ],
    allow_credentials=True,
    allow_methods=["GET", "POST", "PATCH", "DELETE"],
    allow_headers=["Authorization", "Content-Type"],
)
```

origin에는 스킴, 호스트, 포트가 모두 포함됩니다. `https://example.com`과 `https://www.example.com`은 다른 origin이며, 허용 목록은 환경별 설정에서 주입하는 편이 안전합니다.

## 요청 ID와 처리 시간을 구조화해 기록하기

운영 로그에는 비밀번호, 토큰, 요청 본문 대신 요청 ID, 메서드, 경로, 상태 코드, 처리 시간을 남깁니다. 클라이언트가 보낸 요청 ID를 무조건 신뢰하지 않고 서버가 새 값을 만들 수 있습니다.

```python
import logging
from time import perf_counter
from uuid import uuid4

from fastapi import Request

logger = logging.getLogger("app.requests")


@app.middleware("http")
async def log_request(request: Request, call_next):
    request_id = uuid4().hex
    started = perf_counter()
    response = await call_next(request)
    elapsed_ms = (perf_counter() - started) * 1000

    response.headers["X-Request-ID"] = request_id
    logger.info(
        "request_complete",
        extra={
            "request_id": request_id,
            "method": request.method,
            "path": request.url.path,
            "status_code": response.status_code,
            "elapsed_ms": round(elapsed_ms, 2),
        },
    )
    return response
```

예외가 발생해도 로그를 남기려면 `try`와 `finally`로 측정 경계를 감싸고 예외 처리기와 중복 기록되지 않도록 로깅 정책을 정합니다. 로그 집계 시스템에서는 요청 ID로 프록시와 애플리케이션 로그를 연결합니다.
