## APIRouter로 기능별 경로를 조합하기

애플리케이션이 커지면 경로를 리소스별 라우터로 나누고 진입점에서 조합합니다. 라우터의 `prefix`와 `tags`를 한곳에 두면 실제 URL과 API 문서 그룹을 함께 관리할 수 있습니다.

```python
# app/api/posts.py
from fastapi import APIRouter

router = APIRouter(prefix="/posts", tags=["posts"])


@router.get("")
def read_posts() -> list[dict[str, str]]:
    return [{"title": "Router"}]
```

```python
# app/main.py
from fastapi import FastAPI

from app.api.posts import router as posts_router

app = FastAPI()
app.include_router(posts_router)
```

```bash
curl --silent http://127.0.0.1:8000/posts
```

```json
[{"title":"Router"}]
```

`include_router()`를 빠뜨리면 파일에 데코레이터가 있어도 애플리케이션에는 경로가 등록되지 않습니다. `/openapi.json`의 `paths`를 확인하면 등록 여부를 빠르게 진단할 수 있습니다.

## Annotated와 Depends로 공통 입력 재사용하기

의존성은 인증, 페이지네이션, 저장소처럼 여러 경로에서 반복되는 준비 작업과 계약을 재사용합니다. `Annotated` 별칭을 만들면 실제 값의 타입을 유지하면서 의존성 선언을 짧게 만들 수 있습니다.

```python
from typing import Annotated

from fastapi import Depends, Query


class Pagination:
    def __init__(
        self,
        offset: Annotated[int, Query(ge=0)] = 0,
        limit: Annotated[int, Query(ge=1, le=100)] = 20,
    ) -> None:
        self.offset = offset
        self.limit = limit


Page = Annotated[Pagination, Depends()]


@router.get("")
def read_posts(page: Page) -> dict[str, int]:
    return {"offset": page.offset, "limit": page.limit}
```

FastAPI는 같은 요청 안에서 같은 의존성의 결과를 기본적으로 캐시합니다. 서로 다른 값이 꼭 필요할 때만 `Depends(dependency, use_cache=False)`를 고려합니다.

## yield 의존성으로 요청 단위 자원 정리하기

파일, DB 세션, 외부 클라이언트처럼 반드시 닫아야 하는 자원은 `yield` 의존성으로 관리합니다. `yield` 앞은 경로 연산 전에, `finally`는 응답 처리 뒤에 실행됩니다.

```python
from collections.abc import Iterator
from typing import Annotated

from fastapi import Depends


class Repository:
    def close(self) -> None:
        print("repository closed")


def get_repository() -> Iterator[Repository]:
    repository = Repository()
    try:
        yield repository
    finally:
        repository.close()


Repo = Annotated[Repository, Depends(get_repository)]


@router.get("/count")
def count_posts(repository: Repo) -> dict[str, int]:
    return {"count": 0}
```

초기화 도중 예외가 나면 `yield` 이후에 도달하지 않으므로, 생성된 자원만 정리하도록 의존성을 구성합니다. DB 세션의 commit과 rollback 정책도 경로마다 흩뜨리지 말고 같은 경계에서 정합니다.

## def와 async def를 호출하는 작업에 맞게 선택하기

`async def`는 네트워크 대기처럼 await 가능한 비동기 라이브러리를 호출할 때 적합합니다. 일반 `def` 경로 연산은 FastAPI가 스레드 풀에서 실행하므로 동기 DB 드라이버나 파일 API를 사용할 때 이벤트 루프를 직접 막지 않습니다.

```python
import asyncio


@router.get("/remote")
async def read_remote() -> dict[str, str]:
    # 실제 코드에서는 await http_client.get(...) 같은 비동기 호출을 사용합니다.
    await asyncio.sleep(0.01)
    return {"source": "remote"}


@router.get("/local")
def read_local() -> dict[str, str]:
    # 동기 라이브러리를 호출하는 경로는 일반 def로 선언합니다.
    return {"source": "local"}
```

`async def` 안에서 `time.sleep()`, 동기 HTTP 요청, 큰 파일 복사를 직접 실행하면 그 시간 동안 같은 이벤트 루프의 다른 요청도 지연됩니다. 함수 선언만 바꾼다고 내부 라이브러리가 비동기로 변하지는 않습니다.

## lifespan으로 프로세스 시작과 종료 관리하기

lifespan은 애플리케이션 프로세스가 요청을 받기 전 준비 작업과 종료 시 정리 작업을 연결합니다. 모델 로딩이나 연결 풀 생성처럼 프로세스마다 한 번 필요한 작업에 사용합니다.

```python
from collections.abc import AsyncIterator
from contextlib import asynccontextmanager

from fastapi import FastAPI


@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    app.state.ready = True
    try:
        yield
    finally:
        app.state.ready = False


app = FastAPI(lifespan=lifespan)
```

여러 워커를 실행하면 lifespan도 워커마다 실행됩니다. 스키마 마이그레이션처럼 전체 배포에서 한 번만 실행해야 하는 작업은 lifespan이 아니라 별도 배포 단계로 분리합니다.

## 미들웨어와 예외 처리기의 책임 구분하기

미들웨어는 모든 HTTP 요청의 전후를 감싸므로 요청 ID, 처리 시간, 공통 헤더에 적합합니다. 예외 처리기는 특정 도메인 오류를 일관된 HTTP 응답으로 바꿀 때 사용합니다.

```python
from time import perf_counter

from fastapi import Request
from fastapi.responses import JSONResponse


class PostNotFoundError(Exception):
    def __init__(self, post_id: int) -> None:
        self.post_id = post_id


@app.middleware("http")
async def add_process_time(request: Request, call_next):
    started = perf_counter()
    response = await call_next(request)
    response.headers["X-Process-Time"] = f"{perf_counter() - started:.6f}"
    return response


@app.exception_handler(PostNotFoundError)
async def handle_post_not_found(
    request: Request,
    exc: PostNotFoundError,
) -> JSONResponse:
    return JSONResponse(
        status_code=404,
        content={"detail": f"Post {exc.post_id} not found"},
    )
```

예상하지 못한 모든 예외를 미들웨어에서 성공 응답으로 바꾸면 오류 추적과 상태 코드가 왜곡됩니다. 처리할 수 있는 도메인 예외만 변환하고 나머지는 로깅 후 서버 오류로 남깁니다.
