## 파서, 설정, 작업 함수를 분리하기

실전 CLI는 모듈을 가져오는 순간 `parse_args()`나 파일 작업을 실행하지 않습니다. 파서 생성, 명시적 설정 변환, 실제 작업, 렌더링, 진입점을 분리하면 테스트가 실제 프로세스와 전역 `sys.argv`에 의존하지 않습니다.

```python
from collections.abc import Sequence

def build_parser():
    ...

def parse_options(argv: Sequence[str] | None = None):
    return build_parser().parse_args(argv)

def main(argv: Sequence[str] | None = None) -> int:
    options = parse_options(argv)
    ...
    return 0

if __name__ == '__main__':
    raise SystemExit(main())
```

`argv=None`이면 `parse_args()`가 실제 `sys.argv[1:]`을 사용합니다. 테스트에서는 `main(['sample.txt', '--limit', '3'])`처럼 토큰을 직접 넣습니다. `main()`은 정수를 반환하고 실제 진입점만 `SystemExit`로 바꾸므로 실행 정책과 종료를 한곳에서 관리할 수 있습니다.

다음 예제는 하나 이상의 텍스트 파일에서 자주 나온 단어를 집계합니다. 일반 결과는 표준 출력, 진단과 오류는 표준 오류로 분리하고, 일부 파일에 실패하면 나머지는 처리한 뒤 상태 코드 1을 반환합니다.

## 완성형 wordfreq 명령 구현하기

파서가 `Path`, 양의 정수, 출력 형식처럼 입력 자체의 구조를 검증합니다. 파일 읽기와 단어 집계는 별도 함수가 맡습니다.

```python
# wordfreq.py
import argparse
from collections import Counter
from collections.abc import Sequence
from dataclasses import dataclass
import json
from pathlib import Path
import re
import sys

@dataclass(frozen=True)
class Options:
    files: list[Path]
    limit: int
    ignore_case: bool
    output_format: str
    verbose: int

def positive_int(value: str) -> int:
    try:
        number = int(value)
    except ValueError as error:
        raise argparse.ArgumentTypeError('정수를 입력해야 합니다.') from error

    if number < 1:
        raise argparse.ArgumentTypeError('1 이상의 정수를 입력해야 합니다.')
    return number

def build_parser() -> argparse.ArgumentParser:
    parser = argparse.ArgumentParser(
        prog='wordfreq',
        description='텍스트 파일에서 자주 나온 단어를 집계합니다.',
        epilog='예: wordfreq notes/*.txt --limit 20 --format json',
        formatter_class=argparse.ArgumentDefaultsHelpFormatter,
        allow_abbrev=False,
    )
    parser.add_argument('files', nargs='+', type=Path, metavar='FILE')
    parser.add_argument('--limit', type=positive_int, default=10, help='파일별 단어 수')
    parser.add_argument(
        '--ignore-case',
        action=argparse.BooleanOptionalAction,
        default=True,
        help='대소문자를 구분하지 않음',
    )
    parser.add_argument(
        '--format',
        choices=('text', 'json'),
        default='text',
        help='결과 출력 형식',
    )
    parser.add_argument('-v', '--verbose', action='count', default=0)
    return parser

def parse_options(argv: Sequence[str] | None = None) -> Options:
    args = build_parser().parse_args(argv)
    return Options(
        files=args.files,
        limit=args.limit,
        ignore_case=args.ignore_case,
        output_format=args.format,
        verbose=args.verbose,
    )

def count_words(path: Path, ignore_case: bool) -> Counter[str]:
    text = path.read_text(encoding='utf-8')
    words = re.findall(r'\w+', text)
    if ignore_case:
        words = [word.casefold() for word in words]
    return Counter(words)

def render_text(results: dict[str, list[tuple[str, int]]]) -> str:
    lines: list[str] = []
    for filename, entries in results.items():
        lines.append(f'[{filename}]')
        lines.extend(f'{word}\t{count}' for word, count in entries)
    return '\n'.join(lines)

def main(argv: Sequence[str] | None = None) -> int:
    options = parse_options(argv)
    results: dict[str, list[tuple[str, int]]] = {}
    failures = 0

    for path in options.files:
        if options.verbose:
            print(f'wordfreq: 읽는 중: {path}', file=sys.stderr)
        try:
            counts = count_words(path, options.ignore_case)
        except (OSError, UnicodeError) as error:
            print(f'wordfreq: 읽을 수 없음: {path}: {error}', file=sys.stderr)
            failures += 1
            continue
        results[str(path)] = counts.most_common(options.limit)

    if options.output_format == 'json':
        print(json.dumps(results, ensure_ascii=False, indent=2))
    elif results:
        print(render_text(results))

    return 1 if failures else 0

if __name__ == '__main__':
    raise SystemExit(main())
```

`Options`는 동적으로 속성이 생기는 `Namespace`를 애플리케이션 내부의 명시적인 타입으로 바꿉니다. `count_words()`는 파서를 모르고, `render_text()`는 파일을 읽지 않습니다. 이 분리는 기능별 테스트와 출력 형식 추가를 쉽게 만듭니다.

예제의 `\w+`는 유니코드 문자·숫자·밑줄이 이어진 부분을 단어로 보고 하이픈과 문장 부호에서 나눕니다. 제품의 단어 정의가 다르면 파서가 아니라 `count_words()`의 토큰화 규칙과 테스트를 함께 바꿉니다.

`sample.txt`에 다음 내용이 있다고 가정합니다.

```text
Alpha beta beta
alpha
```

```bash
python wordfreq.py sample.txt --limit 2
```

```text
[sample.txt]
alpha	2
beta	2
```

대소문자를 구분하려면 `--no-ignore-case`, 기계가 읽을 결과가 필요하면 `--format json`을 사용합니다. 셸의 `notes/*.txt` 확장은 셸이 수행하며, 확장된 각 경로가 `files` 위치 인자에 들어옵니다.

## 입력 오류와 실행 오류의 종료 상태 구분하기

파서가 발견한 잘못된 호출은 기본 동작에 맡깁니다. 예를 들어 `--limit 0`, 지원하지 않는 `--format yaml`, 파일 위치 인자 누락은 사용법과 오류를 표준 오류에 쓰고 상태 코드 2로 종료합니다. 도움말은 표준 출력에 쓰고 상태 코드 0으로 종료합니다.

파싱에는 성공했지만 파일이 없거나 UTF-8로 읽을 수 없는 경우는 실행 오류입니다. 예제는 해당 오류를 표준 오류에 쓰고 다른 파일을 계속 처리한 뒤 상태 코드 1을 반환합니다. 따라서 다음처럼 자동화에서 결과와 성공 여부를 독립적으로 다룰 수 있습니다.

```bash
python wordfreq.py good.txt missing.txt --format json > result.json
status=$?
```

`result.json`에는 읽기에 성공한 파일의 유효한 JSON만 남고, 실패 설명은 표준 오류로 나갑니다. `status`는 1입니다. JSON 결과에 진단 문구를 섞지 않기 때문에 다른 프로그램이 표준 출력을 안정적으로 파싱할 수 있습니다.

모든 실패에서 추적 정보를 그대로 보여 주면 내부 경로와 구현 세부가 불필요하게 노출되고 사용자가 행동하기 어렵습니다. 기본 오류는 대상과 해결에 필요한 원인을 짧게 보여 주고, 개발자가 원인을 조사해야 할 때만 `-v` 수준이나 별도 디버그 정책으로 세부 정보를 늘립니다.

## pytest로 파서와 main의 계약 검증하기

파서 테스트는 기본값, 변환, 플래그, 잘못된 호출을 고정합니다. `main()` 테스트는 임시 파일과 캡처된 표준 출력을 사용해 실제 작업과 종료 상태를 검증합니다.

```python
# test_wordfreq.py
from pathlib import Path

import pytest

from wordfreq import build_parser, main

def test_parser_converts_options() -> None:
    args = build_parser().parse_args([
        'notes.txt',
        '--limit', '3',
        '--no-ignore-case',
        '-vv',
    ])

    assert args.files == [Path('notes.txt')]
    assert args.limit == 3
    assert args.ignore_case is False
    assert args.verbose == 2

def test_parser_rejects_non_positive_limit() -> None:
    with pytest.raises(SystemExit) as error:
        build_parser().parse_args(['notes.txt', '--limit', '0'])

    assert error.value.code == 2

def test_help_exits_successfully(capsys) -> None:
    with pytest.raises(SystemExit) as error:
        build_parser().parse_args(['--help'])

    captured = capsys.readouterr()
    assert error.value.code == 0
    assert '텍스트 파일에서 자주 나온 단어' in captured.out
    assert captured.err == ''

def test_main_prints_counts(tmp_path, capsys) -> None:
    source = tmp_path / 'notes.txt'
    source.write_text('Python cli python', encoding='utf-8')

    status = main([str(source), '--limit', '2'])

    captured = capsys.readouterr()
    assert status == 0
    assert f'[{source}]\npython\t2\ncli\t1\n' == captured.out
    assert captured.err == ''

def test_main_reports_missing_file(capsys) -> None:
    status = main(['missing.txt'])

    captured = capsys.readouterr()
    assert status == 1
    assert captured.out == ''
    assert '읽을 수 없음: missing.txt' in captured.err
```

도움말은 공개 문서이므로 성공 상태 0, 표준 출력, 명령 목적을 함께 검사합니다. 전체 도움말 문자열을 스냅샷으로 고정하면 Python 패치 버전의 줄바꿈 변화에도 테스트가 깨질 수 있으므로 중요한 조각만 확인하는 편이 유지보수에 유리합니다.

## 배포 전 실제 셸에서 경계 입력 점검하기

자동 테스트 뒤에는 사용자가 실행할 방식으로 짧은 스모크 테스트를 합니다. 최소한 `--help`, 정상 파일, 없는 파일, 빈 파일, UTF-8이 아닌 파일, 공백이나 대시로 시작하는 경로, `--format json`, 잘못된 옵션의 출력 위치와 종료 상태를 확인합니다.

다음 스크립트는 `sample.txt`가 준비된 상태에서 핵심 성공 경로와 사용법 오류의 상태 코드를 검사합니다. 모든 검사가 통과하면 출력 없이 0으로 끝납니다.

```bash
set -eu

python wordfreq.py --help >/dev/null
python wordfreq.py sample.txt --format json >/dev/null

set +e
python wordfreq.py sample.txt --limit 0 >/dev/null 2>&1
status=$?
set -e

test "$status" -eq 2
```

문제가 생기면 다음 경계를 순서대로 확인합니다.

- 값이 예상과 다르면 셸이 만든 토큰과 `Namespace`를 먼저 확인합니다.
- 옵션이 갑자기 모호해졌다면 긴 옵션 약어와 가변 길이 `nargs`를 확인합니다.
- 오류가 JSON에 섞이면 일반 결과는 표준 출력, 진단은 표준 오류인지 확인합니다.
- 테스트에서 프로세스가 종료되면 `main(argv)`에 토큰 목록을 전달했는지 확인합니다.
- 파일이 파싱 도중 열리면 `type`에는 `Path`만 두고 입출력을 작업 함수로 이동합니다.

Python 3.12의 세부 동작은 [argparse API 문서](https://docs.python.org/3.12/library/argparse.html)와 [argparse 튜토리얼](https://docs.python.org/3.12/howto/argparse.html)을 기준으로 확인합니다. 다른 Python 버전을 지원한다면 사용하는 액션과 폐기 예정 기능을 그 버전의 공식 문서에서도 다시 확인합니다.
