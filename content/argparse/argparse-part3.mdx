## subparsers로 작업마다 독립된 입력 문법 만들기

`git commit`과 `git log`처럼 한 프로그램이 서로 다른 작업을 제공한다면 `add_subparsers()`로 하위 명령을 만듭니다. 각 하위 파서는 자기 작업에 필요한 인자만 가지므로 거대한 옵션 집합보다 도움말과 오류가 명확합니다.

```python
import argparse

parser = argparse.ArgumentParser(prog='inventory')
commands = parser.add_subparsers(
    dest='command',
    required=True,
    metavar='COMMAND',
)

add_parser = commands.add_parser('add', help='재고 항목을 추가')
add_parser.add_argument('name')
add_parser.add_argument('--quantity', type=int, default=1)

list_parser = commands.add_parser('list', help='재고 목록을 표시')
list_parser.add_argument('--out-of-stock', action='store_true')

args = parser.parse_args(['add', 'keyboard', '--quantity', '2'])
print(args)
```

```text
Namespace(command='add', name='keyboard', quantity=2)
```

`required=True`가 없으면 아무 하위 명령도 선택하지 않은 `Namespace`가 반환될 수 있습니다. 명령 없이도 유효한 기본 동작이 없다면 반드시 명시합니다. 결과에는 최상위 파서와 선택한 하위 파서의 속성만 들어오며, 선택하지 않은 `list`의 `out_of_stock`은 생기지 않습니다.

최상위 옵션은 보통 하위 명령 앞에 둡니다. `inventory --verbose add ...`와 `inventory add --verbose ...`를 모두 지원하려고 같은 옵션을 여러 파서에 중복 등록하면 기본값 덮어쓰기와 도움말 중복이 생길 수 있습니다. 공개 문법을 하나로 정하고 도움말 예제에 순서를 보여 줍니다.

## set_defaults로 하위 명령과 실행 함수를 연결하기

선택한 명령 이름을 긴 `if` 문으로 비교하는 대신 각 하위 파서에 실행 함수를 저장할 수 있습니다. 새 명령의 파서와 핸들러를 가까이 두면 확장할 때 기존 분기문을 건드리지 않아도 됩니다.

```python
import argparse
from collections.abc import Callable

Handler = Callable[[argparse.Namespace], int]

def add_item(args: argparse.Namespace) -> int:
    print(f'{args.name}: {args.quantity}개 추가')
    return 0

def list_items(args: argparse.Namespace) -> int:
    scope = '품절 포함' if args.out_of_stock else '재고 있음'
    print(f'조회 범위: {scope}')
    return 0

def build_parser() -> argparse.ArgumentParser:
    parser = argparse.ArgumentParser(prog='inventory')
    commands = parser.add_subparsers(required=True)

    add_parser = commands.add_parser('add')
    add_parser.add_argument('name')
    add_parser.add_argument('--quantity', type=int, default=1)
    add_parser.set_defaults(handler=add_item)

    list_parser = commands.add_parser('list')
    list_parser.add_argument('--out-of-stock', action='store_true')
    list_parser.set_defaults(handler=list_items)
    return parser

args = build_parser().parse_args(['add', 'keyboard', '--quantity', '2'])
status = args.handler(args)
print(f'exit={status}')
```

```text
keyboard: 2개 추가
exit=0
```

`set_defaults()`는 고정값을 `Namespace`에 추가하는 기능이며 함수도 저장할 수 있습니다. 핸들러는 종료 상태를 반환하고, 데이터베이스나 파일 작업은 별도 서비스 함수에 위임하면 파서와 도메인 로직을 독립적으로 테스트하기 쉽습니다.

정적 타입 검사기는 동적으로 생기는 `Namespace` 속성을 완전히 추론하지 못합니다. 규모가 커지면 파싱 직후 `Namespace`를 명시적인 데이터 클래스나 함수 인자로 변환합니다. 파서와 핸들러를 연결하는 값만 `Namespace`에 남기고, 도메인 함수에는 검증된 명시적 타입을 전달하면 내부 코드가 `argparse`에 의존하지 않습니다. Part 4의 `Options`가 이 경계를 완성합니다.

## 상호 배타 그룹과 파싱 후 검증을 나누기

동시에 사용할 수 없는 옵션은 `add_mutually_exclusive_group()`에 둡니다. 파서가 사용법에 관계를 표시하고 충돌을 상태 코드 2의 입력 오류로 처리합니다.

```python
import argparse

parser = argparse.ArgumentParser(prog='deploy')
target = parser.add_mutually_exclusive_group(required=True)
target.add_argument('--staging', action='store_true')
target.add_argument('--production', action='store_true')
parser.add_argument('--confirm-production', action='store_true')

args = parser.parse_args(['--production', '--confirm-production'])

if args.confirm_production and not args.production:
    parser.error('--confirm-production은 --production과 함께 사용해야 합니다.')
```

그룹은 “최대 하나” 또는 `required=True`일 때 “정확히 하나”인 관계만 표현합니다. “A를 쓰면 B도 필요하다”, “두 값의 대소 관계가 맞아야 한다” 같은 교차 규칙은 파싱 뒤에 검사하고 `parser.error()`로 사용자가 고칠 문구를 제공합니다.

`parser.error()`는 사용법과 오류를 표준 오류에 쓰고 상태 코드 2로 종료합니다. 반면 배포 API가 실패하거나 파일을 쓸 수 없는 문제는 명령 문법이 올바른 상태에서 발생한 실행 오류이므로 애플리케이션이 상태 코드 1 같은 정책으로 처리합니다.

Python 3.12에서는 인자 그룹 안에 그룹을 중첩하거나 상호 배타 그룹을 중첩하는 방식이 폐기 예정입니다. 그룹을 최상위 파서에 나란히 두고, 더 복잡한 관계는 명시적인 검증 코드로 표현합니다.

## 옵션 약어를 끄고 모호한 토큰을 통제하기

기본 파서는 모호하지 않은 긴 옵션 접두사를 약어로 허용합니다. 현재 `--configuration`만 있을 때 `--conf`가 동작하더라도 나중에 `--conflict-policy`를 추가하면 기존 호출이 모호해질 수 있습니다. 자동화용 CLI는 `allow_abbrev=False`로 공개 문법을 고정하는 편이 안전합니다.

```python
import argparse

parser = argparse.ArgumentParser(prog='runner', allow_abbrev=False)
parser.add_argument('--configuration')
parser.add_argument('--conflict-policy', choices=('stop', 'overwrite'))

args = parser.parse_args([
    '--configuration', 'ci.toml',
    '--conflict-policy', 'stop',
])
print(args)
```

```text
Namespace(configuration='ci.toml', conflict_policy='stop')
```

`-1` 같은 토큰은 음수 위치 값일 수도 있고 짧은 옵션일 수도 있습니다. 음수처럼 보이는 옵션을 등록하면 해석 규칙이 달라질 수 있으므로 숫자를 옵션 이름으로 만들지 않습니다. 대시로 시작하는 문자열을 확실히 위치 값으로 전달하려면 앞서 배운 `--`를 사용합니다.

가변 길이 위치 인자와 옵션을 복잡하게 섞어 받기 위해 `parse_intermixed_args()`를 선택할 수도 있지만 하위 파서, `REMAINDER`, 일부 상호 배타 조합을 지원하지 않습니다. 먼저 위치 인자를 단순하게 배치하거나 반복 옵션으로 문법을 바꿀 수 있는지 검토합니다.

## parse_known_args로 다른 명령에 인자를 위임하기

래퍼 프로그램이 자기 옵션만 읽고 나머지를 다른 프로그램에 그대로 넘겨야 할 때 `parse_known_args()`를 사용합니다. 일반 애플리케이션에서 오타를 무시하는 용도로 쓰면 안 됩니다.

```python
import argparse

parser = argparse.ArgumentParser(prog='timed', allow_abbrev=False)
parser.add_argument('--repeat', type=int, default=1)

args, remainder = parser.parse_known_args([
    '--repeat', '2',
    'python', '-m', 'pytest', '-q',
])
print(args)
print(remainder)
```

```text
Namespace(repeat=2)
['python', '-m', 'pytest', '-q']
```

알려진 인자가 아닌 토큰은 순서대로 `remainder`에 남습니다. 기본 접두사 약어가 켜져 있으면 위임할 옵션이 우연히 알려진 긴 옵션의 접두사일 때 파서가 소비할 수 있으므로 `allow_abbrev=False`를 함께 사용하는 편이 안전합니다.

외부 프로세스를 실행할 때는 남은 토큰을 하나의 셸 문자열로 합치지 말고 목록 그대로 `subprocess.run(remainder, shell=False)`에 전달합니다. 문자열 결합과 `shell=True`는 공백·따옴표 의미를 바꾸고 신뢰할 수 없는 입력이 셸 명령으로 해석될 위험을 만듭니다.

## 파싱 오류와 종료 동작의 책임 범위 이해하기

기본 `parse_args()`는 타입 변환 실패, 알 수 없는 옵션, 필수 인자 누락을 발견하면 사용법을 표준 오류에 출력하고 `SystemExit(2)`를 발생시킵니다. `--help`와 `--version`도 `SystemExit`를 사용하지만 성공 상태인 0으로 종료합니다. 실행형 CLI에서는 이 기본 동작이 셸과 잘 맞습니다.

```python
import argparse

parser = argparse.ArgumentParser(prog='worker', exit_on_error=False)
parser.add_argument('--jobs', type=int)

try:
    parser.parse_args(['--jobs', 'many'])
except argparse.ArgumentError as error:
    print(f'다시 입력하세요: {error}')
```

```text
다시 입력하세요: argument --jobs: invalid int value: 'many'
```

`exit_on_error=False`는 파서를 다른 프로그램에 포함해 일부 `ArgumentError`를 직접 처리할 때 유용합니다. 모든 종료 동작을 없애는 스위치는 아니며 `--help` 같은 액션은 여전히 종료합니다. 보통의 CLI는 기본값을 유지하고, 테스트에서는 `pytest.raises(SystemExit)`로 종료 상태를 검사하는 편이 실제 동작을 더 정확히 보존합니다.

입력 형식 오류를 잡아 상태 코드 1로 바꾸면 셸 스크립트가 사용법 오류와 실행 실패를 구분할 수 없게 됩니다. `parse_args()`까지는 CLI 문법, 명시적 설정 타입으로 변환한 뒤에는 애플리케이션 작업이라는 경계를 유지합니다. 상세 traceback은 사용자가 요청한 진단 모드나 운영 로그에만 남기고, 기본 CLI 오류는 사용자가 고칠 수 있는 문구로 제공합니다.
