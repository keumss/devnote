## 입력 순서와 결과 계약 정하기

이번 예제는 주문을 전부 목록으로 모으지 않고 한 건씩 읽어 상태별로 집계한 뒤, 두 건씩 저장하는 `order_batches.py`를 만듭니다. `groupby()`는 연속한 같은 키만 묶으므로 `read_orders()`가 `status` 기준으로 정렬된 주문을 준다는 계약이 필요합니다.

```python
# order_batches.py
from collections.abc import Iterable
from dataclasses import dataclass

@dataclass(frozen=True)
class Order:
    order_id: str
    status: str
    amount: int

def read_orders() -> Iterable[Order]:
    yield Order('A-1', 'paid', 12000)
    yield Order('A-2', 'paid', 8000)
    yield Order('A-3', 'paid', 15000)
    yield Order('A-4', 'refunded', -8000)
```

실제 데이터베이스라면 `ORDER BY status`로, 파일이라면 읽기 전에 정렬하는 단계로 이 계약을 보장합니다. 정렬되지 않은 입력에 `groupby()`를 바로 적용하면 같은 상태가 여러 번 나뉩니다.

## groupby로 상태별 합계 만들기

상태별 건수와 금액 합계가 필요할 때는 현재 상태의 주문만 읽어 바로 집계합니다. 이전 그룹을 목록으로 만들지 않으므로 주문 수가 많아도 이미 처리한 주문은 메모리에 남지 않습니다.

```python
from itertools import groupby
from operator import attrgetter

def summarize_by_status(items: Iterable[Order]) -> dict[str, tuple[int, int]]:
    summaries = {}

    for status, group in groupby(items, key=attrgetter('status')):
        count = 0
        total = 0

        for order in group:
            count += 1
            total += order.amount

        summaries[status] = (count, total)

    return summaries

print(summarize_by_status(read_orders()))
```

```text
{'paid': (3, 35000), 'refunded': (1, -8000)}
```

`group`은 원본 입력을 공유하는 이터레이터입니다. 합계처럼 한 번의 계산만 필요하면 반복문에서 바로 끝내고, 주문 자체가 나중에도 필요할 때만 `list(group)`으로 저장합니다.

## batched로 저장 API의 요청 크기 맞추기

저장 API가 한 번에 두 건만 받는다면 `batched()`로 각 상태 그룹을 바로 나눕니다. 마지막 배치는 남은 주문만 담은 짧은 튜플로 전달됩니다.

```python
from itertools import batched, groupby
from operator import attrgetter

def save_batch(status: str, batch: tuple[Order, ...]) -> None:
    order_ids = [order.order_id for order in batch]
    print(status, order_ids)

for status, group in groupby(read_orders(), key=attrgetter('status')):
    for batch in batched(group, 2):
        save_batch(status, batch)
```

```text
paid ['A-1', 'A-2']
paid ['A-3']
refunded ['A-4']
```

배치 크기는 API의 항목 제한과 실패했을 때 다시 처리할 양을 보고 정합니다. 마지막의 짧은 배치를 받지 못하는 API라면 이 단계 전에 그 배치만 별도 정책으로 처리합니다.

## 재시도 전에 저장 작업을 멱등하게 만들기

네트워크 시간 초과처럼 일시적인 실패에만 같은 배치를 재시도합니다. 첫 요청이 서버에서 이미 성공했을 수도 있으므로, 주문 ID를 고유 키로 사용해 같은 배치를 두 번 보내도 결과가 중복되지 않게 만듭니다.

```python
from collections.abc import Callable

def save_with_retry(
    save: Callable[[tuple[Order, ...]], None],
    batch: tuple[Order, ...],
    attempts: int = 3,
) -> None:
    for attempt in range(attempts):
        try:
            save(batch)
            return
        except TimeoutError:
            if attempt == attempts - 1:
                raise
```

인증 실패나 잘못된 입력처럼 재시도로 해결되지 않는 오류는 잡지 않고 호출자에게 전달합니다. 실제 서비스에서는 재시도 횟수뿐 아니라 대기 시간과 실패 기록도 요구사항에 맞춰 추가합니다.

## 생성기 입력으로 한 번 소비하는 집계 테스트하기

`summarize_by_status()`가 생성기를 한 번만 읽는지 확인하려면 테스트에도 목록 대신 생성기를 전달합니다. 결과는 각 상태의 건수와 합계라는 작은 값만 검사합니다.

```python
# test_order_batches.py
from order_batches import Order, summarize_by_status

def test_summarizes_sorted_generator() -> None:
    items = (
        Order(order_id, status, amount)
        for order_id, status, amount in [
            ('A-1', 'paid', 12000),
            ('A-2', 'paid', 8000),
            ('A-3', 'refunded', -8000),
        ]
    )

    assert summarize_by_status(items) == {
        'paid': (2, 20000),
        'refunded': (1, -8000),
    }
```

생성기는 소비한 뒤 다시 시작하지 않습니다. 다음 테스트에 새 입력이 필요하면 같은 생성기를 재사용하지 말고 새로 만들며, 정렬되지 않은 입력을 거부해야 한다면 그 경우를 별도 테스트로 추가합니다.

Python 3.12의 `groupby()` 공유 이터레이터와 `batched()`의 마지막 짧은 배치 규칙은 [itertools 공식 문서](https://docs.python.org/3.12/library/itertools.html)를 기준으로 확인합니다.
