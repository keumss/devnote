## 이벤트 처리 요구사항과 지연 파이프라인 설계하기

실전 예제로 줄 단위 이벤트를 읽어 유효한 레코드만 변환하고, 연속 사용자 구간을 요약한 뒤 API 전송 크기로 묶는 파이프라인을 만듭니다. 입력 전체를 리스트로 만들지 않고 `filterfalse()`, `starmap()`, `groupby()`, `batched()`를 단계별로 연결합니다. 원본은 사용자 ID 순서로 정렬되어 있다고 가정해 `groupby()`가 같은 사용자를 한 그룹으로 묶게 합니다.

```python
from collections.abc import Iterable, Iterator
from dataclasses import dataclass

@dataclass(frozen=True)
class Event:
    user_id: str
    kind: str
    duration_ms: int

def parse_line(line: str) -> tuple[str, str, str]:
    parts = tuple(part.strip() for part in line.split(','))
    if len(parts) != 3:
        raise ValueError(f'필드는 3개여야 합니다: {line!r}')
    return parts
```

실제 파일은 `with open(path, encoding='utf-8') as file:` 블록 안에서 파이프라인 소비까지 마쳐야 합니다. 지연 이터레이터만 반환하고 파일을 먼저 닫으면 나중에 값을 읽을 때 `ValueError`가 발생합니다. 외부 데이터의 오류를 건너뛸지 전체 작업을 중단할지도 먼저 정합니다.

## 선택과 변환 단계를 작은 함수로 구현하기

빈 줄과 `#` 주석을 `filterfalse()`로 제외하고, `starmap()`으로 파싱된 세 필드를 `Event` 생성 함수의 인자로 펼칩니다. 각 단계가 받는 값과 반환하는 값을 좁게 유지하면 어느 단계에서 검증이 실패했는지 테스트하기 쉽습니다.

```python
from itertools import filterfalse, starmap

def is_ignored(line: str) -> bool:
    stripped = line.strip()
    return not stripped or stripped.startswith('#')

def make_event(user_id: str, kind: str, duration: str) -> Event:
    if not user_id or not kind:
        raise ValueError('user_id와 kind는 비어 있을 수 없습니다.')

    try:
        duration_ms = int(duration)
    except ValueError as error:
        raise ValueError('duration_ms는 정수여야 합니다.') from error

    if duration_ms < 0:
        raise ValueError('duration_ms는 음수일 수 없습니다.')
    return Event(user_id, kind, duration_ms)

def read_events(lines: Iterable[str]) -> Iterator[Event]:
    active_lines = filterfalse(is_ignored, lines)
    fields = map(parse_line, active_lines)
    return starmap(make_event, fields)
```

`read_events()`를 호출했을 때는 아직 각 줄을 검증하지 않습니다. `list()`, 반복문, `next()` 등으로 결과를 소비할 때 오류가 발생합니다. 실패한 줄 번호가 운영에 중요하다면 `enumerate(lines, start=1)`를 포함한 명시적 제너레이터 함수로 바꾸고 오류 메시지에 번호를 추가합니다.

## groupby와 batched로 사용자 요약을 전송 단위로 묶기

입력이 사용자 ID 순으로 정렬되어 있다는 계약 아래 `groupby()`로 사용자별 요약을 만듭니다. 각 그룹은 즉시 순회해 개수와 총 지연 시간을 계산하므로 이벤트 전체를 저장하지 않습니다. 완성된 요약은 `batched()`가 전송 크기만큼 튜플로 묶습니다.

```python
from itertools import batched, groupby
from operator import attrgetter

def summarize_users(events: Iterable[Event]) -> Iterator[dict[str, object]]:
    for user_id, user_events in groupby(events, key=attrgetter('user_id')):
        count = 0
        total_duration_ms = 0
        kinds: set[str] = set()

        for event in user_events:
            count += 1
            total_duration_ms += event.duration_ms
            kinds.add(event.kind)

        yield {
            'user_id': user_id,
            'event_count': count,
            'total_duration_ms': total_duration_ms,
            'kinds': sorted(kinds),
        }

lines = [
    '# user_id,kind,duration_ms\n',
    'u-1,login,20\n',
    'u-1,view,35\n',
    'u-2,login,15\n',
]

summaries = summarize_users(read_events(lines))
for batch in batched(summaries, 2):
    print(batch)
```

```text
({'user_id': 'u-1', 'event_count': 2, 'total_duration_ms': 55, 'kinds': ['login', 'view']}, {'user_id': 'u-2', 'event_count': 1, 'total_duration_ms': 15, 'kinds': ['login']})
```

사용자 ID가 `u-1, u-2, u-1` 순서로 다시 등장하면 `u-1` 요약도 둘로 나뉩니다. 입력이 정렬되지 않았다면 메모리에서 정렬할지, 딕셔너리로 집계할지, 저장소 쿼리의 정렬을 보장할지 데이터 크기와 운영 환경에 맞춰 결정합니다.

## 소비 시점과 경계 조건을 자동 테스트하기

테스트는 정상 결과뿐 아니라 지연 오류, 불완전한 마지막 배치, 그룹 정렬 전제도 고정해야 합니다. 결과 비교를 위해 테스트 경계에서만 리스트로 구체화하면 애플리케이션 구현은 스트리밍 방식을 유지할 수 있습니다.

```python
import pytest
from itertools import batched

from pipeline import read_events, summarize_users

def test_summarizes_and_batches_events() -> None:
    lines = [
        '\n',
        'u-1,login,20\n',
        'u-1,view,30\n',
        'u-2,login,10\n',
    ]

    batches = list(batched(summarize_users(read_events(lines)), 1))

    assert len(batches) == 2
    assert batches[0][0] == {
        'user_id': 'u-1',
        'event_count': 2,
        'total_duration_ms': 50,
        'kinds': ['login', 'view'],
    }

def test_rejects_negative_duration_when_consumed() -> None:
    events = read_events(['u-1,login,-1\n'])

    with pytest.raises(ValueError, match='음수'):
        next(events)

def test_keeps_short_final_batch() -> None:
    assert list(batched(range(5), 2)) == [(0, 1), (2, 3), (4,)]
```

`batched()`의 마지막 묶음을 반드시 꽉 채워야 하는 업무라면 전송 전에 길이를 검사하는 래퍼와 오류 테스트를 추가합니다. 빈 입력, 필드 개수 오류, 정수가 아닌 지연 시간, 한 사용자만 있는 입력도 중요한 경계 조건입니다.

## 오류 처리와 운영 관찰 지점 정하기

지연 파이프라인에서는 생성보다 소비 중에 예외가 발생하므로 배치 전송 반복문이 오류 처리 경계가 됩니다. 성공한 배치를 재전송하지 않도록 배치 ID나 체크포인트를 기록하고, 전송 실패에 재시도를 적용할 때는 대상 API의 중복 처리 정책을 확인합니다.

```python
def send_all(lines: Iterable[str], batch_size: int) -> int:
    if batch_size <= 0:
        raise ValueError('batch_size는 양수여야 합니다.')

    sent_count = 0
    summaries = summarize_users(read_events(lines))

    for batch in batched(summaries, batch_size):
        send_batch(batch)  # 실패하면 호출자가 재시도 정책을 적용합니다.
        sent_count += len(batch)

    return sent_count
```

운영에서는 읽은 줄 수, 무시한 줄 수, 검증 실패 수, 완성한 사용자 요약 수, 배치 크기와 전송 시간을 관찰합니다. 처리 단계가 느려 입력 생산자를 오래 붙잡을 수 있고 `tee()` 같은 분기 버퍼가 커질 수 있으므로, 여러 하류 소비자가 필요하면 메시지 큐나 저장소를 경계로 두는 편이 복구와 확장에 유리합니다.
