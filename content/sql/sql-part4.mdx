## 업무 규칙을 제약 조건으로 표현하기

애플리케이션 검증만으로는 동시 요청이나 관리 도구를 통한 입력을 막을 수 없습니다. 핵심 규칙은 `NOT NULL`, `CHECK`, `UNIQUE`, 외래 키처럼 데이터베이스 제약 조건에도 기록합니다. 아래 예시는 상품·주문·주문 항목의 최소 구조입니다.

```sql
CREATE TABLE products (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name TEXT NOT NULL,
  stock INTEGER NOT NULL CHECK (stock >= 0),
  price NUMERIC(12, 2) NOT NULL CHECK (price >= 0)
);

CREATE TABLE orders (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  customer_email TEXT NOT NULL,
  status TEXT NOT NULL CHECK (status IN ('pending', 'paid', 'cancelled')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE order_items (
  order_id BIGINT NOT NULL REFERENCES orders(id),
  product_id BIGINT NOT NULL REFERENCES products(id),
  quantity INTEGER NOT NULL CHECK (quantity > 0),
  PRIMARY KEY (order_id, product_id)
);
```

## 입력값은 파라미터로 바인딩하기

문자열을 이어 붙여 SQL을 만들면 SQL 주입 취약점이 생깁니다. 사용하는 드라이버의 파라미터 바인딩 기능을 이용하고, SQL 문법에서 값 자리에만 파라미터를 사용합니다. 테이블명이나 정렬 방향처럼 SQL 구조를 동적으로 고를 때는 허용 목록으로 제한합니다.

```python
import psycopg

with psycopg.connect('postgresql://app:password@localhost/app') as conn:
    with conn.cursor() as cur:
        # %s는 드라이버의 값 자리 표시자입니다. 문자열 연결을 사용하지 않습니다.
        cur.execute(
            'SELECT id, customer_email FROM orders WHERE customer_email = %s',
            ('reader@example.com',),
        )
        print(cur.fetchall())
```

## 재고 차감과 주문 생성을 하나의 트랜잭션으로 처리하기

주문 생성과 재고 차감은 둘 중 하나만 성공하면 안 됩니다. 재고를 줄이는 `UPDATE`에 조건을 넣고, 모든 명령이 성공할 때만 `COMMIT`합니다. 예외가 나면 `ROLLBACK`되어 주문과 재고가 함께 원래 상태로 돌아갑니다.

```sql
BEGIN;

WITH reserved_product AS (
  -- 재고가 충분할 때만 한 행을 수정합니다. UPDATE가 행 잠금을 처리합니다.
  UPDATE products
  SET stock = stock - 2
  WHERE id = 42 AND stock >= 2
  RETURNING id
), new_order AS (
  INSERT INTO orders (customer_email, status)
  SELECT 'reader@example.com', 'pending'
  WHERE EXISTS (SELECT 1 FROM reserved_product)
  RETURNING id
)
INSERT INTO order_items (order_id, product_id, quantity)
SELECT id, 42, 2 FROM new_order;

COMMIT;
-- INSERT된 행 수가 0이면 재고 부족으로 보고 ROLLBACK합니다.
```

## 운영 쿼리는 실행 계획과 실제 데이터로 검증하기

인덱스는 읽기를 빠르게 할 수 있지만 쓰기 비용과 저장 공간을 늘립니다. 추측으로 인덱스를 추가하지 말고, 대표 데이터 양과 실제 조건으로 `EXPLAIN (ANALYZE, BUFFERS)`를 확인합니다. 운영 데이터에서 실행할 때는 쿼리의 부작용과 부하를 먼저 검토합니다.

```sql
CREATE INDEX idx_orders_customer_created_at
  ON orders (customer_email, created_at DESC);

EXPLAIN (ANALYZE, BUFFERS)
SELECT id, status, created_at
FROM orders
WHERE customer_email = 'reader@example.com'
ORDER BY created_at DESC
LIMIT 20;
```
