## logging으로 이벤트를 관찰하는 이유

`logging`은 프로그램 실행 중 일어난 이벤트를 중요도와 함께 남기는 Python 3.12 표준 라이브러리입니다. `print()`는 사용자가 요청한 결과에 적합하고, 로그는 개발자와 운영자가 장애 원인이나 실행 상태를 조사할 때 사용합니다.

예를 들어 CLI가 JSON을 출력한다면 JSON은 표준 출력, 진단 로그는 기본 목적지인 표준 오류로 분리할 수 있습니다. 그러면 다른 프로그램이 결과만 안정적으로 파싱하면서 운영 환경은 로그를 별도로 수집할 수 있습니다.

```python
import logging

print('{"status": "ok"}')
logging.warning('응답 시간이 기준을 초과했습니다.')
```

터미널에서는 두 줄이 함께 보일 수 있지만 출력 스트림은 다릅니다.

```text
{"status": "ok"}
WARNING:root:응답 시간이 기준을 초과했습니다.
```

로그에는 비밀번호, 토큰, 쿠키, 전체 개인정보처럼 수집해서는 안 되는 값을 넣지 않습니다. 낮은 레벨의 로그도 운영 중 활성화되거나 외부 저장소로 전송될 수 있습니다.

## basicConfig로 가장 작은 프로그램 설정하기

`basicConfig()`는 작은 애플리케이션에서 루트 로거의 레벨과 형식을 한 번에 설정합니다. 첫 로그 호출보다 먼저 실행 진입점에서 호출합니다.

```python
# app.py
import logging

logging.basicConfig(
    level=logging.INFO,
    format='%(levelname)s %(name)s: %(message)s',
)

logger = logging.getLogger(__name__)
logger.info('애플리케이션을 시작했습니다.')
```

```text
INFO __main__: 애플리케이션을 시작했습니다.
```

기본 루트 로거의 레벨은 `WARNING`이므로 설정하지 않으면 `INFO`와 `DEBUG`는 보이지 않습니다. `basicConfig()`는 이미 루트에 핸들러가 있으면 기본적으로 아무 변경도 하지 않으므로, 모듈마다 반복 호출하지 않고 프로세스 시작 지점이 설정을 소유해야 합니다.

## 로그 레벨을 사건의 심각도에 맞추기

레벨은 `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL` 순으로 높아집니다. 개발 중 세부 진단은 `DEBUG`, 정상적인 주요 흐름은 `INFO`, 계속 실행할 수 있는 예상 밖의 상황은 `WARNING`에 적합합니다. 작업 하나가 실패했지만 프로그램이 계속 실행한다면 `ERROR`, 프로세스 전체가 더 실행되기 어려운 상황은 `CRITICAL`을 고려합니다.

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format='%(levelname)s %(name)s: %(message)s',
)
logger = logging.getLogger('inventory')

logger.debug('캐시 키를 계산했습니다.')
logger.info('재고 동기화를 시작했습니다.')
logger.warning('응답이 지연되어 재시도합니다.')
logger.error('상품 하나를 저장하지 못했습니다.')
```

```text
INFO inventory: 재고 동기화를 시작했습니다.
WARNING inventory: 응답이 지연되어 재시도합니다.
ERROR inventory: 상품 하나를 저장하지 못했습니다.
```

`DEBUG`가 없는 것이 핵심입니다. 레벨은 메시지의 주제가 아니라 사건의 심각도와 사용자가 취할 행동을 기준으로 일관되게 선택합니다. 정상적인 요청마다 `WARNING`을 남기면 실제 경고가 묻힙니다.

## getLogger로 모듈별 이벤트 기록하기

애플리케이션 코드에서는 `logging.getLogger(__name__)`으로 모듈마다 로거를 하나 만듭니다. `__name__`은 패키지 경로와 같은 점(`.`) 구분 계층을 이루므로 `shop.api`의 이벤트를 `shop` 단위로 설정할 수 있습니다.

```python
# shop/api.py
import logging

logger = logging.getLogger(__name__)

def find_product(product_id: int) -> dict[str, int]:
    logger.info('상품을 조회합니다. product_id=%s', product_id)
    return {'id': product_id}
```

값은 f-string으로 미리 합치지 않고 `%s` 자리표시자와 로그 인자로 전달합니다. 비활성 레벨에서는 문자열 포맷을 생략할 수 있고, 메시지 형식과 값의 경계도 분명해집니다.

라이브러리와 하위 모듈은 핸들러를 직접 추가하거나 전역 설정을 바꾸지 않고 이벤트만 기록합니다. 실행 파일이 출력 위치와 형식을 정해야 재사용하는 애플리케이션의 정책을 침범하지 않습니다.

작은 단일 프로세스가 파일에 직접 기록할 때는 `basicConfig(filename='app.log', encoding='utf-8', ...)`처럼 대상과 인코딩을 지정합니다. 기본 모드는 이어쓰기이며 `filemode='w'`는 매번 새 파일이 필요한 진단 작업에만 사용합니다. 콘솔과 파일의 기록 범위를 나누는 방법은 Part 2에서 다룹니다.
