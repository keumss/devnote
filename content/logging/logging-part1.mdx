---
title: "Part 1. 기초: 프로그램의 실행 흐름 기록하기"
---

`logging`은 프로그램이 무엇을 했고 어떤 문제가 생겼는지 실행 중에 남기는 Python 표준 라이브러리입니다. `print()`와 달리 중요도, 출력 형식, 목적지를 나눌 수 있어 CLI와 서버 모두에서 같은 방식으로 사용합니다.

## getLogger(__name__)로 모듈별 기록 시작하기

함수나 모듈의 실행 사실을 남길 때는 모듈 이름을 가진 logger를 한 번 만들고 재사용합니다. `__name__`을 쓰면 패키지 구조에 맞춰 logger 이름도 함께 바뀝니다.

```python
import logging


logger = logging.getLogger(__name__)


def calculate_total(price: int, quantity: int) -> int:
    logger.info('calculating order total')
    return price * quantity
```

`Logger`를 직접 생성하지 않고 `getLogger()`를 호출합니다. 같은 이름으로 여러 번 호출해도 같은 logger를 얻으므로, 모듈 최상단의 `logger` 하나를 함수들이 공유하면 됩니다.

## basicConfig로 CLI의 기본 출력 정하기

작은 CLI에서 로그의 최소 레벨과 출력 형식을 한 번 정하려면 프로그램 진입점에서 `basicConfig()`를 호출합니다. 기본 root logger의 레벨은 `WARNING`이므로 `INFO`를 보고 싶다면 명시해야 합니다.

```python
import logging


logging.basicConfig(
    level=logging.INFO,
    format='%(levelname)s %(name)s %(message)s',
)
logger = logging.getLogger('price_cli')

logger.info('loaded 3 products')
logger.warning('one product has low stock')
```

```text
INFO price_cli loaded 3 products
WARNING price_cli one product has low stock
```

라이브러리 모듈은 `basicConfig()`를 호출하지 않습니다. 애플리케이션의 진입점만 출력 정책을 정해야 사용하는 쪽이 파일, 콘솔, 테스트 환경에 맞게 바꿀 수 있습니다.

## DEBUG·INFO·WARNING으로 사건의 중요도 표현하기

개발 중 자세한 상태, 정상적인 업무 진행, 곧 문제가 될 수 있는 상태를 구분하려면 알맞은 로그 레벨을 선택합니다. 레벨은 오류를 제어하는 예외 처리 대신이 아니라, 관찰할 사건의 심각도를 나타냅니다.

```python
import logging


logger = logging.getLogger('inventory')

logger.debug('loaded inventory from cache')
logger.info('reserved 2 items')
logger.warning('stock is low for SKU-11')
logger.error('inventory file could not be saved')
```

`DEBUG`는 문제를 분석할 개발자용 상세 정보, `INFO`는 예상한 정상 흐름, `WARNING`은 프로그램이 계속 동작하지만 확인할 만한 상태에 사용합니다. 요청 하나를 실패시킨 문제는 `ERROR`, 프로그램이 계속 실행하기 어려운 상태는 `CRITICAL`로 남깁니다.

## %s 인수로 필요한 로그만 포매팅하기

계산 비용이 있거나 자주 호출되는 경로에서 값을 로그에 넣을 때는 f-string을 먼저 만들지 말고 logger에 형식 문자열과 인수를 따로 전달합니다. 해당 레벨이 꺼져 있으면 문자열 조합을 하지 않습니다.

```python
import logging


logger = logging.getLogger('orders')
order_id = 'O-100'
item_count = 3

logger.debug('order %s contains %d items', order_id, item_count)
```

`logger.debug(f'order {order_id} contains {item_count} items')`는 DEBUG가 꺼져도 f-string을 먼저 계산합니다. 로그 메시지는 문자열 템플릿을 고정하고 값은 인수로 전달하면 검색·집계할 메시지의 형태도 일정하게 유지됩니다.

## logger.exception으로 실패 원인과 traceback 남기기

예외를 사용자에게 실패로 알리면서 조사에 필요한 traceback도 남기려면 `except` 블록 안에서 `logger.exception()`을 호출합니다. 이 메서드는 `ERROR` 레벨에 현재 예외 정보를 함께 기록합니다.

```python
import logging


logger = logging.getLogger('importer')


def read_quantity(value: str) -> int:
    try:
        return int(value)
    except ValueError:
        logger.exception('quantity could not be parsed: %r', value)
        raise


read_quantity('three')
```

```text
ERROR importer quantity could not be parsed: 'three'
Traceback (most recent call last):
  ...
ValueError: invalid literal for int() with base 10: 'three'
```

`exception()`은 예외를 처리하거나 다시 던지는 코드와 함께 씁니다. 예외가 없는 일반 분기에서 사용하면 의미 없는 traceback을 남기므로, 그 경우에는 `error()`나 `warning()`을 선택합니다.

## stderr로 로그와 정상 결과 나누기

CLI가 다른 프로그램과 파이프로 연결될 때 정상 결과는 표준 출력으로, 조사용 로그는 표준 오류로 나눕니다. `basicConfig()`의 기본 stream은 `stderr`이므로 결과만 `print()`로 씁니다.

```python
import logging


logging.basicConfig(level=logging.INFO, format='%(levelname)s %(message)s')
logger = logging.getLogger('convert')

logger.info('converting source.csv')
print('converted.csv')
```

```text
# stderr
INFO converting source.csv

# stdout
converted.csv
```

호출자는 `python convert.py > result.txt`로 결과 파일만 받을 수 있습니다. 로그 메시지에 비밀번호, 인증 헤더, 원본 요청 본문처럼 민감할 수 있는 입력을 넣지 않는 원칙도 이때부터 지킵니다.

공식 Python [logging 기본 튜토리얼](https://docs.python.org/3.12/howto/logging.html), [logging API](https://docs.python.org/3.12/library/logging.html)를 기준으로 구성했습니다.
