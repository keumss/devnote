## Logger, Handler, Formatter의 역할 나누기

로거는 호출을 받아 `LogRecord`를 만들고, 핸들러는 레코드를 콘솔·파일 같은 목적지로 보냅니다. 포매터는 목적지에 기록할 문자열의 구조를 정합니다. 로거와 핸들러가 모두 레벨을 가지므로, 어느 코드의 이벤트를 만들지와 어느 목적지에 보낼지를 따로 제어할 수 있습니다.

```python
import logging

logger = logging.getLogger('shop')
logger.setLevel(logging.DEBUG)

console = logging.StreamHandler()
console.setLevel(logging.WARNING)
console.setFormatter(logging.Formatter(
    '%(levelname)s %(name)s: %(message)s',
))
logger.addHandler(console)

logger.debug('재고 조회를 시작합니다.')
logger.warning('재고가 부족합니다.')
```

```text
WARNING shop: 재고가 부족합니다.
```

로거는 두 호출을 모두 레코드로 만들지만, 콘솔 핸들러가 `DEBUG` 레코드를 거릅니다.

## 로거 계층에서 유효 레벨을 상속하고 재정의하기

점(`.`)으로 구분한 이름은 로거 계층을 만듭니다. 새 로거의 레벨은 기본값 `NOTSET`이며, 이때 가장 가까운 조상 로거의 설정값을 유효 레벨로 사용합니다. 패키지 전체에 기본값을 주고 조사할 모듈만 더 자세한 레벨로 재정의할 때 유용합니다.

```python
import logging

shop = logging.getLogger('shop')
shop.setLevel(logging.INFO)

catalog = logging.getLogger('shop.catalog')  # shop의 INFO를 상속합니다.
checkout = logging.getLogger('shop.checkout')
checkout.setLevel(logging.DEBUG)

for logger in (catalog, checkout):
    level = logging.getLevelName(logger.getEffectiveLevel())
    print(f'{logger.name}: {level}')
```

```text
shop.catalog: INFO
shop.checkout: DEBUG
```

애플리케이션 모듈에서 `getLogger(__name__)`을 사용하면 패키지 구조가 그대로 이 계층이 됩니다. 대부분은 자식 로거의 레벨을 건드리지 않고, 상위 패키지 로거에서 기본 수준을 관리합니다.

## 핸들러 레벨로 목적지마다 기록 범위 나누기

하나의 로거가 만든 레코드를 여러 핸들러가 서로 다른 기준으로 처리할 수 있습니다. 다음 설정은 상세 로그는 파일에 남기고 운영 콘솔에는 경고 이상만 보냅니다.

```python
import logging

formatter = logging.Formatter('%(levelname)s %(name)s: %(message)s')

console = logging.StreamHandler()
console.setLevel(logging.WARNING)
console.setFormatter(formatter)

diagnostic_file = logging.FileHandler('debug.log', encoding='utf-8')
diagnostic_file.setLevel(logging.DEBUG)
diagnostic_file.setFormatter(formatter)

shop = logging.getLogger('shop')
shop.setLevel(logging.DEBUG)
shop.addHandler(console)
shop.addHandler(diagnostic_file)

checkout = logging.getLogger('shop.checkout')
checkout.debug('PG 요청을 전송했습니다.')
checkout.warning('PG 응답이 지연됩니다.')
```

콘솔에는 경고 한 줄만 보이고, `debug.log`에는 두 줄이 모두 기록됩니다.

```text
# 콘솔
WARNING shop.checkout: PG 응답이 지연됩니다.

# debug.log
DEBUG shop.checkout: PG 요청을 전송했습니다.
WARNING shop.checkout: PG 응답이 지연됩니다.
```

로거 레벨은 레코드 생성 범위, 핸들러 레벨은 목적지별 출력 범위입니다. 상세 기록이 필요하다면 로거와 해당 핸들러의 기준이 모두 충분히 낮아야 합니다.

## propagate를 이해해 중복 로그 막기

자식 로거의 `propagate` 기본값은 `True`입니다. 레코드는 자기 핸들러를 거친 뒤 조상 로거의 핸들러에도 전달되므로, 같은 목적지의 핸들러를 자식과 조상 양쪽에 붙이면 중복 출력됩니다.

```python
import logging

formatter = logging.Formatter('%(name)s: %(message)s')

root_handler = logging.StreamHandler()
root_handler.setFormatter(formatter)
logging.getLogger().addHandler(root_handler)

checkout = logging.getLogger('shop.checkout')
checkout_handler = logging.StreamHandler()
checkout_handler.setFormatter(formatter)
checkout.addHandler(checkout_handler)

checkout.warning('결제를 확인합니다.')  # 자식과 루트에서 한 번씩 출력됩니다.

checkout.propagate = False
checkout.warning('결제를 완료했습니다.')  # 자식 핸들러에서만 출력됩니다.
```

```text
shop.checkout: 결제를 확인합니다.
shop.checkout: 결제를 확인합니다.
shop.checkout: 결제를 완료했습니다.
```

일반적인 애플리케이션은 루트에만 핸들러를 두고 자식의 전파를 유지합니다. 특정 하위 계층의 목적지를 완전히 분리할 때만 그 계층에 핸들러를 모두 붙이고 `propagate=False`로 끊습니다. 이미 자식 로거에서 만들어진 레코드가 전파될 때는 조상 로거의 레벨을 다시 검사하지 않으며, 조상에 연결된 각 핸들러의 레벨이 출력 여부를 결정합니다.
