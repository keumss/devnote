## Logger, Handler, Formatter의 역할 나누기

로거는 로그 호출을 받아 `LogRecord`를 만들고, 핸들러는 레코드를 콘솔·파일 같은 목적지로 보냅니다. 포매터는 목적지에 기록할 문자열의 구조를 정합니다. 이 흐름을 알면 메시지가 사라지거나 중복될 때 어느 설정을 확인해야 하는지 판단할 수 있습니다.

```python
import logging

logger = logging.getLogger('shop')
logger.setLevel(logging.DEBUG)

console = logging.StreamHandler()
console.setLevel(logging.WARNING)
console.setFormatter(logging.Formatter(
    '%(levelname)s %(name)s: %(message)s',
))
logger.addHandler(console)
logger.propagate = False

logger.debug('재고 조회를 시작합니다.')
logger.warning('재고가 부족합니다.')
```

```text
WARNING shop: 재고가 부족합니다.
```

로거 레벨은 어떤 레코드를 만들지 결정하고, 핸들러 레벨은 받은 레코드 중 어느 범위를 목적지로 보낼지 결정합니다. 두 기준 중 하나라도 높으면 낮은 레벨의 이벤트는 출력되지 않습니다.

## 핸들러 레벨로 목적지별 기록 범위 나누기

상세 진단은 파일에 남기고 운영 콘솔에는 경고 이상만 보낼 수 있습니다. 로거는 두 목적지가 필요로 하는 가장 낮은 레벨까지 레코드를 만들어야 합니다.

```python
import logging
from tempfile import TemporaryDirectory
from pathlib import Path

with TemporaryDirectory() as directory:
    formatter = logging.Formatter('%(levelname)s %(name)s: %(message)s')

    console = logging.StreamHandler()
    console.setLevel(logging.WARNING)
    console.setFormatter(formatter)

    diagnostic = logging.FileHandler(
        Path(directory) / 'debug.log',
        encoding='utf-8',
    )
    diagnostic.setLevel(logging.DEBUG)
    diagnostic.setFormatter(formatter)

    shop = logging.getLogger('example.shop')
    shop.setLevel(logging.DEBUG)
    shop.addHandler(console)
    shop.addHandler(diagnostic)
    shop.propagate = False

    shop.debug('PG 요청을 전송했습니다.')
    shop.warning('PG 응답이 지연됩니다.')
    diagnostic.close()

    print((Path(directory) / 'debug.log').read_text(encoding='utf-8'), end='')
```

콘솔에는 경고 한 줄이 나오고, 마지막 `print()`로 확인한 파일에는 두 줄이 있습니다.

```text
WARNING example.shop: PG 응답이 지연됩니다.
DEBUG example.shop: PG 요청을 전송했습니다.
WARNING example.shop: PG 응답이 지연됩니다.
```

실제 애플리케이션에서는 핸들러를 함수나 설정 객체로 한 번 구성합니다. 테스트나 반복 초기화에서 같은 로거에 핸들러를 계속 추가하면 메시지도 반복됩니다.

## 로거 계층에서 유효 레벨 상속하기

점으로 구분한 이름은 로거 계층을 만듭니다. 새 로거의 레벨은 기본값 `NOTSET`이며, 가장 가까운 조상의 설정값을 유효 레벨로 사용합니다. 패키지 전체에 기본 수준을 주고 조사할 모듈만 자세히 볼 때 유용합니다.

```python
import logging

shop = logging.getLogger('shop')
shop.setLevel(logging.INFO)

catalog = logging.getLogger('shop.catalog')
checkout = logging.getLogger('shop.checkout')
checkout.setLevel(logging.DEBUG)

for logger in (catalog, checkout):
    level = logging.getLevelName(logger.getEffectiveLevel())
    print(f'{logger.name}: {level}')
```

```text
shop.catalog: INFO
shop.checkout: DEBUG
```

대부분의 모듈은 자기 레벨을 직접 정하지 않고 애플리케이션 설정을 상속합니다. 코드 곳곳의 `setLevel()`은 환경별 정책을 추적하기 어렵게 만듭니다.

## propagate를 이해해 중복 로그 막기

자식 로거의 `propagate` 기본값은 `True`입니다. 레코드는 자기 핸들러를 거친 뒤 조상 핸들러에도 전달되므로 같은 목적지의 핸들러를 자식과 조상 양쪽에 붙이면 중복 출력됩니다.

일반적인 애플리케이션은 루트에 핸들러를 한 번 두고 자식 로거의 전파를 유지합니다. 특정 하위 계층의 목적지를 완전히 분리할 때만 그 계층에 필요한 핸들러를 붙이고 `propagate=False`로 끊습니다.

```python
import logging

root = logging.getLogger()
root.handlers.clear()
root.setLevel(logging.WARNING)
root.addHandler(logging.StreamHandler())

checkout = logging.getLogger('shop.checkout')
checkout.warning('한 번만 기록됩니다.')
```

```text
한 번만 기록됩니다.
```

이미 자식 로거가 만든 레코드가 전파될 때 조상 로거의 레벨은 다시 검사하지 않습니다. 조상에 연결된 각 핸들러의 레벨이 최종 출력 여부를 결정합니다.

## logger.exception으로 실패 맥락 남기기

처리 중인 예외의 traceback이 필요하면 `except` 블록 안에서 `logger.exception()`을 사용합니다. 이 호출은 `ERROR` 레벨 메시지와 현재 예외 정보를 함께 기록합니다.

```python
import logging

logger = logging.getLogger(__name__)

def load_price(raw_price: str) -> int:
    try:
        return int(raw_price)
    except ValueError:
        logger.exception('가격 변환에 실패했습니다. raw_price=%r', raw_price)
        raise
```

예상된 잘못된 입력을 사용자가 바로 고칠 수 있다면 traceback 없이 짧은 경고가 더 적합할 수 있습니다. 반대로 개발자가 호출 경로를 조사해야 하는 예상 밖의 실패에는 traceback이 유용합니다.

같은 예외를 여러 계층에서 반복 기록하지 않습니다. 예외를 복구하거나 사용자용 오류로 바꾸는 경계처럼 실패의 의미를 가장 잘 설명할 수 있는 한 지점에서 기록합니다.
