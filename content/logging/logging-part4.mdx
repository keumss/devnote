## dictConfig로 애플리케이션 설정을 한 번에 적용하기

실전 설정은 애플리케이션 진입점에서 `logging.config.dictConfig()`로 한 번 적용합니다. 핸들러와 포매터를 모듈마다 추가하지 않으면 초기화 순서와 중복 출력 문제를 줄일 수 있습니다.

```python
import logging
import logging.config

LOGGING = {
    'version': 1,
    'disable_existing_loggers': False,
    'formatters': {
        'standard': {
            'format': '%(asctime)s %(levelname)s %(name)s: %(message)s',
        },
    },
    'handlers': {
        'console': {
            'class': 'logging.StreamHandler',
            'level': 'WARNING',
            'formatter': 'standard',
        },
        'diagnostic': {
            'class': 'logging.FileHandler',
            'level': 'DEBUG',
            'formatter': 'standard',
            'filename': 'app.log',
            'encoding': 'utf-8',
        },
    },
    'root': {
        'level': 'INFO',
        'handlers': ['console', 'diagnostic'],
    },
    'loggers': {
        'shop.checkout': {'level': 'DEBUG'},
    },
}

logging.config.dictConfig(LOGGING)

logger = logging.getLogger('shop.checkout')
logger.debug('결제 요청 본문을 검증했습니다.')  # app.log에만 기록됩니다.
logger.warning('결제 응답이 지연됩니다.')      # 콘솔과 app.log에 기록됩니다.
```

`shop.checkout`의 `DEBUG` 레코드는 전파 과정에서 루트 로거의 `INFO` 레벨을 다시 검사하지 않으므로 파일 핸들러까지 도달합니다. `disable_existing_loggers=False`는 설정 전에 만들어진 서드파티 로거를 의도치 않게 비활성화하지 않도록 명시합니다.

## 환경별 레벨과 서드파티 로그 양 조절하기

개발과 운영의 레벨은 코드에 흩어진 `setLevel()` 호출보다 설정을 만들 때 결정하는 편이 추적하기 쉽습니다. 환경 변수는 허용된 이름인지 검증하고, 애플리케이션 레벨과 콘솔 핸들러 레벨에 함께 적용합니다. 지나치게 많은 로그를 내는 서드파티 패키지는 이름별로 더 높은 기준을 둘 수 있습니다.

```python
import logging.config
import os

level_name = os.getenv('LOG_LEVEL', 'INFO').upper()
allowed_levels = {'DEBUG', 'INFO', 'WARNING', 'ERROR', 'CRITICAL'}
if level_name not in allowed_levels:
    raise ValueError(f'지원하지 않는 LOG_LEVEL입니다: {level_name}')

logging.config.dictConfig({
    'version': 1,
    'disable_existing_loggers': False,
    'handlers': {
        'console': {
            'class': 'logging.StreamHandler',
            'level': level_name,
        },
    },
    'root': {'level': level_name, 'handlers': ['console']},
    'loggers': {
        'urllib3': {'level': 'WARNING', 'propagate': True},
    },
})
```

장애 조사 때 `DEBUG`를 잠시 활성화하면 로그 양과 민감정보 노출 가능성도 함께 커집니다. 필요한 기간과 대상 로거를 좁히고, 조사가 끝나면 원래 수준으로 되돌립니다.

## 로거 체인을 출력해 누락과 중복 원인 찾기

로그가 보이지 않거나 두 번씩 나온다면 호출 지점만 고치기보다 로거 체인을 확인합니다. 유효 레벨, 연결된 핸들러, 각 핸들러의 레벨, `propagate`를 함께 보면 대부분의 설정 문제를 찾을 수 있습니다.

```python
import logging

def describe_logger(name: str) -> None:
    logger = logging.getLogger(name)

    while logger is not None:
        handlers = [
            f'{type(handler).__name__}({logging.getLevelName(handler.level)})'
            for handler in logger.handlers
        ]
        print(
            f'name={logger.name} '
            f'effective={logging.getLevelName(logger.getEffectiveLevel())} '
            f'handlers={handlers} propagate={logger.propagate}'
        )

        if not logger.propagate:
            break
        logger = logger.parent

describe_logger('shop.checkout')
```

유효 레벨이 예상보다 높으면 자식이 상속한 조상 레벨을 확인합니다. 레코드는 만들어지지만 보이지 않으면 핸들러 레벨을 확인하고, 중복되면 같은 목적지의 핸들러가 여러 계층에 붙어 있는지 확인합니다. `propagate=False`가 중간에 있으면 그 위의 핸들러에는 전달되지 않습니다.

## caplog로 중요한 로그 동작 테스트하기

오류 경로의 레벨과 메시지는 테스트로 고정할 수 있습니다. `pytest`의 `caplog`를 사용하면 실제 파일이나 콘솔에 쓰지 않고 `LogRecord`를 검사할 수 있으며, 비밀 값이 남지 않는지도 확인할 수 있습니다.

```python
import logging

def charge(card_number: str) -> None:
    logger = logging.getLogger('shop.payment')
    logger.error('결제를 거절했습니다. card_last4=%s', card_number[-4:])

def test_charge_logs_only_last_four_digits(caplog) -> None:
    with caplog.at_level(logging.ERROR, logger='shop.payment'):
        charge('4111111111111111')

    assert 'card_last4=1111' in caplog.text
    assert '4111111111111111' not in caplog.text
```

문자열 전체보다 `caplog.records`의 `levelno`, `name`, `getMessage()`를 검사하면 포매터 변경에 덜 취약한 테스트를 만들 수 있습니다.
