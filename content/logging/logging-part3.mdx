운영 로그는 모듈마다 제각각 설정하지 않고 시작 지점에서 한 번 구성합니다. 같은 요청이나 작업에 속한 메시지를 찾을 수 있도록 필요한 식별자만 넣고, 민감한 입력은 기록 전에 제외합니다.

## dictConfig로 코드 밖의 설정 구조 만들기

handler·formatter·logger 설정이 많아지면 딕셔너리 구조로 모아 `dictConfig()`에 전달합니다. 애플리케이션 코드는 어떤 logger에 기록할지만 알고, 출력 정책은 설정 함수가 담당합니다.

```python
import logging
import logging.config


LOGGING = {
    'version': 1,
    'disable_existing_loggers': False,
    'formatters': {
        'line': {
            'format': '%(levelname)s %(name)s %(message)s',
        },
    },
    'handlers': {
        'console': {
            'class': 'logging.StreamHandler',
            'formatter': 'line',
            'level': 'INFO',
        },
    },
    'loggers': {
        'shop': {
            'handlers': ['console'],
            'level': 'INFO',
            'propagate': False,
        },
    },
}

logging.config.dictConfig(LOGGING)
logging.getLogger('shop.orders').info('order O-100 created')
```

```text
INFO shop.orders order O-100 created
```

`disable_existing_loggers=False`는 애플리케이션이 설정하기 전에 만들어진 다른 logger를 함부로 끄지 않게 합니다. 배포 환경별 차이는 환경 변수를 읽는 설정 함수에서 `LOGGING` 값을 바꿔 해결합니다.

## extra로 작업 식별자 함께 기록하기

같은 주문 처리에서 나온 로그를 찾으려면 메시지에 매번 문자열로 붙이지 말고 record의 추가 필드로 전달합니다. formatter는 이 필드를 한 줄의 일정한 위치에 출력합니다.

```python
import logging


handler = logging.StreamHandler()
handler.setFormatter(
    logging.Formatter('%(levelname)s order_id=%(order_id)s %(message)s')
)
logger = logging.getLogger('shop.orders')
logger.handlers = [handler]
logger.setLevel(logging.INFO)
logger.propagate = False

logger.info('inventory reserved', extra={'order_id': 'O-100'})
```

```text
INFO order_id=O-100 inventory reserved
```

formatter가 요구하는 필드는 모든 record에 있어야 합니다. `extra`의 키는 `message`, `levelname`처럼 logging이 이미 사용하는 `LogRecord` 필드와 충돌하면 안 됩니다.

## LoggerAdapter로 같은 컨텍스트 재사용하기

하나의 작업에서 여러 로그를 남길 때는 `LoggerAdapter`가 공통 `extra`를 대신 전달하게 할 수 있습니다. 주문 ID를 모든 호출에 반복해서 쓰지 않아도 됩니다.

```python
import logging


logger = logging.getLogger('shop.orders')
order_logger = logging.LoggerAdapter(logger, {'order_id': 'O-100'})

order_logger.info('inventory reserved')
order_logger.info('payment approved')
```

```text
INFO order_id=O-100 inventory reserved
INFO order_id=O-100 payment approved
```

`LoggerAdapter`는 logger를 감싸지만 로그 레벨과 handler는 원래 logger의 설정을 사용합니다. 요청마다 값이 바뀌는 웹 애플리케이션에서는 adapter를 전역으로 보관하지 말고 해당 요청 범위에서 만듭니다.

## Filter로 누락된 컨텍스트에 기본값 주기

일부 로그에만 요청 ID가 있을 때 formatter가 실패하지 않게 하려면 filter에서 기본값을 넣습니다. 이 filter는 record가 handler에 전달되기 전에 실행됩니다.

```python
import logging


class RequestIdFilter(logging.Filter):
    def filter(self, record: logging.LogRecord) -> bool:
        if not hasattr(record, 'request_id'):
            record.request_id = '-'
        return True


handler = logging.StreamHandler()
handler.addFilter(RequestIdFilter())
handler.setFormatter(
    logging.Formatter('%(levelname)s request_id=%(request_id)s %(message)s')
)
```

filter는 누락된 필드를 보완하거나 특정 record를 제외하는 데 사용합니다. 비밀번호나 인증 헤더를 문자열 치환으로 숨기는 마지막 방어선으로 쓰기보다, 처음부터 그런 값을 `extra`와 메시지에 넣지 않는 편이 안전합니다.

## 입력 전문 대신 안전한 식별자만 남기기

실패한 API 요청을 조사할 때도 요청 본문이나 `Authorization` 헤더 전체를 로그에 남기면 개인정보와 비밀값이 복제됩니다. 업무에 필요한 식별자와 오류 종류만 골라 기록합니다.

```python
import logging


logger = logging.getLogger('shop.api')


def reject_payment(order_id: str, reason: str) -> None:
    logger.warning(
        'payment rejected: %s',
        reason,
        extra={'order_id': order_id},
    )
```

`order_id`와 `reason`이 조사에 충분한지 먼저 판단합니다. 카드 번호, 비밀번호, 액세스 토큰, 쿠키, 원본 본문은 로그가 아니라 접근 제어된 업무 데이터 또는 별도의 보안 절차에서 다룹니다.

## caplog로 사용자에게 보이지 않는 로그 계약 검증하기

pytest에서는 `caplog` fixture로 로그의 레벨, 메시지, 추가 필드를 확인합니다. 출력 문자열 전체보다 record의 구조를 검사하면 시간과 formatter 변경에 덜 흔들립니다.

```python
import logging


def reserve(order_id: str) -> None:
    logging.getLogger('shop.orders').info(
        'inventory reserved',
        extra={'order_id': order_id},
    )


def test_reserve_logs_order_id(caplog) -> None:
    caplog.set_level(logging.INFO, logger='shop.orders')

    reserve('O-100')

    record = caplog.records[-1]
    assert record.levelno == logging.INFO
    assert record.getMessage() == 'inventory reserved'
    assert record.order_id == 'O-100'
```

로그가 성공 여부를 결정하는 대상은 아니지만, 운영자가 장애를 구분하는 데 필요한 컨텍스트는 관찰 가능한 계약입니다. 로그에 남기지 않아야 할 값도 테스트 입력과 예상 record에서 다시 확인합니다.

공식 Python [logging.config](https://docs.python.org/3.12/library/logging.config.html), [LoggerAdapter](https://docs.python.org/3.12/library/logging.html#loggeradapter-objects), [Filter](https://docs.python.org/3.12/library/logging.html#filter-objects), pytest [caplog](https://docs.pytest.org/en/stable/how-to/logging.html)를 기준으로 구성했습니다.
