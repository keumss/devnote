## 지연 포맷과 레벨 검사로 불필요한 비용 줄이기

로그 값은 f-string으로 미리 합치지 않고 `%s` 자리표시자와 인자로 전달합니다. 비활성 레벨이면 `logging`이 문자열 포맷을 생략할 수 있습니다. 인자를 만드는 작업 자체가 비싸다면 `isEnabledFor()`로 먼저 확인합니다.

```python
import logging

logger = logging.getLogger(__name__)
logger.setLevel(logging.INFO)

def build_expensive_debug_snapshot() -> dict[str, int]:
    return {'pending_orders': 3}

user_id = 42
logger.debug('사용자 상태를 조회합니다. user_id=%s', user_id)

if logger.isEnabledFor(logging.DEBUG):
    snapshot = build_expensive_debug_snapshot()
    logger.debug('현재 상태: %s', snapshot)
```

이 설정에서는 두 `DEBUG` 호출이 기록되지 않으며 `build_expensive_debug_snapshot()`도 실행되지 않습니다. 비밀번호, 토큰, 쿠키, 전체 개인정보처럼 수집해서는 안 되는 값은 레벨과 관계없이 로그 인자로 넘기지 않습니다.

## logger.exception으로 실패 맥락과 traceback 남기기

처리 중인 예외를 기록할 때는 `logger.exception()`을 사용하면 메시지와 traceback이 함께 `ERROR` 레벨로 남습니다. 예외를 잡은 `except` 블록 안에서 호출하고, 어떤 입력과 작업이 실패했는지 안전한 범위에서 설명합니다.

```python
import logging

logger = logging.getLogger(__name__)

def load_price(raw_price: str) -> int:
    try:
        return int(raw_price)
    except ValueError:
        logger.exception('가격 변환에 실패했습니다. raw_price=%r', raw_price)
        raise
```

같은 예외를 여러 계층에서 반복 기록하면 traceback이 중복됩니다. 예외를 복구하거나 사용자용 오류로 바꾸는 경계처럼, 실패의 의미를 가장 잘 설명할 수 있는 한 지점에서 기록합니다.

## LoggerAdapter로 요청 컨텍스트 반복 줄이기

여러 로그를 같은 요청과 연결하려면 `request_id` 같은 식별자를 `LogRecord`에 추가합니다. 매 호출마다 `extra`를 반복하기보다 `LoggerAdapter`에 공통 컨텍스트를 넣으면 누락을 줄일 수 있습니다.

```python
import logging

handler = logging.StreamHandler()
handler.setFormatter(logging.Formatter(
    '%(levelname)s request_id=%(request_id)s %(message)s',
))

base_logger = logging.getLogger('shop.api')
base_logger.setLevel(logging.INFO)
base_logger.addHandler(handler)
base_logger.propagate = False

logger = logging.LoggerAdapter(base_logger, {'request_id': 'req-8f31'})
logger.info('주문을 생성합니다.')
logger.info('재고를 차감합니다.')
```

```text
INFO request_id=req-8f31 주문을 생성합니다.
INFO request_id=req-8f31 재고를 차감합니다.
```

포매터가 `request_id`를 요구하면 해당 핸들러로 들어오는 모든 레코드에 그 값이 있어야 합니다. 웹 서버처럼 동시 요청이 많은 환경에서는 `contextvars.ContextVar`와 필터를 조합해 요청별 값을 자동으로 주입할 수 있습니다.

## 파일 회전과 중앙 수집 중 운영 방식 선택하기

단일 프로세스가 파일에 직접 기록한다면 회전 없이 파일 크기가 계속 증가합니다. `RotatingFileHandler`로 최대 크기와 보관 개수를 제한할 수 있습니다.

```python
import logging
from logging.handlers import RotatingFileHandler

handler = RotatingFileHandler(
    'app.log',
    maxBytes=10 * 1024 * 1024,
    backupCount=5,
    encoding='utf-8',
)
handler.setFormatter(logging.Formatter(
    '%(asctime)s %(levelname)s %(name)s: %(message)s',
))

logger = logging.getLogger('shop')
logger.setLevel(logging.INFO)
logger.addHandler(handler)
logger.info('애플리케이션을 시작했습니다.')
```

컨테이너나 관리형 실행 환경에서는 보통 `StreamHandler`로 표준 오류에 기록하고 플랫폼의 수집·보존 기능에 맡깁니다. 여러 프로세스가 같은 회전 파일에 직접 쓰면 교체 시점이 충돌할 수 있으므로, 이런 환경에서는 중앙 수집 방식을 우선합니다.
