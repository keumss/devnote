---
title: "Part 1. 기초: 데이터베이스 스키마 변경 기록하기"
---

Alembic은 SQLAlchemy 모델이 바뀔 때 운영 데이터베이스의 스키마를 순서대로 변경하는 도구입니다. 이 노트에서는 파일 SQLite 데이터베이스에 첫 테이블을 만들고 되돌리며, 애플리케이션 코드 대신 revision 파일이 스키마 변경 이력을 관리하는 흐름을 익힙니다.

## alembic init으로 마이그레이션 환경 만들기

이미 데이터가 있는 데이터베이스의 테이블 구조를 버전별로 변경할 때 Alembic을 사용합니다. Python 3.12 가상 환경에 SQLAlchemy 2.x와 Alembic 1.x를 설치하고 빈 학습 디렉터리를 초기화합니다.

```bash
python3.12 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install "SQLAlchemy>=2.0,<3" "alembic>=1.18,<2"
mkdir catalog
cd catalog
alembic init migrations
```

```text
Creating directory .../catalog/migrations ...  done
Creating directory .../catalog/migrations/versions ...  done
Generating .../catalog/alembic.ini ...  done
Generating .../catalog/migrations/env.py ...  done
```

`alembic.ini`는 연결 URL과 명령 설정을, `migrations/env.py`는 모델 메타데이터와 실행 방식을, `migrations/versions/`는 순서가 있는 revision 파일을 보관합니다. 이 환경은 프로젝트마다 한 번 만들고 애플리케이션 코드와 함께 버전 관리합니다.

## DeclarativeBase로 변경 대상 모델 준비하기

Alembic이 비교할 현재 모델을 제공하려면 `models.py`에 SQLAlchemy 선언형 베이스와 모델을 둡니다. 여기서는 상품 이름과 가격을 저장하는 최소 테이블부터 시작합니다.

```python
# models.py
from sqlalchemy import String
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column


class Base(DeclarativeBase):
    pass


class Product(Base):
    __tablename__ = 'product'

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100))
    price: Mapped[int]
```

모델은 원하는 최신 스키마를 표현하지만 그 자체로 기존 데이터베이스를 변경하지 않습니다. 학습·테스트에서 편리한 `Base.metadata.create_all()`도 이미 존재하는 열의 이름이나 제약 조건을 버전별로 변경하지 않으므로, 이 섹션에서는 호출하지 않습니다.

## sqlalchemy.url로 파일 SQLite 연결하기

마이그레이션 결과를 다음 명령에서도 다시 확인하려면 메모리 데이터베이스 대신 파일 SQLite를 사용합니다. `alembic.ini`에서 생성된 예시 URL을 다음 값으로 바꿉니다.

```ini
# alembic.ini
sqlalchemy.url = sqlite:///catalog.db
```

상대 경로 `catalog.db`는 Alembic 명령을 실행하는 현재 디렉터리를 기준으로 합니다. 운영 자격 증명은 설정 파일에 기록하지 않고 Part 3처럼 환경 변수나 배포 시스템에서 전달합니다.

## target_metadata로 모델과 Alembic 연결하기

모델 변경을 자동 감지하려면 `migrations/env.py`가 선언형 베이스의 `MetaData`를 알아야 합니다. 생성된 `target_metadata = None` 부분을 import와 함께 바꿉니다.

```python
# migrations/env.py
from models import Base


target_metadata = Base.metadata
```

`target_metadata`는 현재 Python 모델이 기대하는 테이블·열·제약의 모음입니다. 이 설정은 다음 노트의 `--autogenerate`가 데이터베이스의 현재 상태와 모델을 비교할 때 사용하며, revision을 자동으로 실행하지는 않습니다.

## upgrade와 downgrade로 첫 revision 작성하기

처음 배포할 `product` 테이블을 revision으로 기록합니다. 재현 가능한 예제를 위해 revision ID를 `0001`로 지정한 뒤 생성된 파일의 두 함수를 채웁니다.

```bash
alembic revision --rev-id 0001 -m "create product"
```

```python
# migrations/versions/0001_create_product.py
from collections.abc import Sequence

from alembic import op
import sqlalchemy as sa


revision: str = '0001'
down_revision: str | None = None
branch_labels: str | Sequence[str] | None = None
depends_on: str | Sequence[str] | None = None


def upgrade() -> None:
    op.create_table(
        'product',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('name', sa.String(length=100), nullable=False),
        sa.Column('price', sa.Integer(), nullable=False),
        sa.PrimaryKeyConstraint('id'),
    )


def downgrade() -> None:
    op.drop_table('product')
```

`upgrade()`는 이전 revision에서 현재 revision으로 진행하는 변경이고, `downgrade()`는 그 변경만 되돌립니다. `down_revision`이 이전 ID를 가리키므로 여러 파일이 하나의 순서로 연결됩니다.

## alembic upgrade로 최신 스키마 적용하기

비어 있는 데이터베이스를 최신 revision인 `head`까지 올리고 SQLite의 스키마를 조회합니다.

```bash
alembic upgrade head
python - <<'PY'
import sqlite3

with sqlite3.connect('catalog.db') as connection:
    columns = connection.execute('PRAGMA table_info(product)').fetchall()
    print([(column[1], column[2], column[3]) for column in columns])
PY
```

```text
[('id', 'INTEGER', 1), ('name', 'VARCHAR(100)', 1), ('price', 'INTEGER', 1)]
```

Alembic은 `product`와 함께 현재 revision을 저장하는 `alembic_version` 테이블을 만듭니다. 이미 적용된 revision은 다시 실행하지 않으므로 같은 `upgrade head` 명령을 반복해도 테이블이 중복 생성되지 않습니다.

## current와 history로 적용 상태 확인하기

배포 전에 데이터베이스의 현재 버전과 저장소에 기록된 변경 순서를 구분해서 확인합니다.

```bash
alembic current
alembic history
```

```text
0001 (head)
<base> -> 0001 (head), create product
```

`current`는 연결된 데이터베이스가 실제로 가리키는 revision을, `history`는 revision 파일의 전체 계보를 보여 줍니다. 코드의 `head`와 데이터베이스의 `current`가 다르면 아직 적용하지 않은 migration이 있다는 뜻입니다.

## downgrade로 변경을 되돌리고 결과 확인하기

마지막 변경의 역방향이 유효한지 확인할 때 한 단계 전 revision으로 내립니다. `-1`은 현재 위치의 바로 이전 revision을 뜻합니다.

```bash
alembic downgrade -1
python - <<'PY'
import sqlite3

with sqlite3.connect('catalog.db') as connection:
    tables = connection.execute(
        "SELECT name FROM sqlite_master WHERE type = 'table' ORDER BY name"
    ).fetchall()
    print([name for (name,) in tables])
PY
```

```text
['alembic_version']
```

테이블 삭제처럼 데이터를 잃는 downgrade는 운영 환경에서 자동으로 안전해지지 않습니다. revision을 배포하기 전에 빈 데이터베이스의 전체 upgrade와 가능한 범위의 downgrade를 테스트하고, 복구에 백업이 필요한 변경은 배포 절차에 명시합니다.
