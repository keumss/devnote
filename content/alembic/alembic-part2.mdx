## autogenerate가 감지한 변경을 초안으로 검토하기

모델을 바꾼 뒤 `--autogenerate`를 사용하면 현재 DB 스키마와 `target_metadata`의 차이를 바탕으로 리비전 초안이 만들어집니다. 메시지는 파일 이름이 아니라 변경 의도를 짧게 표현합니다.

```bash
alembic revision --autogenerate -m 'add email to users'
```

```text
INFO  [alembic.autogenerate.compare] Detected added column 'users.email'
Generating .../alembic/versions/<revision>_add_email_to_users.py ...  done
```

대표적인 생성 결과는 다음과 같습니다.

```python
def upgrade() -> None:
    op.add_column(
        'users',
        sa.Column('email', sa.String(length=255), nullable=True),
    )


def downgrade() -> None:
    op.drop_column('users', 'email')
```

자동 생성은 테이블·컬럼 추가와 삭제, nullable 변경, 기본적인 인덱스와 명시적으로 이름 붙인 제약 조건 등을 감지합니다. 하지만 컬럼 이름 변경을 삭제와 추가로 오해할 수 있고 데이터 변환 의도도 알지 못합니다. 생성 후에는 데이터 손실, 잠금 시간, downgrade 가능성을 포함해 두 함수를 모두 검토합니다.

## 기존 행이 있는 테이블에 NOT NULL 컬럼 추가하기

기존 행이 있는 테이블에 기본값 없는 NOT NULL 컬럼을 한 번에 추가하면 DB가 변경을 거부할 수 있습니다. 먼저 nullable 컬럼을 추가하고, 기존 데이터를 채운 다음, 누락된 값이 없는 것을 확인하고 제약을 강화합니다.

```python
from alembic import op
import sqlalchemy as sa

users = sa.table(
    'users',
    sa.column('status', sa.String(length=20)),
)


def upgrade() -> None:
    op.add_column(
        'users',
        sa.Column('status', sa.String(length=20), nullable=True),
    )
    op.execute(
        users.update()
        .where(users.c.status.is_(None))
        .values(status='active')
    )
    op.alter_column(
        'users',
        'status',
        existing_type=sa.String(length=20),
        nullable=False,
    )


def downgrade() -> None:
    op.drop_column('users', 'status')
```

이 방식도 모든 행을 한 트랜잭션에서 갱신하므로 작은 테이블에 적합합니다. 대형 테이블은 새 컬럼 추가, 애플리케이션의 양쪽 스키마 호환, 배치 백필, 제약 강화 단계를 여러 배포로 나눕니다.

## upgrade로 적용하고 현재 리비전 확인하기

`upgrade head`는 현재 DB 리비전에서 최신 head까지 필요한 리비전을 순서대로 실행합니다. 적용 전에는 백업뿐 아니라 DDL 잠금 시간과 대상 DB의 트랜잭션 지원 범위를 확인합니다.

```bash
alembic upgrade head
alembic current
```

```text
INFO  [alembic.runtime.migration] Running upgrade -> <revision>, add email to users
<revision> (head)
```

리비전 ID가 기대한 head와 일치해야 배포 코드와 DB 스키마가 같은 상태입니다. 여러 애플리케이션 워커가 동시에 실행하지 않도록 배포 파이프라인의 일회성 작업에서 한 번만 수행합니다.

## downgrade의 데이터 손실 가능성 판단하기

`downgrade -1`은 단일 head인 현재 경로에서 한 단계 내립니다. 앞의 이메일 예제에서는 컬럼과 저장된 이메일이 함께 사라집니다. 코드 롤백이 필요하다고 DB도 무조건 downgrade해야 하는 것은 아닙니다.

```bash
alembic downgrade -1
alembic current
```

```text
INFO  [alembic.runtime.migration] Running downgrade <revision> -> , add email to users
```

초기 리비전 이전으로 내렸다면 `current`가 출력할 리비전은 없습니다. 운영에서는 이전 애플리케이션이 새 스키마와 호환되는지 먼저 확인하고, 파괴적인 downgrade 대신 후속 수정 리비전을 우선 검토합니다. 되돌리면서 이미 수집한 데이터를 복구할 수 없다면 그 사실을 리비전 설명과 운영 절차에 명시합니다.

## history와 heads로 리비전 그래프 관리하기

리비전은 파일 이름 순서가 아니라 `revision`과 `down_revision`이 연결된 방향성 그래프입니다. `history`는 적용 경로를, `heads`는 최신 끝점을 보여 줍니다.

```bash
alembic history --verbose
alembic heads
```

```text
Rev: <revision> (head)
Parent: <base>
Path: .../<revision>_add_email_to_users.py

<revision> (head)
```

협업 중 같은 부모에서 서로 다른 리비전을 만들면 head가 둘 이상 생길 수 있습니다. 한쪽 파일의 `down_revision`을 임의로 고치면 이미 그 리비전을 적용한 DB의 경로가 깨집니다. 대신 병합 리비전을 만듭니다.

```bash
alembic merge heads -m 'merge migration branches'
alembic heads
```

```text
Generating .../alembic/versions/<merge_revision>_merge_migration_branches.py ...  done
<merge_revision> (head)
```

병합 리비전의 `down_revision`에는 두 기존 head가 튜플로 기록됩니다. 두 경로의 스키마 변경이 충돌한다면 빈 병합 파일로 끝내지 말고 충돌을 해소하는 작업을 명시합니다.
