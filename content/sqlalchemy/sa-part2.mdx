## 1:N 관계를 외래 키와 객체 속성으로 함께 매핑하기

외래 키는 DB가 참조 무결성을 지키고, `relationship()`은 Python 객체 사이를 탐색하게 합니다. 둘은 역할이 다르므로 관계 매핑에는 모두 필요합니다. 다음 예제에서 한 사용자는 여러 게시물을 가지며 각 게시물은 한 작성자를 가집니다.

```python
from __future__ import annotations

from sqlalchemy import ForeignKey, String
from sqlalchemy.orm import Mapped, mapped_column, relationship

class User(Base):
    __tablename__ = 'users'

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(30))
    posts: Mapped[list[Post]] = relationship(
        back_populates='author',
        cascade='all, delete-orphan',
    )

class Post(Base):
    __tablename__ = 'posts'

    id: Mapped[int] = mapped_column(primary_key=True)
    title: Mapped[str] = mapped_column(String(100))
    user_id: Mapped[int] = mapped_column(ForeignKey('users.id'))
    author: Mapped[User] = relationship(back_populates='posts')
```

```python
with Session(engine) as session:
    alice = User(name='Alice')
    alice.posts.append(Post(title='첫 SQLAlchemy 글'))
    session.add(alice)
    session.commit()

    print(alice.posts[0].title)
    print(alice.posts[0].author.name)
```

```text
첫 SQLAlchemy 글
Alice
```

한쪽 컬렉션에 추가하면 `back_populates`가 반대쪽 `author`도 연결합니다. `delete-orphan`은 컬렉션에서 떨어져 부모가 없는 게시물을 삭제하므로 도메인 정책에 맞을 때만 사용합니다. 목록 화면에서 각 사용자의 `posts`를 반복 접근하면 N+1 쿼리가 생길 수 있어 이후 로딩 전략을 함께 배워야 합니다.

## 객체 변경 감지로 한 행 수정하기

세션이 관리하는 객체의 속성을 바꾸면 SQLAlchemy가 변경을 추적하고 flush 시점에 UPDATE를 만듭니다. `commit()` 뒤 DB가 계산한 값이나 트리거 결과까지 확인하려면 `refresh()`로 다시 읽습니다.

```python
from sqlalchemy import select

with Session(engine) as session:
    alice = session.scalars(
        select(User).where(User.name == 'Alice')
    ).one()

    print(f'변경 전: {alice.name}')
    alice.name = 'Alicia'
    session.commit()
    session.refresh(alice)
    print(f'변경 후: {alice.name}')
```

```text
변경 전: Alice
변경 후: Alicia
```

`.one()`은 결과가 없거나 여러 개면 예외를 발생시켜 “정확히 한 사용자”라는 가정을 검증합니다. 요청 처리 중 예외가 발생하면 `rollback()`하고 세션을 종료해야 실패한 트랜잭션 상태를 다음 작업에 재사용하지 않습니다.

## delete와 cascade의 삭제 범위 확인하기

`session.delete()`는 전달한 ORM 객체를 삭제 대상으로 표시합니다. 위 관계의 `cascade='all, delete-orphan'` 때문에 사용자를 삭제하면 그 사용자가 소유한 게시물도 함께 삭제됩니다. 실행 전후 개수를 확인하면 삭제 범위를 테스트할 수 있습니다.

```python
from sqlalchemy import func, select

with Session(engine) as session:
    alicia = session.scalars(
        select(User).where(User.name == 'Alicia')
    ).one()

    before = session.scalar(select(func.count()).select_from(Post))
    session.delete(alicia)
    session.commit()
    after = session.scalar(select(func.count()).select_from(Post))

    print(before, after)
```

게시물이 한 개뿐이었다면 다음 결과가 나옵니다.

```text
1 0
```

실수로 대량 삭제하지 않도록 삭제 조건과 cascade를 테스트합니다. DB의 `ON DELETE CASCADE`와 ORM cascade를 함께 쓸 때는 어느 계층이 책임지는지 정하고 `passive_deletes` 설정까지 일치시켜야 합니다.

## bulk UPDATE의 영향받은 행 수 검증하기

여러 객체를 하나씩 읽어 수정할 필요가 없다면 SQL UPDATE 문을 직접 실행할 수 있습니다. 빠르지만 개별 객체 이벤트나 Python 수준 검증을 거치지 않으므로 단순하고 명확한 변경에 사용합니다.

```python
from sqlalchemy import update

with Session(engine) as session:
    result = session.execute(
        update(User)
        .where(User.age >= 30)
        .values(age=User.age + 1)
    )
    session.commit()

    print(f'수정된 사용자: {result.rowcount}명')
```

```text
수정된 사용자: 2명
```

`rowcount`가 예상 범위를 벗어나면 조건이 너무 넓거나 데이터 전제가 달라진 신호입니다. 중요한 대량 변경은 먼저 같은 WHERE 조건으로 SELECT 개수를 확인하고 트랜잭션 안에서 실행하며, 운영에서는 백업과 롤백 계획을 준비합니다.
