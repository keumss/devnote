## 즉시 로딩으로 N+1 문제 해결 (Eager Loading)

컬렉션을 접근할 때마다 쿼리가 발생하는 N+1 문제를 방지하기 위해 쿼리 단계에서 연관 데이터를 한 번에 가져옵니다.

```python
from sqlalchemy.orm import selectinload, joinedload

with Session(engine) as session:
    # selectinload는 별도 SELECT의 IN 조건으로 컬렉션을 로딩하므로 1:N에 대체로 적합합니다.
    # joinedload는 단일 JOIN으로 스칼라 관계(N:1, 1:1)를 가져올 때 자주 사용합니다.
    stmt = select(User).options(selectinload(User.posts))

    users = session.scalars(stmt).all()
    for u in users:
        # DB에 추가 접근(쿼리) 없이 하위 데이터(posts) 사용 가능!
        print(u.name, [p.title for p in u.posts])
```

## SQL 주입 안전한 원시 쿼리 (Raw SQL)

ORM으로 표현하기 힘들거나 극한의 튜닝이 필요한 쿼리를 파라미터 바인딩과 함께 안전히 실행합니다.

```python
from sqlalchemy import text

with Session(engine) as session:
    # 파라미터가 포함된 SQL
    stmt = text("SELECT id, name FROM users WHERE age >= :min_age")

    # 실행하면서 파라미터 딕셔너리 제공
    results = session.execute(stmt, {"min_age": 20})

    for row in results:
        # row.id 처럼 키로 접근 가능
        print(f"[{row.id}] {row.name}")
```

## FastAPI 환상의 짝꿍: 비동기 설정 (AsyncIO)

FastAPI의 속도를 극대화하기 위해 완전한 비동기(non-blocking) DB 엔진과 세션을 사용합니다.

```python
from sqlalchemy import select
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker

# 드라이버를 asyncpg (Postgres) 등으로 교체해야 함
async_engine = create_async_engine('postgresql+asyncpg://user:pass@localhost/mydb')

AsyncSessionLocal = async_sessionmaker(async_engine, expire_on_commit=False)

async def get_users():
    async with AsyncSessionLocal() as session:
        stmt = select(User)

        # await를 이용해 결과를 비동기적으로 기다림
        result = await session.execute(stmt)
        return result.scalars().all()
```

## 오류 방어: 트랜잭션 수동 제어

비즈니스 로직 중 예외가 발생할 경우, rollback()을 호출해 데이터가 꼬이는 것을 방지합니다.

```python
with Session(engine) as session:
    try:
        session.add(User(name="Daniel", age=40))

        # 여기서 DB 제약조건 위반 또는 서버 에러가 발생했다면?
        # raise Exception("Oops!")

        session.commit()
    except Exception as e:
        # 실패 시 전부 원상복구
        session.rollback()
        print(f"Transaction failed: {e}")
```

## 서비스 계층에서 트랜잭션 경계 만들기

한 요청의 여러 쓰기 작업은 하나의 서비스 함수에서 성공하거나 함께 취소되어야 합니다. 라우터나 화면 코드에 `commit()`을 흩어놓지 말고, 작업 단위가 끝나는 지점에서 한 번만 확정하면 실패 처리를 테스트하기도 쉽습니다.

```python
from sqlalchemy.orm import Session

def transfer_points(session: Session, sender: User, receiver: User, points: int) -> None:
    if points <= 0 or sender.points < points:
        raise ValueError('유효하지 않은 포인트 이체입니다.')

    # flush는 필요한 INSERT/UPDATE를 DB에 보내지만 아직 확정하지는 않습니다.
    sender.points -= points
    receiver.points += points
    session.flush()

    # 호출자가 성공 시점에 한 번만 commit하고, 예외 시 rollback합니다.
```
