## Field(default_factory)로 생성마다 새 기본값 만들기

생성 시점의 시간이나 빈 컬렉션처럼 인스턴스마다 새 값이 필요한 기본값에는 `Field(default_factory=...)`를 사용합니다. 팩터리는 모델을 만들 때마다 호출되므로 생성 시점과 생성 규칙을 선언에서 분명히 볼 수 있습니다.

```python
from datetime import datetime, timezone

from pydantic import BaseModel, Field


class AuditEvent(BaseModel):
    action: str
    created_at: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))
    tags: list[str] = Field(default_factory=list)


first = AuditEvent(action='login')
second = AuditEvent(action='logout')
first.tags.append('security')

print(first.tags)
print(second.tags)
print(first.created_at.tzinfo)
```

```text
['security']
[]
UTC
```

Pydantic은 변경 가능한 기본값을 그대로 선언해도 필요할 때 깊은 복사를 수행하지만, 새 값을 만든다는 의도가 있는 경우에는 `default_factory`가 더 명확합니다. 현재 시간처럼 실행마다 달라지는 값에는 반드시 팩터리를 사용합니다.

## Annotated와 StringConstraints로 재사용할 문자열 계약 만들기

여러 모델에서 같은 길이·공백·패턴 규칙을 사용해야 하면 `Annotated` 타입 별칭으로 묶습니다. 타입 검사기에는 여전히 `str`이고, Pydantic에는 검증 메타데이터가 전달됩니다.

```python
from typing import Annotated

from pydantic import BaseModel, StringConstraints


Username = Annotated[
    str,
    StringConstraints(strip_whitespace=True, min_length=3, max_length=20),
]


class Signup(BaseModel):
    username: Username
    invited_by: Username | None = None


signup = Signup(username='  minji  ', invited_by='admin')
print(signup)
```

```text
username='minji' invited_by='admin'
```

`Username`을 쓰는 모든 모델에 같은 공백 제거와 길이 규칙이 적용됩니다. 기본값·별칭처럼 생성자 시그니처에 영향을 주는 `Field()` 옵션은 일반 대입 형식을 우선 사용하고, 재사용할 타입 제약은 `Annotated`로 분리합니다.

## ConfigDict(extra='forbid')로 알 수 없는 입력 거절하기

클라이언트가 보내면 안 되는 키를 조용히 무시하지 않고 계약 오류로 알려야 할 때 `extra='forbid'`를 설정합니다. 생성·수정 요청 모델에 특히 유용하며, 오타와 예상하지 않은 필드를 입력 경계에서 발견할 수 있습니다.

```python
from pydantic import BaseModel, ConfigDict, ValidationError


class CreateProject(BaseModel):
    model_config = ConfigDict(extra='forbid')

    name: str


try:
    CreateProject(name='devnote', owner='minji')
except ValidationError as error:
    detail = error.errors(include_url=False)[0]
    print(detail['loc'])
    print(detail['type'])
```

```text
('owner',)
extra_forbidden
```

기본 설정인 `extra='ignore'`는 선언하지 않은 키를 버립니다. 외부 서비스가 임시 키를 붙이는 경우처럼 의도적으로 허용할 이유가 있을 때만 `ignore` 또는 `allow`를 선택합니다.

## validation_alias와 serialization_alias로 이름이 다른 외부 API 연결하기

Python 코드의 `snake_case`를 유지하면서 외부 API의 `camelCase` 필드를 읽고 쓸 때 별칭을 사용합니다. 입력 이름과 출력 이름이 같지 않을 수 있으므로, 어떤 경계에 적용하는지 `validation_alias`와 `serialization_alias`로 구분합니다.

```python
from pydantic import BaseModel, Field


class Customer(BaseModel):
    customer_id: int = Field(
        validation_alias='customerId',
        serialization_alias='customerId',
    )
    display_name: str = Field(
        validation_alias='displayName',
        serialization_alias='displayName',
    )


customer = Customer.model_validate({'customerId': '7', 'displayName': '민지'})

print(customer.customer_id)
print(customer.model_dump(by_alias=True))
```

```text
7
{'customerId': 7, 'displayName': '민지'}
```

기본 `model_dump()`는 Python 필드명으로 출력합니다. 외부 응답 계약을 만들 때만 `by_alias=True`를 명시하면, 내부 도메인 이름과 외부 표현을 섞지 않을 수 있습니다.

## model_dump(exclude_unset=True)로 PATCH의 부재와 null 구분하기

부분 수정 요청에서 클라이언트가 보내지 않은 필드는 기존 값을 유지하고, 명시적으로 `null`을 보낸 필드는 비우고 싶을 때 `exclude_unset=True`를 사용합니다. Pydantic은 모델 생성 때 실제로 제공된 필드를 따로 기록합니다.

```python
from pydantic import BaseModel


class UpdateProfile(BaseModel):
    display_name: str | None = None
    bio: str | None = None


omitted = UpdateProfile()
cleared = UpdateProfile(bio=None)

print(omitted.model_dump(exclude_unset=True))
print(cleared.model_dump(exclude_unset=True))
```

```text
{}
{'bio': None}
```

`UpdateProfile()`의 `bio` 속성값도 `None`이지만 입력으로 받은 적은 없습니다. 실제 데이터베이스 갱신에는 이 결과 딕셔너리의 키만 적용해, 누락·null·새 값을 각각 다른 상태 전이로 다룹니다.

## model_dump_json으로 날짜와 Decimal을 JSON으로 직렬화하기

`datetime`, `Decimal`처럼 일반 `json.dumps()`가 바로 다루기 어려운 값이 있는 모델을 JSON 문자열로 내보낼 때 `model_dump_json()`을 사용합니다. Pydantic이 모델 필드의 타입에 맞는 JSON 표현을 만듭니다.

```python
from datetime import date
from decimal import Decimal

from pydantic import BaseModel


class Invoice(BaseModel):
    invoice_id: str
    due_date: date
    total: Decimal


invoice = Invoice(invoice_id='INV-1', due_date='2026-07-31', total='12000.50')
print(invoice.model_dump_json())
```

```text
{"invoice_id":"INV-1","due_date":"2026-07-31","total":"12000.50"}
```

`model_dump_json()`은 문자열을 반환하고, `model_dump()`는 Python 딕셔너리를 반환합니다. HTTP 응답 라이브러리가 모델·딕셔너리를 자체 직렬화할 수 있다면 중복 변환하지 말고 해당 프레임워크의 응답 방식을 따릅니다.

## from_attributes=True로 속성 기반 객체를 응답 모델로 바꾸기

ORM 객체나 기존 클래스처럼 딕셔너리가 아니라 속성으로 값을 제공하는 객체를 읽어 모델로 바꿔야 할 때 `from_attributes=True`를 설정합니다. 경계 모델을 따로 두면 저장용 객체의 민감한 필드가 응답에 섞이는 일을 줄일 수 있습니다.

```python
from pydantic import BaseModel, ConfigDict


class StoredMember:
    def __init__(self, member_id: int, name: str, password_hash: str) -> None:
        self.member_id = member_id
        self.name = name
        self.password_hash = password_hash


class MemberResponse(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    member_id: int
    name: str


stored = StoredMember(7, '민지', 'secret-hash')
response = MemberResponse.model_validate(stored)

print(response.model_dump())
```

```text
{'member_id': 7, 'name': '민지'}
```

`MemberResponse`에 선언하지 않은 `password_hash`는 출력되지 않습니다. 데이터 저장 모델을 곧바로 외부 응답으로 쓰기보다, 생성·수정·공개 모델을 목적별로 분리하는 편이 API 계약을 안전하게 유지합니다.

## strict=True로 변환을 허용하지 않는 경계 만들기

이미 타입이 정확해야 하는 내부 메시지나 관리용 입력에서는 문자열 `'7'`을 정수로 바꾸는 기본 변환을 허용하지 않을 수 있습니다. `ConfigDict(strict=True)`를 설정하면 선언한 타입 자체가 들어와야 합니다.

```python
from pydantic import BaseModel, ConfigDict, ValidationError


class InternalEvent(BaseModel):
    model_config = ConfigDict(strict=True)

    sequence: int


print(InternalEvent(sequence=7))

try:
    InternalEvent(sequence='7')
except ValidationError as error:
    print(error.errors(include_url=False)[0]['type'])
```

```text
sequence=7
int_type
```

엄격 모드는 모든 외부 입력의 기본값은 아닙니다. HTML 폼처럼 문자열만 전달되는 경계에는 변환이 편리할 수 있으므로, 데이터 출처와 실패 정책을 보고 모델별로 선택합니다.

Python 3.12와 Pydantic 2.12의 기본값·별칭·직렬화·설정 동작은 [Pydantic Fields 공식 문서](https://docs.pydantic.dev/2.12/concepts/fields/), [Configuration 공식 문서](https://docs.pydantic.dev/2.12/concepts/config/), [Serialization 공식 문서](https://docs.pydantic.dev/2.12/concepts/serialization/)를 기준으로 확인합니다.
