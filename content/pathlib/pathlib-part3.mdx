## absolute와 resolve의 차이 이해하기

상대 경로를 절대 경로로 표시하는 것과 실제 대상을 해석하는 것은 다른 작업입니다. `absolute()`는 현재 작업 디렉터리를 기준으로 절대 형태를 만들지만 `..`를 정규화하거나 심볼릭 링크를 따라가지 않습니다. `resolve()`는 절대 경로를 만들면서 심볼릭 링크를 해석하고 `..` 조각을 제거합니다.

```python
from pathlib import Path

path = Path('reports') / '..' / 'data.csv'

print(path)
print(path.absolute().is_absolute())
print('..' in path.absolute().parts)
print('..' in path.resolve().parts)
```

```text
reports/../data.csv
True
True
False
```

`resolve(strict=True)`는 모든 경로 조각이 실제로 존재해야 하며 없으면 `FileNotFoundError`를 발생시킵니다. 기본값인 `strict=False`는 존재하는 부분까지 해석하고 나머지를 이어 붙이므로 새 출력 경로를 계산할 때도 사용할 수 있습니다.

`absolute()`와 `resolve()`는 경로를 표시하기 좋게 만드는 동일한 정규화 함수가 아닙니다. 현재 작업 디렉터리 기준의 절대 표기만 필요하면 `absolute()`, `..`와 심볼릭 링크를 반영한 실제 위치가 보안·중복 판정에 필요하면 `resolve()`를 선택합니다.

`expanduser()`는 `~/notes` 같은 홈 표시를 확장합니다. 홈 디렉터리를 결정할 수 없으면 원문을 그대로 반환하는 일부 문자열 API와 달리 `RuntimeError`를 발생시킬 수 있습니다. 애플리케이션 설정을 읽을 때 이 실패도 입력 오류로 다룹니다.

## relative_to와 is_relative_to로 경로 관계 표현하기

`relative_to(base)`는 경로가 `base` 아래에 있을 때 기준을 제거한 새 경로를 반환하고, 그렇지 않으면 `ValueError`를 발생시킵니다. `is_relative_to(base)`는 같은 관계를 불리언으로 확인합니다. 두 메서드는 기본적으로 파일 시스템을 읽지 않는 어휘적 연산입니다.

```python
from pathlib import PurePosixPath

root = PurePosixPath('/srv/site')
asset = PurePosixPath('/srv/site/static/app.css')
other = PurePosixPath('/srv/private/key.txt')

print(asset.is_relative_to(root))
print(asset.relative_to(root))
print(other.is_relative_to(root))
```

```text
True
static/app.css
False
```

Python 3.12에서 `relative_to(other, walk_up=True)`는 필요하면 `..`를 포함해 다른 경로로 가는 상대 표현을 계산합니다. 심볼릭 링크가 있는 실제 파일 시스템 관계를 반영해야 한다면 먼저 양쪽을 `resolve()`해야 합니다.

`str(candidate).startswith(str(root))`로 포함 관계를 검사하면 안 됩니다. `/srv/site-backup`도 `/srv/site`로 시작하고, 대소문자와 구분자 규칙도 직접 처리해야 합니다. 경로 관계에는 경로 조각을 이해하는 `is_relative_to()`를 사용합니다.

## 외부 입력을 허용된 루트 아래로 제한하기

사용자 입력으로 다운로드나 압축 해제 대상 경로를 만들 때는 `../`와 절대 경로로 기준 디렉터리를 벗어나는 경로 순회 공격을 막아야 합니다. 먼저 절대 입력을 거부하고, 기준과 후보를 실제 경로로 해석한 뒤 포함 관계를 검사합니다.

```python
from pathlib import Path
from tempfile import TemporaryDirectory

def path_within(root: Path, user_value: str) -> Path:
    supplied = Path(user_value)
    if supplied.is_absolute():
        raise ValueError('절대 경로는 허용하지 않습니다.')

    resolved_root = root.resolve(strict=True)
    candidate = (resolved_root / supplied).resolve(strict=False)
    if not candidate.is_relative_to(resolved_root):
        raise ValueError('허용된 디렉터리를 벗어났습니다.')
    return candidate

with TemporaryDirectory() as directory:
    root = Path(directory) / 'uploads'
    root.mkdir()

    accepted = path_within(root, 'users/avatar.png')
    print(accepted.relative_to(root))

    try:
        path_within(root, '../secrets.txt')
    except ValueError as error:
        print(error)
```

```text
users/avatar.png
허용된 디렉터리를 벗어났습니다.
```

이 검사는 이미 존재하는 심볼릭 링크도 해석하지만, 검사 뒤 파일을 열기 전에 공격자가 디렉터리를 심볼릭 링크로 바꾸는 경쟁 조건까지 없애지는 못합니다. 따라서 “경로 계산 결과가 루트 안이다”라는 검증과 “그 파일을 안전하게 열었다”는 보장은 구분해야 합니다. 신뢰할 수 없는 사용자가 같은 파일 트리를 동시에 변경할 수 있는 고위험 환경에서는 프로세스 권한과 저장소를 격리하고, 운영체제의 디렉터리 파일 디스크립터 기준 API 같은 더 강한 보호를 사용해야 합니다.

파일 이름 정책도 별개입니다. 경로가 루트 안에 있어도 확장자, 최대 크기, 기존 파일 덮어쓰기 허용 여부를 작업 목적에 맞게 추가 검증합니다.

## 심볼릭 링크를 명시적으로 다루기

`is_symlink()`는 경로 자체가 심볼릭 링크인지 확인하고, `readlink()`는 링크에 저장된 대상을 경로 객체로 반환합니다. `stat()`과 `is_file()`은 기본적으로 링크를 따라 대상의 정보를 확인하지만 `lstat()`은 링크 자체의 메타데이터를 읽습니다.

```python
from pathlib import Path
from tempfile import TemporaryDirectory

with TemporaryDirectory() as directory:
    root = Path(directory)
    target = root / 'target.txt'
    link = root / 'latest.txt'

    target.write_text('current', encoding='utf-8')
    link.symlink_to(target.name)

    print(link.is_symlink())
    print(link.readlink())
    print(link.resolve() == target.resolve())
    print(link.read_text(encoding='utf-8'))
```

심볼릭 링크를 지원하는 환경의 예상 결과입니다. Windows에서는 링크 생성 권한이나 개발자 모드 설정이 필요할 수 있습니다.

```text
True
target.txt
True
current
```

링크의 저장 대상은 링크가 위치한 디렉터리를 기준으로 해석될 수 있으므로 `readlink()` 결과가 절대 경로라고 가정하지 않습니다. 링크 생성, 교체, 삭제 코드는 실제 파일을 건드리는지 링크만 건드리는지 테스트로 고정합니다.

두 경로 문자열이 달라도 같은 파일을 가리키는지 확인하려면 `samefile()`을 사용할 수 있습니다. 두 대상에 접근할 수 없으면 `OSError`가 발생할 수 있으므로 단순 문자열 비교의 안전한 대체라고 생각하지 말고 실제 파일 동일성이 필요한 시점에만 호출합니다.

## Path.walk로 큰 트리를 제어하며 순회하기

Python 3.12에 추가된 `Path.walk()`는 각 디렉터리마다 `(directory, dirnames, filenames)`를 생성합니다. 기본 하향식 순회에서는 `dirnames` 목록을 제자리에서 수정해 불필요한 하위 트리를 건너뛸 수 있습니다.

```python
from pathlib import Path

def python_sources(root: Path):
    ignored = {'.git', '.venv', '__pycache__'}

    for directory, dirnames, filenames in root.walk():
        dirnames[:] = sorted(
            name for name in dirnames if name not in ignored
        )
        for name in sorted(filenames):
            if name.endswith('.py'):
                yield directory / name

for source in python_sources(Path('project')):
    print(source)
```

출력은 `project`의 실제 파일에 따라 달라집니다. `.git`, `.venv`, `__pycache__` 아래의 파일은 방문하지 않고, 각 디렉터리 안에서는 이름순으로 출력합니다.

`walk()`는 기본적으로 디렉터리 심볼릭 링크를 따라가지 않고 이를 `filenames`에 넣습니다. `follow_symlinks=True`를 사용하면 부모를 가리키는 링크 때문에 무한 순환할 수 있으며 `Path.walk()`가 방문한 디렉터리를 자동 추적하지 않습니다. 특별한 요구가 없다면 기본값을 유지합니다.

권한 오류 같은 순회 실패는 기본적으로 무시됩니다. 누락 없는 보고서가 필요하면 `on_error`에서 예외를 다시 발생시키고, 일부 결과라도 필요한 검색이라면 오류 경로를 기록한 뒤 계속하는 정책을 선택합니다.

```python
def stop_on_error(error: OSError) -> None:
    raise error

for directory, dirnames, filenames in Path('project').walk(
    on_error=stop_on_error,
):
    ...
```

## 탐색 비용과 오류 경계를 설계하기

`iterdir()`, `glob()`, `rglob()`, `walk()`는 결과를 지연 생성하므로 필요한 항목만 소비할 수 있습니다. 바로 목록으로 바꾸면 정렬과 개수 계산은 쉬워지지만 모든 경로를 메모리에 보관합니다. 큰 트리에서는 생성자를 유지하며 한 항목씩 처리합니다.

재귀 패턴 `**`는 편리하지만 대규모 디렉터리에서 많은 시간이 걸릴 수 있습니다. 제외할 디렉터리가 분명하거나 방문 오류를 수집해야 한다면 `walk()`로 가지치기하는 편이 제어하기 쉽습니다. 확장자 하나만 확인하려고 각 경로에 `stat()`까지 호출하지 말고 문자열 수준의 `suffix`를 먼저 검사해 시스템 호출을 줄입니다.

```python
from pathlib import Path

def total_markdown_bytes(root: Path) -> int:
    total = 0
    for path in root.rglob('*'):
        if path.suffix.lower() not in {'.md', '.mdx'}:
            continue
        try:
            if path.is_file():
                total += path.stat().st_size
        except OSError as error:
            print(f'건너뜀: {path}: {error}')
    return total
```

`is_file()`과 `stat()` 사이에도 파일 상태는 바뀔 수 있습니다. 예제는 개별 항목 실패를 허용하는 집계 정책이라 오류를 기록하고 계속합니다. 백업이나 배포 검증처럼 하나라도 빠지면 결과를 신뢰할 수 없는 작업은 같은 예외를 다시 발생시켜 전체를 실패시킵니다. 오류를 잡는 범위는 “무엇까지 성공으로 인정할 것인가”라는 제품 정책과 일치해야 합니다.

Python 3.12의 세부 동작은 [pathlib 공식 문서](https://docs.python.org/3.12/library/pathlib.html)를 기준으로 확인합니다. 특히 다른 Python 버전을 지원한다면 `Path.walk()`와 `case_sensitive` 인자처럼 3.12에 추가된 기능을 그 버전 문서에서 다시 확인합니다.
